deliverables:
  - Архитектурная схема RAG-системы (C4 diagram)
  - Спецификация компонентов: chunking, search, generation
  - Требования к данным: источники, частота обновления, качество
  - Оценка стоимости инференса (cost model)
  - План тестирования качества ответов
context: |
  Руководство компании хочет внедрить AI-ассистента для службы поддержки: пользователь
  задаёт вопрос на естественном языке, ассистент отвечает на основе базы знаний из
  10 000 PDF-документов (инструкции, FAQ, регламенты). Критично: ответы должны быть
  точными, со ссылками на источники, без галлюцинаций.
steps:
  - Определить источники знаний: какие документы, форматы, частота обновления, права доступа
  - Выбрать стратегию chunking: размер чанка, overlap, стратегия разбиения (по абзацам, семантическое)
  - Выбрать embedding-модель и векторную БД (Qdrant, Pinecone, pgvector)
  - Спроектировать retrieval: top-K, threshold, hybrid search (semantic + keyword)
  - Спроектировать промпт: system message, инструкция по цитированию, fallback «не знаю»
  - Выбрать LLM: open-source (Llama) или API (GPT-4), оценить стоимость на ожидаемый объём запросов
  - Спроектировать архитектуру: компоненты, их взаимодействие, data flow
  - Определить метрики качества: accuracy ответов (экспертная оценка), hallucination rate, latency, cost
  - Спроектировать систему мониторинга: логирование, алерт при падении качества
pitfalls:
  - Использовать LLM без RAG — модель будет галлюцинировать на вопросы, которых нет в обучении
  - Не учесть latency — LLM может отвечать 5-10 секунд, что неприемлемо для чата
  - Забыть про cost — каждый запрос к GPT-4 стоит денег, при 10K запросов/день бюджет может быть высоким
  - Не тестировать на краевых случаях — если пользователь спрашивает на сленге или с ошибками — система должна работать
  - Игнорировать приватность — документы могут содержать персональные данные, их нельзя отправлять во внешние API
