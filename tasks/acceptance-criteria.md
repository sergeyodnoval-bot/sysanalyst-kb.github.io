# Написание acceptance criteria для user story

**Контекст:** Вы — аналитик в команде, которая работает над интернет-магазином. Команда берёт в спринт user story. Разработчики просят уточнить acceptance criteria — без них непонятно, когда story можно считать готовой.

## Цель задачи

Написать acceptance criteria и BDD-сценарии для заданной user story, чтобы команда могла взять её в разработку и проверить результат.

## User story

> **Как** менеджер поддержки  
> **Я хочу** фильтровать заказы по статусу и дате  
> **Чтобы** быстро находить проблемные заказы

## Пошаговый подход

### Шаг 1. Поймите, что нужно

Прочитайте user story и выделите ключевые моменты:

- Кто пользователь? Менеджер поддержки
- Что он хочет делать? Фильтровать заказы
- По каким параметрам? Статус и дата
- Зачем? Быстро находить проблемные заказы

### Шаг 2. Выделите сценарии

**Happy path:** менеджер открывает страницу заказов, выбирает фильтр «Статус = Доставлен», видит только доставленные заказы.

**Граничные случаи:**
- По фильтру ничего не найдено (что показывать?)
- Выбраны оба фильтра (как они комбинируются? AND или OR?)
- Дата в будущем (нужно ли обрабатывать?)
- Пользователь не авторизован

### Шаг 3. Сформулируйте acceptance criteria

1. Менеджер может отфильтровать заказы по статусу (выпадающий список со значениями: «Все», «Новый», «Подтверждён», «Доставлен», «Отменён»)
2. Менеджер может отфильтровать заказы по диапазону дат (с — по)
3. При выборе обоих фильтров они применяются одновременно (логическое AND)
4. По умолчанию фильтры неактивны — показываются все заказы за последние 30 дней
5. Если по фильтру ничего не найдено — отображается сообщение «Нет заказов, соответствующих фильтру»

### Шаг 4. Напишите BDD-сценарии

**Scenario 1: Фильтрация по статусу (happy path)**

*Given* менеджер авторизован в системе  
*And* в системе есть заказы со статусами «Новый» и «Доставлен»  
*When* менеджер выбирает статус «Доставлен»  
*Then* отображаются только заказы со статусом «Доставлен»  
*And* количество заказов соответствует количеству доставленных

**Scenario 2: Фильтр не дал результатов**

*Given* менеджер авторизован в системе  
*And* нет заказов со статусом «Отменён»  
*When* менеджер выбирает статус «Отменён»  
*Then* отображается сообщение «Нет заказов, соответствующих фильтру»  
*And* менеджер может сбросить фильтр и увидеть все заказы

## Критерии выполнения

- [ ] Acceptance criteria покрывают happy path и минимум 2 граничных случая
- [ ] Каждый критерий сформулирован как конкретное условие (можно проверить)
- [ ] BDD-сценарии следуют формату Given/When/Then
- [ ] Сценарии не описывают реализацию, только поведение системы

## Типичные ошибки

**Слишком общие критерии.** «Фильтр работает быстро» — непроверяемо. «Результаты фильтрации отображаются за 2 секунды» — можно проверить.

**Путать с тест-кейсами.** Acceptance criteria отвечает на вопрос «что сделано?», тест-кейс — «как проверить?». Для приведённых выше AC не нужно писать «открыть страницу → выбрать статус → нажать кнопку» — это шаги тестирования, а не условия готовности.

**Писать критерии до понимания сценариев.** Сначала подумайте, какие сценарии вообще есть, потом формулируйте критерии.
