Написание user story

Контекст: После сбора требований (см. задачу «Сбор требований у стейкхолдера») пришло время формализовать их в user stories. Product Owner утвердил, что функциональность «История заказов покупателя для менеджера» — в приоритете.

Цель задачи​

Написать качественную user story с acceptance criteria и BDD-сценариями, которую команда разработки сможет взять в спринт.

Пошаговый подход​

Шаг 1. Сформулируйте user story​

Кто будет пользоваться функцией? Какую проблему решает? Зачем она нужна?

Роль: Менеджер поддержки клиентов Действие: Просматривать историю заказов конкретного покупателя Причина: Чтобы быстро отвечать на вопросы клиентов о статусе заказов

Готовая формулировка: Как менеджер поддержки, я хочу видеть историю заказов покупателя на одном экране, чтобы быстро отвечать на вопросы о статусе и составе заказов.

Шаг 2. Запишите acceptance criteria​

Условия, при которых story считается готовой:

  1. Менеджер может найти покупателя по номеру телефона или email
  2. На экране отображаются все заказы покупателя за всё время (а не только последние)
  3. Каждый заказ показывает: номер, дату, статус, сумму, состав товаров
  4. Можно открыть детали заказа (состав, адрес доставки, история статусов)
  5. Если у покупателя нет заказов — отображается сообщение «У покупателя нет заказов»

Шаг 3. Напишите BDD-сценарии​

Scenario 1: Просмотр истории заказов (happy path)

Given менеджер авторизован в системе And существует покупатель Иванов Иван с тремя заказами When менеджер вводит номер телефона +7 (999) 123-45-67 в поле поиска Then отображается список из 3 заказов покупателя And каждый заказ содержит: номер, дату, сумму, статус And менеджер видит состав первого заказа стрелкой раскрытия

Scenario 2: Покупатель не найден

Given менеджер авторизован в системе When менеджер вводит номер телефона +7 (999) 000-00-00 And покупатель с таким телефоном не зарегистрирован Then отображается сообщение «Покупатель не найден» And менеджер может попробовать найти по email

Шаг 4. Создайте задачу в Jira​

  1. Откройте проект в Jira
  2. Нажмите «Create Issue», выберите тип Story
  3. Заполните Summary: скопируйте user story
  4. В Description вставьте acceptance criteria и BDD-сценарии
  5. Выберите приоритет (Medium)
  6. Добавьте лейблы: support, orders, mvp
  7. Нажмите «Create»

Шаг 5. Проверьте, что story готова к спринту​

Пройдите по критериям INVEST:

  • Independent — story про историю заказов не зависит от других функций
  • Negotiable — детали (какие поля показывать) можно обсудить с командой
  • Valuable — менеджеры быстрее отвечают клиентам
  • Estimable — типовой экран со списком, понятно, что делать
  • Small — укладывается в одну итерацию
  • Testable — есть acceptance criteria и BDD-сценарии

Критерии выполнения​

  • User story написана по шаблону «Как / Я хочу / Чтобы»
  • Acceptance criteria покрывают основной сценарий и минимум один граничный случай
  • BDD-сценарии следуют формату Given / When / Then
  • Задача создана в Jira с типом Story и заполненными полями

Типичные ошибки​

Писать story без причины (части «чтобы»). Если не указать ценность, разработчики могут сделать не то. Например, история «Как менеджер, я хочу видеть историю заказов» без «чтобы» может превратиться в бесконечный список заказов без поиска и детализации.

Слишком крупная story (epic вместо story). «Как менеджер, я хочу управлять заказами» — это epic, а не story. Разбейте на отдельные истории: просмотр, редактирование, отмена, возврат.

Путать acceptance criteria с тест-кейсами. Acceptance criteria — условия готовности, понятные заказчику. Тест-кейсы — технические шаги проверки для тестировщика. Acceptance criteria: «Заказы фильтруются по дате». Тест-кейс: «Открыть страницу, выбрать дату с 01.01 по 31.01, нажать фильтр, проверить, что отображаются заказы только за этот период».
