## Контекст

Вы — системный аналитик в платёжном агрегаторе. Клиенты (мерчанты) жалуются на расхождения: их внутренний учёт показывает 1000 транзакций, а в платёжном агрегаторе — 1002. Нужно спроектировать систему сверки (reconciliation) для ежедневной сверки между данными агрегатора и данными банка-эквайера.

**Требования:**
- Сверка происходит каждую ночь (batch, T+1)
- Агрегатор соединяется с банком через SFTP (выгрузка CSV/XML)
- Необходимо выявлять: missing transactions, amount mismatches, duplicate transactions, orphan settlements
- Результат — отчёт с расхождениями (dashboard + email)

## Что нужно сделать

1. **Схема данных** — какие таблицы нужны для хранения транзакций агрегатора, банковских транзакций, результатов сверки
2. **Алгоритм сверки** — описать шаги матчинга (по каким полям сравниваем, fuzzy match?, что считается mismatch)
3. **Отчёт** — описать структуру отчёта по расхождениям (dashboard view + CSV export)
4. **Error handling** — что если банк не прислал выписку? что если формат выписки изменился?
5. **Non-functional requirements** — время выполнения (окно — 4 часа), объём данных (1 млн транзакций)

## Критерии приемки

- [ ] Алгоритм сверки покрывает 4+ типа расхождений с примерами
- [ ] Схема данных содержит не менее 4 таблиц с полями и связями
- [ ] Отчёт содержит минимум 3 вью (краткий обзор, детальный, по мерчантам)
- [ ] Учтён сценарий: количество совпало, но сумма различается (partial match)
- [ ] Описаны метрики качества сверки: match rate, false positive rate
