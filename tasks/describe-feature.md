# Описать существующую фичу

**Контекст:** Вы — новый аналитик в команде. Документации нет. Заказчик просит разобраться, как работает восстановление пароля, и описать это для команды. Вы будете действовать как «reverse engineer» — открыть приложение, поэкспериментировать и задокументировать результат.

## Цель задачи

Описать существующую функциональность без доступа к исходному коду — через исследование приложения. Получить документ, который сможет использовать и разработчик, и тестировщик, и новый аналитик.

## Пошаговый подход

### Шаг 1. Найдите экран

Откройте приложение. Найдите экран входа. Нажмите «Забыли пароль?» или «Восстановить пароль». Запишите, как вы туда попали (URL, последовательность действий).

### Шаг 2. Пройдите весь путь

Полностью пройдите весь процесс восстановления пароля:

1. Ввод email
2. Проверка почты (если приходит письмо)
3. Переход по ссылке из письма
4. Ввод нового пароля
5. Подтверждение
6. Вход с новым паролем

На каждом шаге записывайте:
- Что видит пользователь (скриншот)
- Что происходит «под капотом» (какие API вызываются)
- Что будет, если сделать что-то не так (неправильный email, истёкший токен)

### Шаг 3. Нарисуйте sequence diagram

Изобразите взаимодействие участников:

```
Пользователь → Браузер: нажимает «Забыли пароль?»
Браузер → Сервер: POST /api/auth/forgot-password {email}
Сервер → БД: найти пользователя по email
Сервер → Email Service: отправить письмо с токеном
Браузер → Пользователь: «Проверьте почту»
...
```

### Шаг 4. Опишите неосновные сценарии

| Сценарий | Что происходит |
|----------|---------------|
| Email не зарегистрирован | Показывается сообщение или нет? (security issue) |
| Письмо не пришло | Есть ли кнопка «Отправить ещё раз»? |
| Ссылка устарела | Что говорит система? |
| Пароль слишком простой | Есть ли валидация на клиенте и сервере? |

### Шаг 5. Задокументируйте результат

Создайте страницу «Восстановление пароля — описание функциональности»:

1. **Назначение** — зачем нужна эта функция
2. **Основной сценарий** — пошагово с экранами
3. **Sequence diagram** — взаимодействие компонентов
4. **Неосновные сценарии** — таблица ошибок
5. **UI/UX заметки** — скриншоты с аннотациями

## Критерии выполнения

- [ ] Пройден весь путь восстановления пароля (от начала до конца)
- [ ] Sequence diagram описывает минимум 3 участника (User, Frontend, Backend)
- [ ] Описаны минимум 3 неосновных сценария
- [ ] Скриншоты снабжены аннотациями
- [ ] Результат задокументирован в Confluence или Google Docs

## Пример хорошего результата

> **Сценарий: успешное восстановление пароля**
> 
> 1. Пользователь нажимает «Забыли пароль?» на экране входа
> 2. Система показывает форму ввода email
> 3. Пользователь вводит email и нажимает «Отправить»
> 4. Система проверяет, есть ли такой email в БД:
>    - Если нет — всё равно показывает «Проверьте почту» (безопасность: не сообщаем, зарегистрирован ли пользователь)
>    - Если есть — генерирует токен, сохраняет в БД (срок действия: 1 час), отправляет письмо
> 5. Пользователь открывает почту, видит письмо с ссылкой вида `https://example.com/reset-password?token=abc123`
> 6. Пользователь переходит по ссылке
> 7. Система проверяет токен:
>    - Токен существует → открывается форма ввода нового пароля
>    - Токен истёк → сообщение «Ссылка устарела, запросите новую»
> ...

## Типичные ошибки

**Описывать интерфейс, а не логику.** «Поле ввода email» — очевидно. А что происходит после ввода? Отправляется запрос? Проверяется email? Приходит письмо? Системная логика важнее визуала.

**Забывать про ошибки.** Разработчику нужно знать не только happy path, но и что делать, когда пользователь ввёл неправильный пароль 5 раз, когда токен истёк, когда сервер недоступен.

**Не проверять на свежую голову.** Легко записать то, что *должно* быть, а не то, что *есть* на самом деле. Проверьте каждый шаг — откройте приложение и пройдите путь заново.
