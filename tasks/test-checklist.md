# Подготовка чек-листа для тестирования

**Контекст:** Команда закончила разработку функции «Фильтрация заказов по статусу и дате». Перед демо заказчику нужно проверить, что всё работает. Ваш чек-лист — основа для этой проверки.

## Цель задачи

Составить чек-лист проверок, который покрывает все acceptance criteria и BDD-сценарии. Чек-лист должен быть понятен любому участнику команды (аналитику, тестировщику, разработчику, заказчику).

## Пошаговый подход

### Шаг 1. Изучите документацию

Перечитайте user story, acceptance criteria и BDD-сценарии. Это основа вашего чек-листа.

**User story:** Как менеджер поддержки, я хочу фильтровать заказы по статусу и дате, чтобы быстро находить проблемные заказы.

**Acceptance criteria:**
1. Менеджер может отфильтровать заказы по статусу
2. Менеджер может отфильтровать заказы по диапазону дат
3. При выборе обоих фильтров они применяются одновременно (AND)
4. По умолчанию фильтры неактивны — показываются все заказы за последние 30 дней
5. Если по фильтру ничего не найдено — сообщение «Нет заказов»

### Шаг 2. Выделите проверки

**Happy path (основной сценарий):**
- [ ] Фильтрация по статусу: выбран «Доставлен» → отображаются только доставленные заказы
- [ ] Фильтрация по дате: выбран диапазон 01.03.2024 — 15.03.2024 → отображаются заказы только за этот период
- [ ] Комбинированный фильтр: статус «Новый» + дата за последнюю неделю → результаты соответствуют обоим условиям

**Граничные случаи:**
- [ ] Фильтр не дал результатов → показывается сообщение «Нет заказов, соответствующих фильтру»
- [ ] Все фильтры пустые → показываются все заказы за последние 30 дней
- [ ] Дата «по» меньше «с» → что происходит? (ошибка или сброс?)
- [ ] Выбрано 0 заказов после фильтра → счётчик показывает 0

**Ошибки и исключения:**
- [ ] Пользователь не авторизован → редирект на логин или сообщение об ошибке
- [ ] Сервер недоступен → сообщение «Не удалось загрузить данные»
- [ ] Выбрана будущая дата → пустой результат или ошибка?

### Шаг 3. Сгруппируйте по категориям

| Категория | Количество проверок |
|-----------|-------------------|
| Фильтр по статусу | 3-4 |
| Фильтр по дате | 3-4 |
| Комбинированный фильтр | 2-3 |
| UI/UX | 3-4 |
| Ошибки | 2-3 |

### Шаг 4. Добавьте проверки UI/UX

- [ ] Выпадающий список статуса содержит все значения
- [ ] Выбранный фильтр визуально выделен
- [ ] Счётчик результатов обновляется после применения фильтра
- [ ] Есть кнопка «Сбросить фильтры»
- [ ] Сообщение о пустом результате понятно пользователю

### Шаг 5. Оформите в Jira

1. Создайте задачу типа Test в Jira
2. В Description вставьте чек-лист
3. Добавьте лейблы: `testing`, `checklist`, `orders`
4. Прикрепите ссылки на user story и acceptance criteria

## Критерии выполнения

- [ ] Чек-лист покрывает все acceptance criteria
- [ ] Есть проверки для happy path, граничных случаев и ошибок
- [ ] Каждый пункт конкретен (можно проверить одной операцией)
- [ ] Указан ожидаемый результат для каждого пункта

## Типичные ошибки

**Слишком общие пункты.** «Проверить фильтрацию» — не годится. «Выбрать статус «Доставлен» → убедиться, что отображаются только доставленные заказы» — конкретно и проверяемо.

**Только happy path.** Самое очевидное («кнопка работает») обычно проверяют все. А вот «что будет, если выбрать дату в будущем» — забывают. Хороший чек-лист = happy path + граничные случаи + ошибки.

**Чек-лист без ожидаемого результата.** «Выбрать статус» — и что должно произойти? Каждый пункт должен содержать ожидание: «выбрать статус → отображаются только заказы с этим статусом».
