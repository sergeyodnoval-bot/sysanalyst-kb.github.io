## Контекст

Вы — системный аналитик в e-commerce компании. Стартап запускает маркетплейс одежды. Нужно спроектировать Order Management System (OMS) — машину состояний заказа.

**Требования:**
- Продавцы (sellers) подключаются через API
- Модели: FBO (маркетплейс хранит), FBS (продавец хранит, маркетплейс доставляет), DBS (продавец хранит и доставляет)
- Оплата: карты, SBP, BNPL, наличные при получении
- Возвраты: 14 дней, возможен частичный возврат
- Трекинг: от создания до получения

## Что нужно сделать

1. **Машина состояний заказа** — таблица статусов (12+) с описанием переходов между ними (диаграмма состояний)
2. **Таблица переходов** — для каждого перехода указать: событие, условие, система-инициатор, действия
3. **Сплит заказа** — описать логику разделения заказа на несколько отправлений (разные склады, разные продавцы)
4. **Обработка ошибок** — что если платёж не прошёл? а если товара нет на складе? а если курьер потерял посылку?
5. **Схема данных OMS** — ER-диаграмма (orders, order_items, payments, shipments, history)

## Критерии приемки

- [ ] Машина состояний содержит не менее 12 статусов с полным графом переходов
- [ ] Таблица переходов содержит не менее 10 строк (событие → условие → результат)
- [ ] Описан механизм сплита заказа (один заказ → несколько отправлений)
- [ ] Учтены 3+ error-сценария с решением
- [ ] ER-диаграмма OMS содержит не менее 5 таблиц
