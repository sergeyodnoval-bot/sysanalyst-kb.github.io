:::info[TL;DR]
Подготовить чек-лист для аттестации ГИС по ФСТЭК: категорирование, меры защиты (приказ № 239), модель угроз, документация, организационные меры. Система: 500 операторов, 50 администраторов, Astra Linux + Postgres Pro + КриптоПро, интеграция с СМЭВ.
:::

## Контекст

ГИС для выдачи справок гражданам. Система обрабатывает ПД:

- **Пользователи:** 500 операторов, 50 администраторов
- **Данные:** ФИО, паспорт, СНИЛС, телефон — всего 10 ТБ
- **Инфраструктура:** Astra Linux Special Edition, Postgres Pro Enterprise, КриптоПро CSP
- **Интеграции:** СМЭВ (3 внешних сервиса: ФНС, СФР, МВД)
- **Портал:** веб-интерфейс для операторов, без внешнего портала

## Цель задачи

Подготовить чек-лист аттестации по ФСТЭК: категорирование, меры защиты, модель угроз, документация, организационные меры.

## Пошаговый подход

### Шаг 1: Категорирование

**Исходные данные:** ПД (ФИО, паспорт, СНИЛС) → уровень защищённости?

| Фактор | Значение | Категория |
|--------|----------|-----------|
| Категория ПД | ФИО + паспорт = особая категория (ст. 10 152-ФЗ) | УЗ-2 |
| Количество субъектов | 100K+ | УЗ-2 |
| Технология | Веб-доступ через интернет | УЗ-2 (внешний контур) |
| Интеграция | СМЭВ — межведомственное взаимодействие | УЗ-2 |

**Вывод:** УЗ-2. Обоснование: обработка ПД (особая категория — паспортные данные), доступ через интернет, СМЭВ-интеграция.

### Шаг 2: Меры защиты (приказ № 239)

| Группа | Мера | Конкретная реализация |
|--------|------|----------------------|
| **ИАФ** | Идентификация и аутентификация | ЕСИА + УКЭП (для администраторов), логин/пароль (для операторов) |
| **УПД** | Управление доступом | RBAC: 3 роли (Администратор, Оператор, Аудитор) |
| **ОЛС** | Ограничение программной среды | Astra Linux (режим «Орёл»), белые списки ПО |
| **ЗИС** | Защита среды виртуализации | zVirt, изоляция VM |
| **ЗНИ** | Защита сети | ViPNet Coordinator (СМЭВ-канал), МЭ (UserGate) |
| **ЗСВ** | Криптография | КриптоПро CSP, УКЭП (ГОСТ Р 34.10), шифрование БД |
| **РСБ** | Регистрация и аудит | Логирование: journald → центральный лог (ClickHouse), хранение 5 лет |
| **АВЗ** | Антивирус | Kaspersky Endpoint Security (сертифицирован ФСТЭК) |

### Шаг 3: Модель угроз

| № | Угроза | Источник | Тип | Вероятность | Ущерб | Меры защиты |
|---|--------|----------|-----|-------------|-------|-------------|
| 1 | SQL-инъекция через веб-форму | Внешний (H1) | Техническая | Высокая | Критический | WAF, ORM, валидация |
| 2 | Перехват СМЭВ-трафика | Внешний (H1) | Сетевая | Средняя | Высокий | ViPNet, TLS 1.2+ |
| 3 | НСД администратора | Внутренний (H4) | Инсайдер | Низкая | Критический | Аудит, RBAC, УКЭП |
| 4 | Утечка БД (хищение дампа) | Внешний (H1) | Техническая | Средняя | Критический | Шифрование БД, СЗИ НСД |
| 5 | Социальная инженерия (звонок оператору) | Внешний (H1) | Социальная | Высокая | Средний | Обучение, инструкции |
| 6 | Вирус / шифровальщик | Внешний + Внутренний | Вредоносное ПО | Средняя | Высокий | Kaspersky, белые списки |

### Шаг 4: Документация

| № | Документ | Для чего |
|---|----------|----------|
| 1 | Техническое задание на систему | Исходные требования |
| 2 | Проектная документация (РД) | Описание архитектуры |
| 3 | Акт категорирования | Обоснование УЗ-2 |
| 4 | Модель угроз | Перечень угроз и их анализ |
| 5 | Модель нарушителя | Типы нарушителей |
| 6 | Частное ТЗ на СЗИ | Что конкретно внедряем |
| 7 | Рабочая документация СЗИ | Настройки, конфигурация |
| 8 | Инструкция администратора безопасности | Как администрировать |
| 9 | Инструкция пользователя | Как работать |
| 10 | Протокол испытаний | Результаты тестирования |
| 11 | Акт внедрения СЗИ | Что внедрено |
| 12 | Заключение по результатам испытаний | ФСТЭК |

### Шаг 5: Организационные меры

| № | Документ | Содержание |
|---|----------|-----------|
| 1 | Приказ о назначении ответственного за защиту ПД | Кто отвечает за безопасность ГИС |
| 2 | Инструкция по антивирусной защите | Как часто обновлять, что делать при вирусе |
| 3 | Положение о парольной защите | Длина 8+, смена раз в 90 дней, блокировка после 3 попыток |
| 4 | Регламент реагирования на инциденты | Кто, когда, как реагирует на утечку |
| 5 | План контроля и аудита | Ежеквартальная проверка |

## Критерии приемки

- [ ] Класс защищённости определён (УЗ-2) и обоснован
- [ ] Выбраны меры защиты по каждой группе (8 мер)
- [ ] Модель угроз содержит 6 типов угроз с вероятностью и ущербом
- [ ] Перечень документов: 12 позиций
- [ ] Организационные меры: 5 приказов/инструкций/регламентов

## Пример хорошего результата

**Фрагмент чек-листа:**

```
ГИС: «Справки-Онлайн»
УЗ: 2 (ПД: ФИО + паспорт, интернет, СМЭВ)

ИАФ: ЕСИА + УКЭП
УПД: RBAC (3 роли)
ОЛС: Astra Linux Special Edition «Орёл»
ЗИС: zVirt
ЗНИ: ViPNet + UserGate
ЗСВ: КриптоПро CSP
РСБ: journald → ClickHouse, 5 лет
АВЗ: Kaspersky Endpoint Security

Угроза #1: SQL-инъекция — WAF + ORM (высокая, критический)
Угроза #2: Перехват СМЭВ — ViPNet (средняя, высокий)
...
```

## Типичные ошибки

- **УЗ выбран неправильно.** Если ГИС обрабатывает ПД (паспорт, СНИЛС) — УЗ-2 минимум. УЗ-3 — только для систем без ПД. Ошибка: УЗ-1 для системы с паспортными данными.
- **Меры защиты формальные.** «ИАФ — обеспечить аутентификацию» — слишком абстрактно. Должно быть: «ИАФ — ЕСИА (SAML 2.0) + УКЭП для администраторов, пароль 8+ символов для операторов».
- **Модель угроз скопирована.** Типовая модель не учитывает СМЭВ-канал (перехват, подмена), веб-интерфейс (SQL-инъекции), Astra Linux (специфичные уязвимости).
- **Нет протокола испытаний.** ФСТЭК требует реальные протоколы: что тестировали, какие результаты, какие замечания.
- **Нет организационных мер.** Без приказа о назначении ответственного и инструкций — аттестацию не примут.
