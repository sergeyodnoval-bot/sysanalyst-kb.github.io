# Git — контроль версий

Git — самая популярная система контроля версий. Позволяет отслеживать изменения в коде, работать параллельно над разными фичами и откатывать ошибки.

## Основная идея

Git хранит **слепки (snapshots)** файлов, а не изменения (diffs). Каждый коммит — это полная копия состояния проекта с ссылкой на предыдущее состояние. Благодаря этому Git работает быстро даже на огромных проектах.

## Ключевые понятия

**Repository (репозиторий)** — папка проекта, в которой Git отслеживает изменения.

**Commit (коммит)** — точка сохранения. Содержит сообщение, автора, дату и ссылку на предыдущий коммит.

**Branch (ветка)** — независимая линия разработки. Ветка main — основная, feature-ветки — для новых фич.

**HEAD** — указатель на текущий коммит и ветку.

**Working directory — Staging — Repository.** Git-файл проходит три состояния: изменён (modified) → добавлен в индекс (staged) → закоммичен (committed).

## Базовые команды

| Команда | Что делает |
|---------|-----------|
| `git init` | Создать новый репозиторий |
| `git clone <url>` | Скопировать существующий репозиторий |
| `git add <file>` | Добавить файл в staging |
| `git commit -m "msg"` | Создать коммит |
| `git status` | Посмотреть состояние |
| `git log` | История коммитов |
| `git diff` | Показать незакоммиченные изменения |
| `git push` | Отправить коммиты на сервер |
| `git pull` | Забрать изменения с сервера |
| `git branch` | Список веток |
| `git checkout -b <name>` | Создать и переключиться на ветку |
| `git merge <branch>` | Влить ветку в текущую |

## Типичный рабочий процесс

1. `git checkout -b feature/new-api` — создать ветку
2. `git add . && git commit -m "feat: add new endpoint"` — коммитить изменения
3. `git push origin feature/new-api` — отправить на GitHub
4. Открыть Pull Request
5. После ревью: `git checkout main && git merge feature/new-api`

## Аналитику достаточно знать

- `git clone`, `git add`, `git commit`, `git push`, `git pull` — для работы с документацией
- `git status`, `git diff` — чтобы понять, что изменилось
- Концепцию веток и коммитов — чтобы говорить с разработчиками на одном языке

## Как Git помогает аналитику

- Версионирование документации (кто и когда изменил требование)
- Совместная работа над ADR и спецификациями
- Code review для документов (коллеги комментируют изменения)
- История решений: каждый ADR — отдельный коммит
