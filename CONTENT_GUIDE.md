# CONTENT_GUIDE.md — Руководство по созданию контента

Этот файл — источник истины для создания любого контента на сайте.
Читай его ПЕРЕД написанием каждой новой статьи, технологии или задачи.

## Общие принципы

### Стиль и тон
- **Уважай читателя.** Не сюсюкай, не упрощай до примитива. Читатель — умный взрослый человек, который может не знать терминов, но способен понять сложные вещи.
- **Будь понятным.** Если предложение можно упростить без потери смысла — упрощай.
- **Используй метафоры из реальной жизни.** Ресторан, кухня, стройка, почта — то, что знакомо каждому. Метафора должна объяснять, а не запутывать.
- **Конкретика > абстракция.** Не «система должна быть быстрой», а «страница должна загружаться за 500 мс».
- **Никакой воды.** Каждое предложение должно нести смысл. Если можно убрать — убирай.
- **Активный залог.** Не «данные обрабатываются сервером», а «сервер обрабатывает данные».

### Структура каждой единицы контента
1. **TL;DR** — 1-2 предложения сути. Читатель должен понять, о чём речь, за 10 секунд.
2. **Введение через метафору или пример** — зацепить читателя, связать с известным.
3. **Основной текст** — раскрыть тему, 300-800 слов для статей.
4. **Практическая часть** — «почему это важно для аналитика» или «как это применить».
5. **Ключевые термины** — 3-7 терминов с краткими пояснениями.
6. **Что дальше** — ссылки на связанные материалы.
7. **Проверь себя** — 2-3 вопроса с ответами для самопроверки.

### Язык
- Русский язык, без англицизмов без необходимости.
- Термины на английском — только если нет устоявшегося русского аналога (API, REST, BPMN).
- При первом упоминании термина на английском — давать русский перевод или пояснение.

### Форматирование
- **Заголовки:** H2 для основных секций, H3 для подпунктов.
- **Списки:** маркированные для перечислений, нумерованные для последовательностей.
- **Код:** в тройных обратных кавычках с указанием языка.
- **Жирный:** для ключевых терминов при первом упоминании.
- **Картинки:** обязательно с `alt` для доступности.
- **Таблицы:** для сравнений и структурированных данных.

### Расширение статей до образовательных материалов

При переработке существующих статей в полноценные обучающие материалы применяй шаблон:

1. **Learning Objectives** — секция «Для кого эта статья» с 3-5 bullet points «после прочтения вы:».
2. **Глоссарий терминов** — ключевые термины с пояснениями при первом упоминании, сводная таблица в начале или отдельная секция.
3. **Конкретные примеры из реальных продуктов** — называй имена платформ, сервисов, игр с цифрами (DAU, revenue, retention). Не «популярный мессенджер», а «Telegram (900M MAU, 2M каналов, MTProto)».
4. **Mermaid-диаграммы (2+)** — архитектурные схемы, sequence-диаграммы процессов, статусные модели, графы зависимостей.
5. **Таблицы с цифрами** — сравнения, метрики, нормы, cost-таблицы.
6. **Практический кейс** — разбор реальной ситуации: проблема → анализ → решение → результат.
7. **Quiz (5 вопросов)** — не 2-3, а 5 вопросов с ответами. Первые 2 — базовые, 3-4 — средние, 5 — на понимание trade-off.
8. **Ссылки на официальные мануалы** — в конце статьи таблица «Ссылки для самостоятельного изучения»: что, описание, URL.
9. **`estimated_time` — 30-45 мин** (отражает глубину материала; 15-20 мин оставь только для вводных статей).

## 📚 Статьи (`docs/`)

### Назначение
Основной учебный материал. Теория, концепции, how-to.

### Расположение
`docs/<category>/<id>.md`, где category — одна из:
- `basics` — основы для L0-L1
- `requirements` — работа с требованиями
- `modeling` — моделирование и нотации
- `integration` — интеграции и API
- `data` — данные и БД
- `architecture` — архитектура и проектирование
- `process` — процессы и методологии
- `soft` — софт-скиллы
- `specialization` — специализации

### Шаблон frontmatter

```yaml
---
id: <category>/<id>           # полный ID, например "basics/what-is-protocol"
title: "Полный заголовок статьи"
sidebar_label: "Короткое название для меню"
level: <1-10>                 # L0=1, L1=2, L2=3, L3=5, L4=7, L5=9
category: <enum>              # из списка выше
tags: [<tag1>, <tag2>]        # 3-7 тегов для поиска

# Граф зависимостей
prerequisites: []             # ID статей, которые нужно знать ДО этой
leads_to: []                  # ID статей, которые откроются ПОСЛЕ этой
related: []                   # связанные, но не обязательные темы

# Метаданные
estimated_time: <number>      # минуты на чтение (5-30)
difficulty: <1-5>             # субъективная сложность
audience: <string>            # опционально: "beginner", "junior", "middle"
---
```

### Правила написания
1. **Одна статья = одна идея.** Не пытайся охватить всё. Если тема большая — разбей на несколько статей с зависимостями.
2. **Начинай с «зачем».** Почему читателю это важно? Где он это встретит в работе?
3. **Давай контекст перед деталями.** Сначала общая картина, потом нюансы.
4. **Иллюстрируй примерами.** Каждый абстрактный тезис — с примером из практики.
5. **Указывай ограничения.** Не только «когда использовать», но и «когда НЕ использовать».
6. **Ссылайся на источники.** Если утверждение основано на стандарте или книге — ставь сноску (см. раздел «Цитирование»).

### Объём
- L0 (beginner): 300-500 слов
- L1-L2 (junior): 500-800 слов
- L3-L5 (middle-senior): 800-1500 слов

### Связи с другими типами контента
Статья может ссылаться на:
- Другие статьи (через `prerequisites`, `leads_to`, `related`)
- Технологии (через упоминание в тексте + поле `requires_articles` у технологии)
- Задачи (через упоминание + поле `requires_articles` у задачи)

**Актуализация `docLabels.ts`:** После добавления любого нового файла (статья, технология, задача) — добавь запись в `src/components/docLabels.ts`. Иначе в ссылках будет показываться имя файла вместо названия статьи/технологии/задачи. Держи файл синхронизированным с контентом.

В тексте статьи можно явно указывать:
```markdown
## Используемые технологии
- [OpenAPI](/tech/openapi) — для описания API
- [Postman](/tech/postman) — для тестирования

## Практика
Попробуйте применить знания в задаче: [Спроектировать REST API](/tasks/design-rest-api)
```

## 🔧 Технологии (`tech/`)

### Назначение
Справочник конкретных инструментов, стандартов, методологий, книг.

### Расположение
`tech/<id>.md` (без подкатегорий)

### Подтипы (`tech_type`)
- `technology` — технологический стек, протокол, платформа (Kafka, PostgreSQL, REST)
- `tool` — программный инструмент (Jira, Figma, Postman, Camunda)
- `standard` — формальный стандарт (ISO, RFC, ГОСТ, BABOK)
- `methodology` — методология/фреймворк (Scrum, DDD, C4, TOGAF)
- `book` — книга/фундаментальный труд (Коберн, Evans)

### Шаблон frontmatter

```yaml
---
id: <id>                      # уникальный ID, например "openapi"
title: "Полное название"
sidebar_label: "Короткое название"
type: technology                # фиксированное значение
tech_type: <enum>               # technology | tool | standard | methodology | book
category: <enum>                # та же таксономия, что у статей
tags: [<tag1>, <tag2>]

# Справочная информация
official_url: "<url>"           # обязательно для standard и book
github: "<url>"                 # опционально
vendor: "<name>"                # вендор/автор
license: "<license>"            # для technology/tool

# Версионность (для standard и book)
version: "<version>"            # версия стандарта/издания
published_date: <YYYY-MM-DD>    # дата публикации
superseded_by: <id>             # ID новой версии, если есть

# Для пометки "новых" технологий
first_seen: <year>              # год появления (бейдж "🆕 New" если ≤ 3 года)

# Связи
requires_articles: []           # ID статей, которые нужно знать
used_in_tasks: []               # ID задач, где применяется
alternatives: []                # ID альтернативных технологий

# Метаданные
difficulty: <1-5>
estimated_time: <number>        # минуты на прочтение справки

# Для standard и book — цитирование
citation_key: <key>             # короткий ключ для сносок (например, "iso25010")
citation_format: |              # готовый формат для вставки в сноски
  Полное библиографическое описание.
  URL: https://...
---
```

### Структура статьи о технологии

1. **Что это такое** — 1-2 предложения, суть.
2. **Для чего используется** — типичные сценарии применения.
3. **Ключевые концепции** — 3-5 основных понятий (для технологии/методологии).
4. **Когда использовать** — плюсы и сильные стороны.
5. **Когда НЕ использовать** — ограничения и слабые стороны.
6. **Альтернативы** — сравнение с аналогами (таблица).
7. **Как начать** — ссылки на документацию, туториалы.
8. **Ссылки** — официальные ресурсы.

### Особенности по подтипам

**Для `standard`:**
- Обязательно указать `official_url`, `version`, `published_date`
- В тексте — краткое содержание стандарта и как его применять
- Критика и ограничения стандарта
- Связанные стандарты

**Для `book`:**
- Автор, год, издательство, ISBN
- Ключевые идеи книги (3-5 пунктов)
- Для кого эта книга
- Как применять идеи на практике

**Для `tool`:**
- Что делает инструмент
- Основные возможности
- Плюсы и минусы
- Альтернативы
- Ссылка на бесплатную версию / trial

### Объём
- 300-600 слов (справочный формат, не учебник)

## 🎯 Задачи (`tasks/`)

### Назначение
Типовые рабочие сценарии аналитика — описание того, что аналитик реально делает на работе.
Это НЕ учебные упражнения с решениями, а справочник типовой деятельности.

### Расположение
`tasks/<id>.md` (без подкатегорий)

### Калибровка сложности (difficulty)

| difficulty | Уровень аналитика | Характер задачи |
|---|---|---|
| 1 | L1 Junior (0-1 год) | Воспроизвести по образцу, под присмотром |
| 2 | L2 Strong Junior (1-2 года) | Самостоятельно типовую задачу в знакомом контексте |
| 3 | L3 Middle (2-4 года) | Комплексная задача, несколько артефактов, новые вводные |
| 4 | L4 Senior (4-7 лет) | Неоднозначная задача, trade-off, менторство |
| 5 | L5 Lead/Architect (7+ лет) | Стратегическая задача, влияние на организацию |

**Правило:** `difficulty` задачи ≈ минимальный `level` аналитика, которому её можно поручить.

### Шаблон frontmatter

```yaml
---
id: <id>
title: "Название задачи"
sidebar_label: "Короткое название"
type: task                      # фиксированное значение
category: <enum>                # включая soft
difficulty: <1-5>               # калибруется по уровню аналитика
estimated_time: <number>        # минуты на выполнение в реальных условиях

# Что нужно изучить ПЕРЕД выполнением
requires_articles: []           # ID статей
requires_tech: []               # ID технологий/инструментов/стандартов

# Что должно получиться на выходе
deliverables: []                # список артефактов

# Контекст задачи (типовая ситуация на работе)
context: |
  Многострочный текст...

# Чек-лист шагов (как аналитик подходит к задаче)
steps: []

# Типичные ошибки
pitfalls: []

# Связь задача → задача (цепочки)
next_tasks: []                  # ID задач, которые логически следуют
previous_tasks: []              # опционально

# Связь с треками
part_of_tracks: []              # ID треков
---
```

### Структура статьи о задаче

1. **Контекст** — типовая ситуация на работе (из frontmatter, но расширенная).
2. **Цель задачи** — что должно получиться в итоге.
3. **Пошаговый подход** — как аналитик подходит к задаче (3-7 шагов).
4. **Критерии выполнения** — чек-лист, по которому можно оценить результат.
5. **Пример хорошего результата** — фрагмент артефакта или описание.
6. **Типичные ошибки** — что часто идёт не так и как этого избежать.
7. **Связанные материалы** — ссылки на статьи и технологии.

### Правила написания
1. **Реалистичный контекст.** Задача должна выглядеть как реальная рабочая ситуация, а не как учебное упражнение.
2. **Конкретные артефакты.** Не «сделать анализ», а «получить матрицу НФТ по ISO 25010».
3. **Пошаговый подход, а не решение.** Описывай, КАК аналитик думает и действует, а не готовый ответ.
4. **Учитывай неопределённость.** В реальных задачах не всё ясно с самого начала — это нормально.
5. **Включай коммуникацию.** Большинство задач требуют общения с людьми — это важно отразить.

### Объём
- 500-1000 слов

## 📖 Правило цитирования

Любое утверждение, основанное на стандарте, книге или исследовании, должно иметь сноску со ссылкой на источник.

### Формат сносок в тексте

```markdown
Согласно ISO 25010:2011, качество ПО оценивается по 8 характеристикам [^iso25010].

Как отмечает Коберн [^cuburn2001], use case должен иметь чёткие предусловия.

[^iso25010]: ISO/IEC 25010:2011 — Systems and software engineering...
             URL: https://www.iso.org/standard/35733.html
[^cuburn2001]: Alistair Coburn. Writing Effective Use Cases. Addison-Wesley, 2001.
```

### Обязательные поля в `tech/` для `standard` и `book`

- `official_url` — ссылка на оригинал
- `citation_key` — короткий ключ для сносок (например, `iso25010`)
- `citation_format` — готовый формат для вставки в сноски

### Когда цитирование обязательно
- Утверждения из стандартов (ISO, RFC, ГОСТ)
- Определения из книг (BABOK, Коберн, Evans)
- Статистика и исследования
- Прямые цитаты

### Когда можно без сноски
- Общепринятые знания (что такое HTTP, что такое БД)
- Личный опыт и мнения (с пометкой «на практике», «обычно»)
- Метафоры и примеры

## ✅ Чек-лист перед публикацией

Перед коммитом нового контента проверь:

### Для всех типов
- [ ] Frontmatter заполнен полностью, все обязательные поля на месте
- [ ] ID уникален и соответствует имени файла
- [ ] Теги заполнены (3-7 штук)
- [ ] Связи (prerequisites, requires_articles и т.д.) ссылаются на существующие ID
- [ ] TL;DR есть и содержит суть в 1-2 предложениях
- [ ] Структура соблюдена (заголовки, списки, примеры)
- [ ] Нет орфографических ошибок
- [ ] Все ссылки на другие материалы работают

### Для статей
- [ ] Объём соответствует уровню (300-1500 слов)
- [ ] Есть практическая часть «почему это важно для аналитика»
- [ ] Есть блок «Ключевые термины» (3-7 терминов)
- [ ] Есть блок «Проверь себя» с ответами
- [ ] Упомянуты связанные технологии и задачи (если применимо)

### Для технологий
- [ ] Выбран правильный `tech_type`
- [ ] Для `standard` и `book` заполнены `official_url`, `citation_key`
- [ ] Для новых технологий указан `first_seen`
- [ ] Есть сравнение с альтернативами
- [ ] Указаны сценарии «когда использовать» и «когда НЕ использовать»

### Для задач
- [ ] `difficulty` соответствует уровню аналитика
- [ ] Контекст реалистичный (как реальная рабочая ситуация)
- [ ] Есть конкретные артефакты в `deliverables`
- [ ] Пошаговый подход описан (не решение, а процесс)
- [ ] Есть типичные ошибки
- [ ] Связи с другими задачами через `next_tasks` (если применимо)

## 📝 Примеры готового контента

Используй как образец стиля и структуры:

**Статьи L0 (для «балерин»):**
- `docs/basics/what-is-software.md` — метафора «рецепт торта»
- `docs/basics/how-computer-works.md` — метафора «кухня»
- `docs/basics/who-is-system-analyst.md` — метафора «архитектор дома»
- `docs/basics/client-server-basics.md` — метафора «ресторан»
- `docs/basics/what-is-api.md` — метафора «меню ресторана»
- `docs/basics/it-roles.md` — метафора «стройка дома»
- `docs/basics/sa-day-in-life.md` — реальный рабочий сценарий

**Технологии:**
- `tech/openapi.md` (standard) — с версией, citation_key
- `tech/postman.md` (tool) — с vendor, license
- `tech/scrum.md` (methodology) — с first_seen
- `tech/browser.md`, `tech/figma.md`, `tech/confluence.md` — базовые инструменты

**Задачи:**
- `tasks/describe-system-to-friend.md` (difficulty: 1) — метафора «объясни ребёнку»
- `tasks/find-analyst-in-team.md` (difficulty: 1) — социальная задача

## 🚫 Чего НЕ делать

- **Не пиши «для галочки».** Каждый файл должен нести реальную пользу.
- **Не копируй из других источников дословно.** Пересказывай своими словами, указывай источники.
- **Не используй жаргон без пояснения.** Если термин нужен — объясни его при первом упоминании.
- **Не делай статьи-«простыни».** Если больше 1500 слов — разбей на несколько.
- **Не создавай «висячие» материалы.** Каждая статья/технология/задача должна иметь хотя бы одну связь с другими.
- **Не решай задачи за читателя.** В задачах описывай процесс, а не готовый ответ.
- **Не игнорируй связи.** Если материал связан с другим — укажи это в frontmatter.

## 🔄 Процесс создания нового контента

1. **Определи тип:** статья, технология или задача?
2. **Определи место в графе:**
   - Какие prereq нужны?
   - Куда это ведёт?
