# PROJECT_CONTEXT.md — Концепция проекта "База знаний системного аналитика"

## Что это такое
Статический сайт-энциклопедия, содержащий все знания, необходимые системному аналитику.
Ключевая особенность — **граф знаний**: статьи связаны между собой зависимостями,
что позволяет видеть предметную область как единую карту и двигаться от простого к сложному.

## Зачем это нужно (проблема)
Системному аналитику нужно знать много разных областей: требования, моделирование,
интеграции, архитектуру, БД, софты. В интернете информация разбросана, нет понимания:
- С чего начать новичку?
- Что я упустил, будучи middle-специалистом?
- Как связаны между собой BPMN, UML и C4?

Наш сайт решает эти три проблемы через визуализацию зависимостей и треки обучения.

## Целевая аудитория
1. **Новичок (Junior)** — хочет войти в профессию, ищет структурированный путь.
2. **Middle/Senior** — использует как энциклопедию, точечно изучает смежные темы.
3. **Руководитель** — формирует планы развития для команды.

## Ключевые сценарии использования
1. **Определить свой уровень.** Пройти тест → получить рекомендацию стартовой точки.
2. **Изучить тему последовательно.** Выбрать трек (например, "Junior SA") и идти по нему.
3. **Найти конкретную статью.** Через поиск или карту знаний.
4. **Увидеть связи.** Открыть статью и понять, что нужно знать до неё и что она открывает.

## Продуктовые принципы
- **Открытость.** В MVP весь контент бесплатен и без авторизации.
- **Статичность.** Сайт — это статический HTML, хостится бесплатно, грузится мгновенно.
- **Расширяемость.** Архитектура позволяет потом добавить:
  - Платный контент (через поле `access` в frontmatter)
  - Авторизацию и прогресс пользователя
  - Сертификационные экзамены
  - Персональные рекомендации на основе ML
- **Контент — король.** Добавление новой статьи = создание одного .md файла.
  Никаких миграций БД, никаких админок в MVP.

## Что НЕ является частью проекта (в MVP)
- Форум / комментарии / соц. функции
- Авторизация и личные кабинеты
- Платежи и подписки
- Мобильное приложение
- Мультиязычность (только русский)
- Видеоконтент собственного производства (только встраивание YouTube)

## Успех проекта измеряется
- Количество статей в графе (цель MVP: 70+ штук с валидными связями)
- Глубина просмотра (пользователь читает >1 статьи за сессию)
- Прохождение теста на уровень до конца
- SEO-трафик по запросам "системный аналитик + [тема]"

## Треки обучения
Треки определяются в `src/data/tracks-data.ts` как TypeScript-объекты с типом `TrackDef`.
Каждый трек состоит из этапов (`stages`), а этапы — из элементов (`items`) трёх типов:
`article`, `tech`, `task`. 

### Структура трека
```typescript
interface TrackDef {
  id: string;          // уникальный ID
  title: string;       // название трека
  description: string; // описание
  stages: TrackStage[]; // этапы
}
interface TrackStage {
  title: string;
  description: string;
  items: TrackItem[];   // {type: 'article'|'tech'|'task', id: string, folder?: string}
}
```

### Сопутствующие данные
- `itemLabels` — маппинг `{type → {id → label}}` для отображения названий в треках
- `itemColors` — цвет для каждого типа (`article`, `tech`, `task`)
- `itemIcons` — иконка (односимвольная) для каждого типа
- `trackItemPath(item)` — функция, генерирующая URL: `/docs/{folder}/{id}`, `/tech/{id}` или `/tasks/{id}`
- Массив `allTracks` экспортируется и используется на странице `/tracks`

## Статьи (контент)
Статьи организованы по папкам-категориям в `docs/`. Всего 8 категорий:
- `basics/` — основы IT (компьютер, ОС, сеть, программирование, профессия SA)
- `requirements/` — требования (user stories, BDD, стейкхолдеры)
- `modeling/` — моделирование (BPMN, UML, C4, DFD, ERD, State, Component)
- `integration/` — интеграции (REST, OpenAPI, SOAP, брокеры, EDA, Event Storming)
- `data/` — данные (SQL, нормализация, NoSQL, ER-моделирование)
- `architecture/` — архитектура (слои, монолит vs микросервисы, CQRS, SOLID, ADR, EDA)
- `process/` — процессы (SDLC, Scrum, Kanban, estimation, ретроспективы)
- `soft/` — soft skills (критическое мышление, деловая переписка)

### Frontmatter статьи
Обязательные поля: `id`, `title`, `sidebar_label`, `level` (1-10), `category`, `tags`, 
`prerequisites` (массив ID формата `category/article-id`), `leads_to`, `related`, 
`estimated_time`, `difficulty` (1-5).

## Swizzle-компонент Layout
Единый swizzle в `src/theme/DocItem/Layout/index.tsx` обрабатывает все три типа контента:
- `pluginId === 'default'` (статьи) — блоки UsedTechnologies, PracticalTasks
- `pluginId === 'tech'` (технологии) — блоки AppliedInTasks, AlternativesList, CitationBlock
- `pluginId === 'tasks'` (задачи) — блоки NextTasksList, TypicalPitfalls
- Для всех типов — PrerequisitesList, RequiredKnowledge, NextStepsList, TrackNav, RelatedTopics

## Плагин графа знаний
Определён inline в `docusaurus.config.ts` как `knowledgeGraphPlugin()`. Читает frontmatter
из `docs/**/*.md`, `tech/**/*.md` и `tasks/**/*.md`. Строит узлы и рёбра 5 типов
(prerequisite, enables, required_for, next_task, alternative). Данные доступны через
`useAllPluginInstancesData('knowledge-graph')`.

## Архитектурные решения (высокоуровнево)
- **Docusaurus 3** как основа — из коробки есть docs, sidebar, поиск, версии.
- **Контент = Markdown с frontmatter** — метаданные статьи (зависимости, уровень, теги)
  живут рядом с текстом, в том же репозитории.
- **Граф строится автоматически** на этапе сборки из frontmatter всех файлов в docs/, tech/, tasks/.
- **Тесты = JSON-файлы** в папке `/tests`, логика на клиенте (без бэкенда).
- **Прогресс пользователя** (в будущем) — localStorage в MVP, сервер потом.

## Архитектурные решения (Этап 2 — технологии и задачи)

### Три типа контента
| Тип | Папка | URL | Плагин | Ключевое поле |
|-----|-------|-----|--------|--------------|
| Статья | `docs/` | `/docs/[id]` | default (preset) | — |
| Технология | `tech/` | `/tech/[id]` | plugin-content-docs (id: tech) | `tech_type` |
| Задача | `tasks/` | `/tasks/[id]` | plugin-content-docs (id: tasks) | `type: task` |

### Подтипы tech_type
- `standard` — стандарты/спецификации (OpenAPI). Имеют версию, дату публикации, citation_key
- `tool` — инструменты (Postman). Имеют vendor
- `methodology` — методологии (Scrum). Имеют official_url
- `book` — книги (зарезервировано, пока не используется)

### Версионность для standard
Поля `version`, `published_date`, `first_seen`, `citation_key` обязательны для стандартов.
`citation_key` используется для форматирования сносок в статьях.

### first_seen для бейджа "New"
Технология считается "новой", если `currentYear - firstSeen <= 3`. Бейдж 🆕 NEW
отображается на карте знаний и в блоках UsedTechnologies.

### next_tasks для задач
Поле `next_tasks` в задачах указывает логически следующие задачи. На карте знаний
отображается как ребро типа `next_task` (жирная фиолетовая стрелка).

### Калибровка difficulty по уровню аналитика
| Значение | Уровень |
|----------|---------|
| 1 | Trainee / Intern |
| 2 | Junior / Strong Junior |
| 3 | Middle |
| 4 | Senior |
| 5 | Lead / Architect |

### Типы связей графа
| Тип | Цвет/стиль | Направление | Источник |
|-----|-----------|-------------|----------|
| prerequisite | серая сплошная | статья → статья | `prerequisites` |
| enables | зелёная пунктирная | статья → технология | `requires_articles` |
| required_for | оранжевая штрих-пунктирная | технология → задача | `requires_tech` |
| next_task | фиолетовая жирная | задача → задача | `next_tasks` |
| alternative | серая без стрелки | технология → технология | `alternatives` |
| leads_to | — | статья → статья | `leads_to` |
