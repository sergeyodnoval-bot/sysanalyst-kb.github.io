# Переговоры и работа с возражениями

Аналитик постоянно ведёт переговоры: согласование требований, защиты оценки, приоритеты с бизнесом, компромиссы с разработкой.

## Переговоры в работе аналитика

| Ситуация | С кем | Интересы |
|----------|-------|----------|
| Согласование SRS | Бизнес-заказчик | Хочет всё → аналитик объясняет стоимость |
| Приоритеты требований | PO / Product Manager | Хочет фичи → аналитик оценивает сложность |
| Технические ограничения | Разработчик | Говорит «так нельзя» → ищем альтернативу |
| Изменение требований | Стейкхолдер | Хочет добавить → аналитик оценивает impact |
| Защита оценки | PM / Заказчик | «Почему так долго?» → аналитик декомпозирует |

## Модели переговоров

### 1. Harvard Method (Principled Negotiation)

| Принцип | Вопрос |
|---------|--------|
| **People** — отделяйте людей от проблемы | «Я понимаю вашу позицию, давайте посмотрим на факты» |
| **Interests** — фокус на интересах, а не позициях | «Что вам важно: срок или функциональность?» |
| **Options** — ищите взаимовыгодные варианты | «Можно сделать базовую версию сейчас, остальное — в Release 2» |
| **Criteria** — опирайтесь на объективные критерии | «Давайте посмотрим на приоритет по RICE» |

### 2. BATNA (Best Alternative To a Negotiated Agreement)

Что вы сделаете, если не договоритесь?

**Пример:**
- Стейкхолдер хочет новую фичу за 20 дней
- Вы не можете (сроки горят)
- BATNA: «Я выношу это на prioritization board. Если комитет одобрит — сдвигаем другие задачи»

### 3. YES-AND (Импровизация)

Не «нет, это невозможно», а «да, и вот что для этого нужно»:

```
Стейкхолдер: «Давайте добавим чат с поддержкой в этот релиз»
Аналитик:    «Да, хорошая идея. Это потребует интеграции с мессенджером,
              примерно 15 дней. Если сдвинем каталог — успеваем. Что важнее?»
```

## Работа с возражениями

### Техника: Feel — Felt — Found

| Шаг | Фраза |
|-----|-------|
| **Feel** | «Я понимаю, как вы себя чувствуете» |
| **Felt** | «Другие заказчики тоже так чувствовали» |
| **Found** | «Но они обнаружили, что…» |

### Типовые возражения

| Возражение | Ответ |
|-----------|-------|
| «Почему так долго?» | «Давайте я покажу декомпозицию: 5 дней на интеграцию + 3 дня на тесты» |
| «Это же просто!» | «Кажется простым, но есть нюансы: [список подзадач]» |
| «В прошлый раз сделали за 2 дня» | «Та задача была проще: не было интеграции с CRM» |
| «Зачем это нужно?» | «Если мы это не сделаем, то [последствие]» |
| «А давайте без документации» | «Без документации тестировщик не сможет проверить, и баги уйдут в прод» |

## Трудные переговоры

### Deadline pressure (давление сроками)

```
Заказчик: «Нужно до пятницы, иначе проект закроют»
Аналитик:  «Понимаю срочность. Давайте выберем: делаем 50% функциональности
            с базовым тестированием или 100% с риском багов. Что выбираете?»
```

### Scope creep (разрастание требований)

```
Стейкхолдер: «И ещё: добавим интеграцию с сервисом доставки»
Аналитик:   «Отличная идея. Я добавляю в backlog с приоритетом. 
             Это повлияет на срок каталога — вы готовы сдвинуть каталог?»
```

## Типичные ошибки

- **Сказать «да» и провалить сроки** — лучше честно отказаться
- **Сказать «нет» без альтернативы** — «нет» сжигает мосты
- **Эмоциональная реакция** — «вы ничего не понимаете» → теряете доверие
- **Переговоры как война** — победитель/проигравший → проигрывают оба
- **Нет BATNA** — если не договоритесь, что будете делать?
