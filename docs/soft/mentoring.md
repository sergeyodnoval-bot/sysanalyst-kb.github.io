# Менторство для аналитиков

Менторство — это передача опыта от более опытного аналитика менее опытному. В отличие от коучинга, ментор **делится своими знаниями и опытом**, а не просто задаёт вопросы.

## Ментор vs Коуч

| | Ментор | Коуч |
|--|--------|------|
| **Роль** | Эксперт, делится опытом | Фасилитатор, задаёт вопросы |
| **Фокус** | Профессиональные навыки | Личное развитие |
| **Формат** | «Я делал так — вот что получилось» | «А что ты думаешь? А какие варианты?» |
| **Результат** | Конкретный совет, решение | Осознание, план действий |

## Зачем становиться ментором

- **Рост уровня** — объясняя, учишься сам (Feynman technique)
- **Лидерство** — менторство необходимо для Senior+
- **Команда** — менти быстрее выходит на уровень
- **Карьера** — менторство = аргумент для повышения до Senior / Lead

## Форматы менторства

| Формат | Как часто | Длительность | Описание |
|--------|-----------|-------------|----------|
| **1:1 встречи** | 1 раз в 1–2 недели | 30–45 мин | Разбор конкретных вопросов |
| **Shadowing** | По запросу | День / спринт | Менти смотрит, как ментор работает |
| **Code/artefact review** | После каждой задачи | 15–30 мин | Рецензия на SRS, диаграмму, тест-кейс |
| **Pair analysis** | На сложной задаче | 1–2 часа | Делают вместе: одна задача — два аналитика |

## Структура 1:1 встречи

```
  ┌──────────────────────────────────┐
  │  1. Status (5 мин)               │
  │  Что сделано, что нет            │
  ├──────────────────────────────────┤
  │  2. Topic (15–20 мин)            │
  │  Главный вопрос / проблема       │
  ├──────────────────────────────────┤
  │  3. Skill (10 мин)               │
  │  Короткий урок / чек-лист        │
  ├──────────────────────────────────┤
  │  4. Feedback (5 мин)             │
  │  Что получилось / что улучшить   │
  ├──────────────────────────────────┤
  │  5. Next steps (5 мин)           │
  │  Цель до следующей встречи       │
  └──────────────────────────────────┘
```

## Что обсуждать

| Тема | Примерные вопросы |
|------|------------------|
| **Requirements** | Как писать AC? Как собирать требования? |
| **Modelling** | Какую диаграмму выбрать? Как показать интеграцию? |
| **Stakeholders** | Как убедить бизнес? Как работать с конфликтами? |
| **Career** | Что учить? Куда расти? Что почитать? |
| **Soft skills** | Как говорить «нет»? Как проводить интервью? |

## Как давать feedback

### SBI (Situation — Behavior — Impact)

| Элемент | Описание | Пример |
|---------|----------|--------|
| **S**ituation | Когда и где | «На вчерашней встрече с заказчиком…» |
| **B**ehavior | Что сделал | «…ты сказал "да" на новую фичу, не оценив сроки» |
| **I**mpact | К чему привело | «…теперь разработчики не успевают к релизу» |

### Правила feedback

- **Конкретно** — не «ты плохо пишешь SRS», а «в FR-5 не указано исключение»
- **Своевременно** — не через месяц, а на следующий день
- **Двусторонне** — спросите «а что я могу делать иначе как ментор?»
- **Баланс** — 3 позитивных + 1 развивающий

## Типичные ошибки ментора

- **Дать готовое решение** — «сделай вот так» вместо «давай подумаем вместе»
- **Слишком много советов** — менти перегружен, ничего не применяет
- **Нет структуры** — встреча превращается в светскую беседу
- **Нет follow-up** — договорились, но на следующей встрече не проверили
- **Критика без поддержки** — только указал на ошибки, не сказал, как исправить
