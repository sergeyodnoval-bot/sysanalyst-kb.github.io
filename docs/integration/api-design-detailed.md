# Проектирование REST API — детально

Спроектировать API — значит договориться о контракте между системами до того, как написана первая строка кода. Хороший дизайн API экономит недели переделок и делает интеграцию предсказуемой.

## Процесс проектирования

Проектирование REST API — не линейная последовательность шагов, а итеративный процесс. На практике аналитик проходит несколько этапов, возвращаясь к предыдущим по мере уточнения требований.

### 1. Выделение ресурсов

Ресурс — это сущность предметной области, с которой работают пользователи системы. Не каждая таблица в базе становится ресурсом: ресурс должен быть значимым для бизнеса.

**Как выделить ресурсы:**

- Выпишите ключевые существительные из требований (заказ, пользователь, товар, платёж)
- Отбросьте те, что нужны только для внутренней логики
- Сгруппируйте по иерархии: что кому принадлежит
- Проверьте, что на каждый ресурс есть хотя бы один бизнес-сценарий

Пример для интернет-магазина:

```
/users           — покупатели и продавцы
/users/{id}/orders  — заказы конкретного пользователя
/products        — товары
/categories      — категории товаров
/payments        — платежи
```

### 2. Определение действий над ресурсами

Для каждого ресурса определите, какие операции нужны бизнесу. Сопоставьте их с HTTP-методами:

| Метод | Действие | Идемпотентность | Безопасность |
|-------|----------|-----------------|--------------|
| GET | Прочитать | ✓ | ✓ |
| POST | Создать | ✗ | ✗ |
| PUT | Полностью заменить | ✓ | ✗ |
| PATCH | Частично изменить | ✗ | ✗ |
| DELETE | Удалить | ✓ | ✗ |

**Идемпотентность** — повторный запрос с теми же данными даёт тот же результат. GET /users/42 всегда вернёт одни и те же данные (пока ресурс не изменится). PUT /users/42 с тем же телом — ресурс останется в том же состоянии.

**Безопасность** — запрос не меняет состояние сервера. GET можно вызывать сколько угодно.

### 3. Проектирование URI

URI — это адрес ресурса, по которому к нему обращаются.

**Правила хорошего URI:**

- Только существительные во множественном числе: `/orders`, не `/order` или `/getOrders`
- Иерархия через вложенность: `/users/42/orders/7`
- Параметры для фильтрации, а не часть пути: `/orders?status=active`, не `/orders/active`
- kebab-case для составных: `/order-items`
- Версия в начале пути: `/v1/users`

**Чего избегать:**

- Глаголов в пути: `/getUsers`, `/createOrder` — используйте HTTP-методы
- Фильтрации через тело POST: для чтения данных используйте GET
- Слишком глубокой вложенности: `/a/b/c/d/e` — пересмотрите иерархию

### 4. Проектирование запросов и ответов

**Тело запроса (request body):**

POST и PUT всегда с телом в JSON. PATCH тоже, но содержит только изменяемые поля.

```json
POST /api/users
{
  "name": "Иван Иванов",
  "email": "ivan@example.com",
  "role": "analyst"
}
```

**Тело ответа:**

Все ответы в едином формате. Поля — в camelCase. Даты — в ISO 8601.

```json
{
  "id": 42,
  "name": "Иван Иванов",
  "email": "ivan@example.com",
  "role": "analyst",
  "createdAt": "2025-03-15T10:30:00Z"
}
```

**Пагинация:**

Если ресурсов много — возвращайте порционно.

```json
GET /api/users?page=2&limit=20
---
{
  "data": [ ... ],
  "pagination": {
    "page": 2,
    "limit": 20,
    "total": 150,
    "totalPages": 8
  }
}
```

**Фильтрация и сортировка:**

Параметры запроса для фильтрации, сортировки и поиска.

```
GET /api/users?role=analyst&sort=name&order=asc
GET /api/orders?status=pending&createdAfter=2025-01-01
```

### 5. Обработка ошибок

Ошибки должны быть предсказуемыми и информативными.

```json
HTTP/1.1 422 Unprocessable Entity
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Email already exists",
    "details": [
      { "field": "email", "reason": "unique_violation" }
    ]
  }
}
```

**Карта статус-кодов для REST API:**

| Код | Когда использовать |
|-----|-------------------|
| 200 OK | Успешный GET, PUT, PATCH |
| 201 Created | Успешный POST (создание) |
| 204 No Content | Успешный DELETE |
| 400 Bad Request | Невалидный запрос (ошибка в теле) |
| 401 Unauthorized | Нет аутентификации |
| 403 Forbidden | Нет прав на операцию |
| 404 Not Found | Ресурс не найден |
| 409 Conflict | Конфликт (дубликат, устаревшая версия) |
| 422 Unprocessable Entity | Валидация данных не пройдена |
| 429 Too Many Requests | Превышен лимит запросов |
| 500 Internal Server Error | Непредвиденная ошибка сервера |

## Роль аналитика

На практике аналитик редко проектирует API в одиночку. Обычно это совместная работа с разработчиками и архитектором. Ваша задача — не написать идеальный URI, а обеспечить, чтобы API отражал бизнес-требования и был понятен всем участникам команды.

**Что делает аналитик:**

1. Собирает требования к интеграции: какие данные, как часто, в каком формате
2. Описывает ресурсы и их атрибуты в терминах предметной области
3. Проектирует сценарии запросов и ответов (happy path + ошибки)
4. Согласовывает контракт с командами-потребителями
5. Документирует решения в OpenAPI

## Ключевые термины

- **Ресурс** — сущность, к которой обращаются через API (пользователь, заказ, товар)
- **Идемпотентность** — свойство операции: повторный запрос не меняет состояние
- **Статус-код** — трёхзначный код в ответе HTTP, указывающий на результат операции
- **Пагинация** — разбивка списка ресурсов на страницы
- **Контракт API** — соглашение о форматах запросов и ответов между клиентом и сервером
