# Версионирование API

API живёт дольше, чем любая single-page app. Через год после запуска появятся новые требования, которые потребуют изменить формат данных или добавить обязательные поля. Без версионирования каждое такое изменение сломает клиентов, которые уже используют API.

## Когда нужно версионирование

Версионировать API нужно, когда изменение **необратимо** и **ломает обратную совместимость**:

- Удаление поля из ответа
- Изменение типа поля (строка → число)
- Добавление обязательного параметра
- Переименование ресурса или эндпоинта
- Изменение логики статус-кодов

**Когда можно обойтись без версионирования:**

- Добавление нового поля в ответ (клиенты его просто игнорируют)
- Добавление нового эндпоинта
- Добавление необязательного параметра

Правило эволюции API: **Будьте снисходительны к тому, что принимаете, и строги к тому, что отдаёте** (Postel's law, он же Robustness Principle).

## Стратегии версионирования

### URL-based (в пути)

Наиболее распространённый подход. Версия указывается в URI:

```
/api/v1/users
/api/v2/users
```

**Плюсы:** явно видно в логах и документации, легко тестировать.

**Минусы:** захламляет URI, усложняет маршрутизацию, клиенты жёстко привязаны к версии в коде.

### Header-based (в заголовке)

Версия передаётся в HTTP-заголовке:

```
Accept: application/vnd.myapp.v2+json
```

Или кастомный заголовок:

```
X-API-Version: 2
```

**Плюсы:** URI остаётся чистым, версия не видна в коде клиента.

**Минусы:** не видно в логах без дополнительного парсинга, сложнее тестировать вручную через браузер.

### Parameter-based (в параметре запроса)

Версия как query-параметр:

```
/api/users?version=2
```

**Плюсы:** просто реализовать.

**Минусы:** смешивает версионирование с бизнес-параметрами, легко забыть указать.

## Какую стратегию выбрать

| Критерий | URL | Header | Parameter |
|----------|-----|--------|-----------|
| Наглядность | ★★★★★ | ★★★ | ★★★★ |
| Чистота URI | ★★ | ★★★★★ | ★★★ |
| Простота реализации | ★★★★★ | ★★★ | ★★★★ |
| Web-friendly | ★★★★★ | ★★★ | ★★★★ |

Для внутренних API чаще выбирают URL-версионирование — оно проще и нагляднее. Для публичных API, где важна чистота контракта, — header-based.

## Как долго поддерживать старые версии

Жизненный цикл версии API:

1. **Релиз** новой версии с объявлением даты прекращения поддержки старой
2. **Период сосуществования** — обе версии работают параллельно
3. **Deprecation** — старая версия помечена как устаревшая, но ещё работает
4. **Sunset** — старая версия отключена

Рекомендуемые сроки: **6–12 месяцев** для внутренних API, **18–24 месяца** для публичных.

**Как уведомлять клиентов:**

- Заголовок `Deprecation: true` в ответах старой версии
- Заголовок `Sunset: Sat, 31 Dec 2025 23:59:59 GMT`
- Письменное уведомление за 3 и 1 месяц до отключения

```http
HTTP/1.1 200 OK
Deprecation: true
Sunset: Sat, 31 Dec 2025 23:59:59 GMT
```

## Роль аналитика

Системный аналитик отвечает за то, чтобы решение о версионировании было принято осознанно:

- Фиксирует в документации, какая стратегия выбрана и почему
- Описывает в требованиях, какие изменения считаются ломающими
- Согласует lifecycle версий с командами-потребителями
- Добавляет заголовки deprecation/sunset в спецификацию OpenAPI

## Ключевые термины

- **Backward compatibility** — обратная совместимость: новая версия не ломает старых клиентов
- **Deprecation** — пометка функциональности как устаревшей
- **Sunset** — дата полного отключения старой версии
- **Postel's law** — «будь снисходителен к тому, что принимаешь»
