# Аутентификация и авторизация в API

Любой API, который работает не в изолированной внутренней сети, нуждается в защите. Даже внутренние API обычно требуют хотя бы базовой аутентификации, чтобы знать, кто и какие данные запрашивает.

**Аутентификация** — проверка, что клиент — это тот, за кого себя выдаёт. **Авторизация** — проверка, что клиент имеет право на запрошенное действие.

## Basic Auth

Самый простой способ: логин и пароль передаются в заголовке `Authorization`.

```
Authorization: Basic base64(login:password)
```

**Плюсы:** предельно просто, встроен в HTTP.

**Минусы:** пароль передаётся при каждом запросе (даже по HTTPS это риск), нет контроля сессий, сложно отозвать доступ.

**Где использовать:** только для прототипов и внутренних инструментов. Не для продакшена.

## Token-based (JWT)

Клиент получает токен, который передаёт в каждом запросе:

```
Authorization: Bearer eyJhbGciOiJIUzI1NiJ9...
```

**JWT (JSON Web Token)** — самодостаточный токен, который содержит информацию о пользователе, правах и сроке действия. Сервер может проверить токен без обращения к базе.

**Структура JWT:**

```
header.payload.signature
```

- **header** — тип токена и алгоритм подписи
- **payload** — данные (user_id, roles, exp, iat)
- **signature** — подпись, подтверждающая подлинность

**Плюсы:** stateless (серверу не нужно хранить сессии), удобен для микросервисов.

**Минусы:** сложно отозвать (токен живёт до истечения срока), размер больше, чем session ID.

## OAuth 2.0

Промышленный стандарт делегирования доступа. Используется, когда одно приложение хочет получить доступ к данным пользователя в другом приложении.

**Участники OAuth 2.0:**

- **Resource Owner** — пользователь, владелец данных
- **Client** — приложение, которое запрашивает доступ
- **Authorization Server** — сервер, выдающий токены
- **Resource Server** — сервер с данными (API)

**Типовой flow (Authorization Code):**

```
1. Client → Authorization Server: запрос кода
2. Пользователь подтверждает доступ в браузере
3. Authorization Server → Client: authorisation code
4. Client → Authorization Server: code + client_secret → access_token
5. Client → Resource Server: запрос с access_token
```

**Когда использовать:** сторонние интеграции, Single Sign-On (SSO), доступ к API от имени пользователя.

## API Keys

Простой ключ, который передаётся в заголовке или параметре запроса:

```
X-API-Key: abc123def456
```

**Плюсы:** просто для сервер-серверной интеграции, легко генерировать и отзывать.

**Минусы:** нет разграничения по пользователям, ключ живёт пока не отозван.

**Где использовать:** сервер-серверные интеграции, публичные API с фиксированным доступом.

## Сравнение методов

| Метод | Сложность | Безопасность | Stateless | Отзыв доступа |
|-------|-----------|-------------|-----------|--------------|
| Basic Auth | ★☆☆ | ★☆☆ | ✓ | ✗ |
| JWT | ★★☆ | ★★★ | ✓ | ✗ (до expiry) |
| OAuth 2.0 | ★★★ | ★★★★★ | ✓ | ✓ |
| API Key | ★☆☆ | ★★☆ | ✓ | ✓ |

## Роль аналитика

Аналитик не выбирает алгоритм шифрования, но определяет сценарии доступа:

- Какие роли есть в системе и какие операции им доступны
- Какие данные публичные, а какие требуют аутентификации
- Нужен ли клиентам доступ от имени пользователя (OAuth) или достаточно серверного ключа
- Как часто истекают токены и как происходит их обновление

## Ключевые термины

- **JWT** — JSON Web Token, формат токена с цифровой подписью
- **OAuth 2.0** — протокол делегирования доступа (не путать с логином через соцсети)
- **Access Token** — токен для доступа к ресурсам
- **Refresh Token** — токен для получения нового access token (обычно живёт дольше)
- **Bearer Token** — токен, который предъявляется в заголовке Authorization
