# Документирование API со спецификацией OpenAPI

OpenAPI (ранее известный как Swagger) — это стандарт описания REST API в формате YAML или JSON. Спецификация OpenAPI позволяет описать все эндпоинты, форматы данных, параметры и способы аутентификации.

## Структура OpenAPI-спецификации

Базовый файл спецификации состоит из нескольких секций:

```yaml
openapi: 3.0.3
info:
  title: User API
  version: 1.0.0
paths:
  /users:
    get:
      summary: Получить список пользователей
      responses:
        '200':
          description: Успешный ответ
components:
  schemas:
    User:
      type: object
      properties:
        id: integer
        name: string
```

### Основные секции

- **openapi** — версия спецификации
- **info** — метаданные API
- **servers** — адреса серверов
- **paths** — эндпоинты и их методы
- **components** — переиспользуемые схемы и параметры
- **security** — схемы аутентификации

## Инструменты OpenAPI

### Swagger UI

Интерактивная документация, которая генерируется из спецификации. Позволяет отправлять запросы прямо из браузера.

### Swagger Editor

Редактор с подсветкой синтаксиса и валидацией. Удобен для написания спецификаций.

### Code Generator

Генерирует код клиента или сервера по спецификации. Поддерживает Java, Python, TypeScript и другие языки.

## Преимущества для команды

- **Единый источник правды** — спецификация является контрактом между командами
- **Автоматизация** — генерация кода, тестов, документации
- **Раннее обнаружение ошибок** — валидация на этапе проектирования
- **Интеграция с тестированием** — импорт в Postman, Insomnia

## Роль аналитика

Системный аналитик использует OpenAPI для:
- Документирования требований к API
- Согласования контрактов с разработчиками
- Создания примеров запросов и ответов
- Проверки полноты спецификации

### Практические рекомендации

1. Начинать с описания схем данных (components/schemas)
2. Описывать все возможные коды ответов (не только 200)
3. Добавлять примеры (example) для каждого поля
4. Использовать `$ref` для переиспользования схем
5. Документировать ошибки в стандартном формате
