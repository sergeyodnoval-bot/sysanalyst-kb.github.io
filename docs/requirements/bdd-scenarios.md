# BDD-сценарии и критерии приёмки

:::note
BDD (Behavior-Driven Development) — подход к описанию поведения системы через сценарии формата «Дано / Когда / Тогда» (Given / When / Then). Это язык, понятный всем участникам команды: от заказчика до разработчика и тестировщика.
:::

В статье «User Stories» мы разобрались, как формулировать функциональные требования. Но как сделать эти требования настолько чёткими, чтобы их можно было автоматически тестировать? Как объяснить разработчику и тестировщику, что именно нужно проверить? Ответ — BDD-сценарии.

## Формат Given / When / Then

BDD-сценарий состоит из трёх частей:

### Given (Дано) — контекст

Исходное состояние системы, предварительные условия.

*Given пользователь авторизован в системе*
*Given в корзине есть 3 товара на сумму 4500 рублей*

### When (Когда) — действие

Событие или действие пользователя.

*When пользователь применяет промокод "SAVE10"*
*When пользователь нажимает «Оформить заказ»*

### Then (Тогда) — результат

Ожидаемое поведение системы.

*Then сумма заказа уменьшается на 10%*
*Then заказ получает статус «Оформлен»*
*Then пользователь видит сообщение «Заказ отправлен на кухню»*

## Примеры BDD-сценариев

### User story: скидка по промокоду

*Scenario: Применение промокода при оформлении заказа*
*Given пользователь авторизован в системе*
*And в корзине есть 3 товара на сумму 4500 рублей*
*When пользователь вводит промокод "SAVE10"*
*Then сумма заказа пересчитывается с учётом скидки 10%*
*And новая сумма отображается (4050 рублей)*
*And пользователь видит сообщение «Промокод применён»*

### User story: фильтрация товаров

*Scenario: Фильтрация по цене*
*Given пользователь открыл страницу каталога*
*When пользователь устанавливает фильтр «Цена от 1000 до 5000»*
*Then отображаются только товары в указанном диапазоне*
*And количество товаров на странице обновлено*

*Scenario: Пустой результат фильтрации*
*Given пользователь открыл страницу каталога*
*When пользователь устанавливает фильтр «Цена от 100 000 до 200 000»*
*And в каталоге нет товаров в этом диапазоне*
*Then отображается сообщение «Ничего не найдено»*
*And пользователь видит кнопку «Сбросить фильтры»*

Обратите внимание: для одной user story обычно несколько сценариев — основной (happy path), граничные случаи и сценарий ошибки.

## Правила хорошего BDD-сценария

**Один сценарий = одно поведение.** Если вы проверяете одно действие и один результат — это один сценарий. Если проверяете два разных поведения — два сценария.

**Изолированные шаги.** Каждый шаг описывает ровно одно действие или состояние. «Пользователь вводит промокод и нажимает кнопку» — это два шага.

**Конкретные данные.** Избегайте слов «корректный», «правильный», «существующий». Используйте конкретные значения.

- Плохо: *Given пользователь вводит корректный промокод*
- Хорошо: *Given пользователь вводит промокод "SAVE10"*

**Не пишите технические детали.** BDD-сценарий — это язык бизнеса. Не пишите «система отправляет POST-запрос на /api/orders». Пишите «заказ оформлен».

## Когда нужны BDD-сценарии

BDD особенно полезен когда:

- **Сложная бизнес-логика** — расчёты, скидки, статусы, правила
- **Много граничных случаев** — один основной сценарий и десяток «а что если»
- **Команда только знакомится с проектом** — сценарии служат документацией и тестами одновременно
- **Автоматизация тестирования** — BDD-сценарии можно превратить в автотесты (Cucumber, SpecFlow, Behat)

## Почему это важно для аналитика

BDD-сценарии — это мост между аналитиком и тестировщиком. Аналитик описывает ожидаемое поведение, тестировщик проверяет его по этим же сценариям:

1. **Снижение неоднозначности** — «работает как часы» не принимается
2. **Автоматизация проверок** — BDD-сценарии = тест-кейсы
3. **Единый язык** — одни и те же формулировки в требованиях, тестах и коде
4. **Регрессия** — при изменении функциональности сценарии защищают от поломок

## Проверь себя

**1. Чем BDD-сценарий отличается от обычного списка acceptance criteria?**

BDD-сценарий имеет строгую структуру Given/When/Then и может быть автоматизирован. Обычный список условий — это свободный перечень для ручной проверки.

**2. Сколько сценариев нужно для одной user story?**

Минимум один (happy path). В реальности — 3–7: основной сценарий, граничные случаи, сценарий ошибки.

**3. Зачем в BDD использовать конкретные данные вместо «существующий» или «корректный»?**

Конкретные данные делают сценарий однозначным и автоматизируемым. «Промокод SAVE10» проверяется по-разному для «SAVE10» и «save10» — это важно для тестирования.
