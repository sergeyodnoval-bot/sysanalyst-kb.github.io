# Decision Log

Decision Log (журнал решений) — это документ, в котором фиксируются **ключевые решения** по проекту: почему выбрали эту архитектуру, почему не стали делать эту фичу, почему изменили приоритет.

## Зачем нужен Decision Log

- **Память команды** — через 6 месяцев никто не вспомнит, почему приняли решение
- **Onboarding** — новый аналитик читает лог и понимает контекст
- **Аудит** — можно показать, что решение было обоснованным
- **Предотвращение повторения ошибок** — «мы уже пробовали этот подход — не взлетело»

## Decision Log vs ADR

| | Decision Log | ADR (Architecture Decision Record) |
|--|-------------|-----------------------------------|
| **Фокус** | Любые решения (функция, приоритет, процесс) | Только архитектурные решения |
| **Формат** | Простая таблица / список | Структурированный документ (ADR) |
| **Кто пишет** | Аналитик, PM, любой участник | Архитектор / Senior developer |
| **Где хранят** | Confluence, Google Sheets | Git (рядом с кодом) |

## Формат записи

| Поле | Описание | Пример |
|------|----------|--------|
| **ID** | Уникальный номер | DEC-001 |
| **Дата** | Когда принято | 2026-03-15 |
| **Автор** | Кто принял | Иван Петров |
| **Тема** | Коротко о чём решение | «Выбор БД для каталога товаров» |
| **Контекст** | Почему возник вопрос | «Текущая БД не справляется с поиском» |
| **Рассмотренные варианты** | Что ещё рассматривали | PostgreSQL, Elasticsearch, MongoDB |
| **Решение** | Что выбрали | Elasticsearch |
| **Обоснование** | Почему | «Полнотекстовый поиск, фасеты, скорость» |
| **Последствия** | Что изменится | «Нужен отдельный кластер, данные дублируются» |
| **Статус** | Proposed / Accepted / Deprecated / Superseded | Accepted |

### Пример в Confluence

```
DEC-001: Выбор БД для каталога товаров
────────────────────────────────────────
Дата: 2026-03-15
Автор: Иван Петров
Статус: Accepted

Контекст:
  При росте каталога до 50k товаров PostgreSQL перестал
  справляться с поиском по характеристикам (latency > 5 с).

Варианты:
  1. Оптимизировать PostgreSQL (индексы, шардинг) — временная мера
  2. Elasticsearch — полнотекстовый поиск, фасеты, быстрый
  3. MongoDB — гибкая схема, но поиск слабее

Решение:
  Elasticsearch 8.x

Обоснование:
  - Полнотекстовый поиск из коробки
  - Фасетная навигация (фильтры по характеристикам)
  - p95 < 100 мс на 100k товаров
  - Опыт команды (использовали на прошлом проекте)

Последствия:
  + Скорость поиска
  + Фасеты
  − Нужен отдельный кластер
  − Данные дублируются (Postgres + ES)
  − Доработка ETL (синхронизация данных)
```

## Где хранить

| Инструмент | Плюсы | Минусы |
|-----------|-------|--------|
| **Confluence** | Поиск, теги, ссылки на задачи | Не в Git |
| **Google Docs** | Простой доступ | Хаос без структуры |
| **Git (decision-log/)** | Версионирование, PR | Неудобно не-технарям |
| **Jira (спец. issue type)** | Связано с задачами | Неудобно читать много записей |

## Типичные ошибки

- **Не пишут** — «мы и так помним» (через месяц не помнят)
- **Слишком много** — каждое микросообщение «выбрали цвет кнопки»
- **Нет обоснования** — «решили так и всё» (бесполезно для будущего)
- **Не обновляют статус** — решение устарело, но в логе висит как актуальное
