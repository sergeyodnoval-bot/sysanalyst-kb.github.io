# User Stories

:::note
User story — это короткое описание функции с точки зрения пользователя. Формат: `«Как <роль>, я хочу <цель>, чтобы <причина>»`. User story — не спецификация, а повод для обсуждения.
:::

Когда требования собраны (как делать — читайте в «Коммуникация со стейкхолдерами»), их нужно как-то записать. Можно писать длинные формальные спецификации на сотни страниц. А можно — короткие карточки, которые инициируют обсуждение в команде. Второй подход — это user stories.

User story (пользовательская история) пришла из Agile-разработки и стала стандартом де-факто для описания функциональных требований в современных IT-командах.

## Формат user story

Классический шаблон:

```
Как <роль пользователя>
Я хочу <цель / действие>
Чтобы <причина / ценность>
```

**Примеры:**

- *Как покупатель, я хочу отфильтровать товары по цене, чтобы быстрее найти то, что могу себе позволить.*
- *Как менеджер, я хочу получать уведомление о новом заказе, чтобы сразу начать его обрабатывать.*
- *Как курьер, я хочу видеть список заказов на карте, чтобы построить оптимальный маршрут.*

Три части отвечают на ключевые вопросы: **кто**, **что**, **зачем**.

## Критерии хорошей user story: INVEST

Мы уже упоминали INVEST в статье «Что такое требование». Вот как он работает конкретно для user story:

- **Independent** — история самодостаточна, её можно реализовать отдельно
- **Negotiable** — можно обсудить детали с командой, это не жёсткая спецификация
- **Valuable** — каждая история несёт ценность для пользователя или бизнеса
- **Estimable** — команда может оценить объём работы
- **Small** — влезает в одну итерацию (спринт)
- **Testable** — есть чёткие критерии приёмки

### Плохие примеры

- *«Как пользователь, я хочу удобный интерфейс»* — не тестируется, неоценимо
- *«Система должна поддерживать авторизацию через соцсети и отправлять email-уведомления»* — две истории в одной
- *«Как администратор, я хочу настроить систему»* — слишком размыто, что именно настраивать?

## Acceptance Criteria (критерии приёмки)

К user story обязательно прикладывают **acceptance criteria** — условия, при которых история считается выполненной.

Форматы описания:

### Список условий

```
- Покупатель может выбрать диапазон цен от 1000 до 5000 рублей
- Результаты фильтрации отображаются не дольше 2 секунд
- Если товаров не найдено, показывается сообщение «Ничего не найдено»
- Фильтр работает на десктопе и мобильной версии
```

### BDD-сценарии

Более формальный подход — Given / When / Then (подробнее в статье «BDD-сценарии»).

## Где хранить user stories

На практике user stories ведут в системах управления проектами:

- **Jira** — стандарт для Agile-команд (см. [Jira](/tech/jira))
- **Trello, Notion, Yandex Tracker** — альтернативы
- **Confluence** — для долгосрочной документации (см. [Confluence](/tech/confluence))

Каждая story в Jira — это задача (issue) с полями: описание, приоритет, исполнитель, статус, прикреплённые файлы.

## Почему это важно для аналитика

User story — основной рабочий инструмент аналитика в Agile-команде:

- **Понятна всем** — заказчику, разработчику, тестировщику
- **Инициирует диалог** — короткая формулировка провоцирует вопросы и уточнения
- **Фокусирует на ценности** — каждая story отвечает на вопрос «зачем»
- **Легко приоритизировать** — stories можно переставлять в бэклоге

## Ключевые термины

| Термин | Значение |
|--------|---------|
| **User story** | Короткое описание функции от лица пользователя |
| **Acceptance criteria** | Условия, при которых story считается готовой |
| **Бэклог** | Упорядоченный список user stories и задач |
| **Спринт** | Итерация разработки (обычно 1-2 недели) |
| **INVEST** | Критерии качества user story |

## Проверь себя

**1. Зачем в user story часть «чтобы» (причина)?**

Чтобы явно указать ценность для пользователя и не делать фичу «потому что попросили». Если причину сформулировать не получается — возможно, story не нужна.

**2. Что такое acceptance criteria?**

Набор условий, по которым команда и заказчик проверяют, что user story реализована правильно. Если критерий не выполнен — story не принимается.

**3. Можно ли писать user story на технические задачи (например, «переписать модуль авторизации на новом фреймворке»)?**

Технически можно, но теряется смысл — для кого это ценность? Такие задачи лучше оформлять как технические задачи (tech task) или включать как подзадачи в функциональную story.
