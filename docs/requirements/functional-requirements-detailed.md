# Функциональные требования (детально)

Функциональное требование — это описание того, **что система должна делать**: действие, поведение, реакция на ввод, бизнес-правило.

## Виды функциональных требований

| Тип | Описание | Пример |
|-----|----------|--------|
| **Бизнес-правила** | Ограничения или правила из бизнеса | «Скидка не более 30%» |
| **Ввод/вывод** | Что система принимает и отдаёт | «Загрузить файл в формате CSV» |
| **Поведение** | Реакция на действие пользователя | «При клике на кнопку открывается форма» |
| **Обработка данных** | Как данные преобразуются | «Рассчитать НДС = сумма × 0.2» |
| **Интеграция** | Взаимодействие с внешними системами | «Отправить заказ в CRM» |

## Как формулировать

### Формат: «Система должна [действие] [условие] [результат]»

| Плохо | Хорошо |
|-------|--------|
| «Пользователь может искать товары» | «Система должна предоставлять поиск товаров по названию и артикулу» |
| «Сделать отчёт по продажам» | «Система должна генерировать отчёт по продажам за период с группировкой по дням» |

### Checklist хорошего FR

- [ ] Однозначно: нет фраз «как правило», «в большинстве случаев», «по возможности»
- [ ] Проверяемо: можно написать тест (passed/failed)
- [ ] Одна функция на одно требование (не список)
- [ ] Не содержит реализацию («как» — для разработчика)

## Способы описания

### 1. Таблица (матрица)

| ID | Функция | Вход | Выход | Исключение |
|----|---------|------|-------|------------|
| FR-1 | Регистрация | email, пароль | 200 OK, токен | email уже занят → 409 |
| FR-2 | Поиск товаров | строка поиска | список товаров | ничего не найдено → пустой список |

### 2. Gherkin (BDD)

```gherkin
Feature: Поиск товаров
  Scenario: Успешный поиск по названию
    Given пользователь на странице каталога
    When он вводит "ноутбук" в строку поиска
    Then система показывает список товаров, содержащих "ноутбук"
```

### 3. User Story + Acceptance Criteria

```gherkin
US-101: Как клиент, я хочу искать товары по артикулу

AC1: Поиск по полному артикулу (ABC-12345) → один товар
AC2: Поиск по части артикула (ABC) → список товаров
AC3: Поиск по несуществующему артикулу → «ничего не найдено»
```

## Функциональные требования и User Stories

| | User Story | Функциональное требование |
|--|-----------|--------------------------|
| **Фокус** | Ценность для пользователя | Действие системы |
| **Формат** | «Как… я хочу… чтобы…» | «Система должна…» |
| **Уровень** | Высокий (epic/story) | Детальный (можно тестировать) |
| **Кто пишет** | PO / Аналитик | Аналитик |
| **Изменчивость** | Меняется реже | Меняется при уточнении |

На практике US декомпозируется в FR:

```
US: Как кассир, я хочу пробить товар по штрихкоду

  FR-1: Система должна сканировать штрихкод с камеры
  FR-2: Система должна найти товар по штрихкоду в БД
  FR-3: Если товар не найден — показать ошибку «товар не найден»
```

## Документирование FR

- В Jira / Confluence: каждый FR — отдельная задача или подзадача
- В SRS (Software Requirements Specification): таблица всех FR
- В тест-кейсах: каждый FR → минимум один тест
- В API-спецификации: FR превращаются в эндпоинты и схемы

## Типичные ошибки

- **Смешение с UI** — «кнопка должна быть синей» это не FR, а требование к дизайну
- **Смешение с архитектурой** — «данные должны храниться в PostgreSQL» это не FR
- **Неполнота** — описаны только happy path, исключения пропущены
- **Забыты нефункциональные требования** — скорость, безопасность, нагрузка
