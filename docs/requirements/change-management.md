# Управление изменениями требований

Требования меняются. Всегда. Задача аналитика — не запретить изменения, а **управлять ими**: оценивать влияние, согласовывать, документировать.

## Почему требования меняются

- **Бизнес меняется** — рынок, регуляция, конкуренты
- **Пользователи увидели прототип** — поняли, что хотели на самом деле
- **Технические ограничения** — выяснилось, что реализация сложнее
- **Новый стейкхолдер** — пришёл с новыми идеями
- **Закон/регулятор** — изменились требования ЦБ, ФНС

## Change Request (CR)

**CR** — формальный запрос на изменение требований.

### Структура CR

| Поле | Описание |
|------|----------|
| **ID** | CR-001 |
| **Автор** | Кто запрашивает |
| **Дата** | Когда создан |
| **Описание** | Что предлагается изменить |
| **Причина** | Почему нужно изменить |
| **Влияние** | Какие требования, системы, сроки затрагивает |
| **Priority** | Critical / High / Medium / Low |
| **Status** | New / Analysis / Approved / Rejected / Implemented |

### Процесс

```
  ┌─────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐
  │  New     │→ │ Analysis │→ │ Approved  │→ │ Implement│
  │  CR      │   │ (impact) │   │ /Rejected │   │ ed       │
  └─────────┘   └──────────┘   └──────────┘   └──────────┘
                      │
                      │ Если сложно:
                      ▼
                ┌──────────┐
                │  Triage  │
                │ (приори- │
                │  тизация)│
                └──────────┘
```

## Impact Analysis (Анализ влияния)

Перед тем как принять CR, нужно понять:

| Аспект | Вопрос |
|--------|--------|
| **Scope** | Какие требования нужно изменить? |
| **Schedule** | Как изменится срок сдачи? |
| **Cost** | Сколько дополнительных ресурсов нужно? |
| **Risk** | Какие риски несёт изменение? |
| **Architecture** | Требует ли изменения архитектуры? |

**Результат:** Impact Matrix

| Требование | Тип изменения | Трудоёмкость | Риск |
|-----------|--------------|-------------|------|
| FR-23 | Изменение логики | 2 дня | Низкий |
| FR-45 | Новая функция | 10 дней | Средний |
| ARC-3 | Изменение архитектуры | 20 дней | Высокий |

## Baselines (Базовые версии)

**Baseline** — зафиксированная версия требований, от которой считаются изменения.

- Baseline 1.0 — требования утверждены → разработка началась
- Baseline 1.1 — CR-001 принят → требования обновлены
- Baseline 2.0 — первый релиз → новый baseline

Каждое изменение — версия требований.

## Как управлять в Agile

| Инструмент | Как работает |
|-----------|-------------|
| **Backlog Grooming** | Регулярная переоценка и переприоритизация |
| **MVP-first** | Новое требование → сравниваем с MVP, решаем |
| **Timebox** | Изменение можно внести, если влезает в спринт |
| **Icebox** | Неприоритетные CR → в «лёд» (тают медленно) |
| **Feature freeze** | За 2 недели до релиза — никаких новых фич |

## Типичные ошибки

- **Каждое изменение — немедленно в разработку** — без анализа влияния
- **Нет baseline** — непонятно, что изменилось с прошлой версии
- **Устные изменения** — «нам вчера Петя сказал поменять»
- **Scope creep** — проект растёт без пересмотра бюджета и сроков
- **CR без приоритета** — все хотят всё и сразу
