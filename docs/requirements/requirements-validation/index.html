Валидация требований

Валидация требований — это проверка правильности, полноты и непротиворечивости требований до того, как разработка началась.

Зачем валидировать​

  • Исправление ошибки в требованиях на этапе анализа стоит в 10 раз дешевле, чем на этапе разработки, и в 100 раз дешевле, чем в продакшене
  • 40–60% багов в проектах — из-за ошибок в требованиях

Критерии качества требований​

КритерийВопросПлохоХорошо
CorrectnessТребование верно отражает бизнес?«Система считает налог» (какой?)«Система считает НДС = сумма × 20%»
CompletenessВсё ли описано?«Пользователь может войти»Описаны все способы входа, ошибки
ConsistencyНет ли противоречий?«Макс. размер файла 10 MB» и «Макс. размер 50 MB»Единые значения
UnambiguityОднозначно ли?«Быстрая обработка»«Обработка < 2 секунд»
VerifiabilityМожно проверить?«Удобный интерфейс»«Новый пользователь оформляет заказ за 3 клика»
TraceabilityПонятно, откуда взялось?—Есть ссылка на источник (интервью, закон)

Методы валидации​

1. Formal Review (инспекция)​

Структурированная проверка группой:

  1. Автор рассылает требования участникам
  2. Каждый участник ищет ошибки (checklist)
  3. Встреча: зачитываются найденные проблемы
  4. Фиксация: автор исправляет

2. Walkthrough (прогон)​

Автор сам проводит по требованиям команду, объясняет логику. Участники задают вопросы.

3. Prototyping (прототипирование)​

Требования превращаются в прототип (Figma, clickable mockup). Пользователь «пробует» и уточняет.

Лучший способ валидации — пользователь видит прототип, а не читает спецификацию.

4. Checklist-based validation​

Чек-лист, который проверяет требования:

□ Каждое требование имеет уникальный ID
□ Каждое требование тестируемо
□ Нет дублирующихся требований
□ Указаны исключения (не только happy path)
□ Учтены нефункциональные требования
□ Требования согласованы со стейкхолдерами

5. Test case design​

Попробуйте написать тест-кейсы к требованиям. Если тест написать нельзя — требование невалидно.

Типичные проблемы, которые находит валидация​

ПроблемаПример
Пропущенные альтернативыОписан только happy path
ПротиворечияFR-1 и FR-2 требуют разного поведения одной функции
Недостижимые требования«99.999% uptime» без бюджета на инфраструктуру
Избыточные требованияНе влияют на бизнес-цель
Жаргон«Маржевая прибыль при консолидации» без определения

Кто участвует​

РольЧто делает
Аналитик (автор)Готовит требования, исправляет замечания
Senior аналитик / АрхитекторПроверяет полноту и непротиворечивость
РазработчикПроверяет реализуемость
ТестировщикПроверяет тестируемость
Бизнес-заказчикПодтверждает бизнес-корректность

Типичные ошибки​

  • Валидация слишком поздно — когда требования уже в разработке
  • Валидация без пользователя — только аналитики проверяют аналитиков
  • Нет чек-листа — проверка «на глаз»
  • Формальная подпись — согласовали, но не читали
