# Нефункциональные требования (NFR)

Нефункциональные требования (NFR, Non-Functional Requirements) описывают **как система должна работать**, а не **что она должна делать**. Их ещё называют quality attributes (атрибуты качества).

## FR vs NFR

| | FR (функциональные) | NFR (нефункциональные) |
|--|--------------------|----------------------|
| **Отвечают на** | «Что система делает?» | «Насколько хорошо система это делает?» |
| **Измерение** | Бинарное (есть/нет) | Шкала (время, процент, количество) |
| **Критичность** | Без FR система бесполезна | Без NFR система бесполезна при нагрузке |
| **Пример** | «Система обрабатывает заказ» | «Система обрабатывает заказ за < 2 секунд» |

## Категории NFR

| Категория | Описание | Пример |
|-----------|----------|--------|
| **Performance** | Скорость, latency, throughput | «Страница грузится не более 2 секунд» |
| **Scalability** | Способность расти | «Система должна выдерживать 10 000 RPS» |
| **Availability** | Доступность | «Uptime 99.9% (не более 8 часов downtime в год)» |
| **Security** | Безопасность | «Пароль хранится в хеше bcrypt» |
| **Reliability** | Надёжность | «Потеря данных не более 0.001%» |
| **Usability** | Удобство | «Новый пользователь оформляет заказ за 3 клика» |
| **Portability** | Переносимость | «Система работает в Linux и Windows» |
| **Maintainability** | Сопровождаемость | «Код покрыт тестами на 80%» |

## Как формулировать NFR

### SMART-подход

Конкретные, измеримые, достижимые, релевантные, ограниченные по времени.

| Плохо | Хорошо (SMART) |
|-------|----------------|
| «Система должна быть быстрой» | «API отвечает на 95% запросов за < 500 мс при нагрузке до 1000 RPS» |
| «Система должна быть безопасной» | «Доступ к API только по JWT-токену, пароли хранятся в bcrypt» |
| «Система должна быть надёжной» | «Uptime 99.9%, RTO < 1 час, RPO < 15 минут» |

### Метрики для NFR

| Атрибут | Метрика | Целевое значение |
|---------|---------|-----------------|
| Доступность | Uptime % | 99.9% (три девятки) |
| Производительность | p95 latency | < 500 мс |
| Пропускная способность | RPS (requests per second) | > 1000 |
| Надёжность | Error rate | < 0.1% |
| Восстановление | RTO (Recovery Time Objective) | < 1 час |
| Потери данных | RPO (Recovery Point Objective) | < 15 минут |

## NFR и архитектура

NFR напрямую влияют на выбор архитектуры:

| NFR | Какая архитектура подходит |
|-----|--------------------------|
| Высокая доступность (99.99%) | Кластеризация, multi-AZ, active-active |
| Низкая latency (< 10 мс) | In-memory кэш, CDN, edge computing |
| Высокая безопасность | Air-gapped, on-premise, HSM |
| Масштабируемость (100k RPS) | Микросервисы, Kafka, auto-scaling |

## Как собирать NFR

1. **Интервью с бизнесом**: «Сколько пользователей? Какой процент downtime допустим?»
2. **Интервью с разработчиками**: «Какие технологии хотите? Какие ограничения?»
3. **Анализ конкурентов**: «Как быстро работает их система?»
4. **SLA / SLO**: Документ с целевыми метриками

## Документирование NFR

В SRS обычно отдельный раздел:

```
## 3. Нефункциональные требования

### 3.1 Производительность
- NFR-1: API отвечает на 95% запросов за < 500 мс
- NFR-2: Страница каталога загружается за < 2 секунды

### 3.2 Доступность
- NFR-3: Uptime 99.9%
- NFR-4: RTO < 1 час, RPO < 15 минут

### 3.3 Безопасность
- NFR-5: Авторизация через JWT
- NFR-6: Все данные передаются по TLS 1.3
```

## Типичные ошибки

- **NFR забыты** — все помнят только функциональные требования
- **NFR не измеримы** — «быстрый», «надёжный», «удобный»
- **NFR без контекста** — «99.9% доступность» без бюджета (это дорого)
- **NFR противоречат друг другу** — высокая безопасность может снижать скорость
