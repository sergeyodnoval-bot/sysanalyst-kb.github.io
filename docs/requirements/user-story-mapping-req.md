# User Story Mapping

User Story Mapping — это техника визуализации продукта через последовательность действий пользователя. Помогает увидеть **полную картину** (MVP, релизы, backlog).

> Story Map уже есть как техника моделирования (см. [User Story Mapping](../modeling/user-story-mapping.md)). Этот материал — фокус на **работу с требованиями** и **планирование релизов**.

## Зачем аналитику Story Mapping

- **Целостная картина** — видно, какие шаги пользователя покрыты, а какие нет
- **MVP виден сразу** — первая строка = минимальный работающий продукт
- **Разрывы в требованиях** — если шаг есть, а user story нет — это дыра
- **Коммуникация с командой** — вся команда видит продукт на одной доске

## Структура

```
┌─────────────────────────────────────────────────────┐
│ Цель: Купить товар в интернет-магазине              │
├──────────┬──────────┬──────────┬──────────┬─────────┤
│ Поиск   │ Выбор    │ Оформл.  │ Оплата   │ Получ. │
│         │          │          │          │         │
├──────────┼──────────┼──────────┼──────────┼─────────┤
│MVP ───── │ ──────── │ ──────── │ ──────── │ ─────── │
│Поиск     │Корзина   │Адрес     │Карта     │Подтв.   │
│по назв.  │          │доставки  │          │email    │
├──────────┼──────────┼──────────┼──────────┼─────────┤
│Release 2 │Поиск по  │Избранное │Промо-    │СБП      │Трекинг  │
│          │категории │          │коды      │         │заказа   │
├──────────┼──────────┼──────────┼──────────┼─────────┤
│Release 3 │Голос.    │Сравнение │Авто-     │Dolyame  │Чат с    │
│          │поиск     │товаров   │заполн.   │         │операт.  │
└──────────┴──────────┴──────────┴──────────┴─────────┘
```

- **Строки**: пользовательские шаги (слева направо хронологически)
- **Столбцы**: детализация каждого шага
- **MVP**: первая строка (минимальный набор)
- **Снизу вверх**: приоритет (чем выше — тем важнее)

## Как построить Story Map

1. **Определите пользователя** — для кого делаем продукт?
2. **Назовите цель** — что пользователь хочет сделать?
3. **Шаги по горизонтали** — от начала до конца (5–10 шагов)
4. **User stories под каждым шагом** — все варианты, детали, исключения
5. **Срежьте MVP** — первая строка = то, без чего продукт не заработает
6. **Распишите релизы** — второй, третий и т.д.

## Story Mapping для планирования

**До:** длинный flat backlog из 200 задач, непонятно с чего начать.

**После:** Story Map → видно, что MVP = 40 задач, Release 2 = ещё 60, остальное — потом.

## Типичные ошибки

- **Слишком детально** — 100 stories на один шаг (начните с крупных)
- **Нет пользователя** — пишете stories для системы, а не для человека
- **MVP не отрезан** — первая строка содержит всё (это не MVP)
- **Не обновляется** — сделали один раз и забыли
