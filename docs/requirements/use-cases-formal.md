# Use Cases (формальные)

Use Case — это описание **взаимодействия пользователя с системой** для достижения конкретной цели. В отличие от User Story (короткая, Agile), формальный Use Case — это детальный, структурированный сценарий.

## User Story vs Use Case

| | User Story | Use Case |
|--|-----------|---------|
| **Формат** | «Как… я хочу… чтобы…» | Структурированный документ |
| **Детализация** | Низкая (prompt для обсуждения) | Высокая (все шаги, альтернативы, исключения) |
| **Объём** | 1–2 предложения | 1–5 страниц |
| **Когда нужен** | Agile, продуктовые команды | Enterprise, сложные системы, контракты |
| **Актор** | Роль пользователя | Конкретный актор |

## Структура формального Use Case

### Основные элементы

| Элемент | Описание | Пример |
|---------|----------|--------|
| **ID** | Уникальный номер | UC-001 |
| **Название** | Глагол + объект | «Оформить заказ» |
| **Актор** | Кто выполняет | «Клиент» |
| **Goal** | Что достигается | «Купить товар» |
| **Preconditions** | Что должно быть до | «Клиент авторизован, корзина не пуста» |
| **Postconditions** | Что после успеха/неудачи | «Заказ создан, письмо отправлено» |
| **Main flow** | Основной сценарий | 1. Клиент → 2. Система → … |
| **Alternative flows** | Альтернативные пути | A1: «Товара нет в наличии» |
| **Exception flows** | Ошибки | E1: «Платёж не прошёл» |

### Пример

```
UC-001: Оформить заказ
───────────────────────
Actor: Клиент
Goal: Оплатить товары в корзине

Preconditions: Клиент авторизован, корзина содержит товары
Postconditions: Заказ создан (статус = new), письмо отправлено

Main Flow:
  1. Клиент нажимает «Оформить заказ»
  2. Система проверяет состав корзины
  3. Система запрашивает способ доставки
  4. Клиент выбирает способ доставки
  5. Система запрашивает способ оплаты
  6. Клиент выбирает способ оплаты
  7. Система создаёт заказ (статус: new)
  8. Система отправляет письмо-подтверждение
  9. Use Case завершён

Alternative Flows:
  A1: Товара нет в наличии
    A1.1 Система сообщает, какие товары недоступны
    A1.2 Клиент убирает товар из корзины
    A1.3 Возврат к шагу 2

  A2: Гость (без регистрации)
    A2.1 Система предлагает заполнить email и телефон
    A2.2 Клиент вводит контактные данные
    A2.3 Возврат к Main Flow шаг 3

Exception Flows:
  E1: Платёж не прошёл
    E1.1 Система показывает ошибку оплаты
    E1.2 Система предлагает другой способ оплаты
    E1.3 Use Case завершён (заказ не создан)
```

## Как писать Use Case

1. Определите актора (кто взаимодействует с системой?)
2. Назовите goal (что он хочет сделать?)
3. Опишите main flow — **happy path**, где всё идёт по плану
4. Добавьте alternative flows — другие варианты того же сценария
5. Добавьте exception flows — что идёт не так
6. Проверьте pre/postconditions

## Когда нужны Use Cases

- **Enterprise-проекты** — нужна полная спецификация до разработки
- **Сложные flows** — много ветвлений, альтернатив, исключений
- **Контракт с заказчиком** — Use Case как часть юридического документа
- **Подрядная разработка** — фиксируем scope (что входит, что нет)
- **Legacy documentation** — восстановление документации по существующей системе

Когда **НЕ** нужны:
- Простой CRUD-интерфейс
- Agile-команда с PO (хватит US + AC)
- Быстрый прототип / MVP

## Use Case и UML

Use Case тесно связан с **Use Case diagram**:

```
┌─────────────────────────────────────┐
│           Интернет-магазин           │
│                                     │
│  ┌──────────┐   ┌──────────────┐   │
│  │Поиск     │   │Оформить заказ│   │
│  │товаров   │   │              │   │
│  └──────────┘   └──────────────┘   │
│  ┌──────────┐   ┌──────────────┐   │
│  │Оставить  │   │Отменить      │   │
│  │отзыв     │   │заказ         │   │
│  └──────────┘   └──────────────┘   │
└─────────────────────────────────────┘
       ↑              ↑
       │              │
   ┌───────┐   ┌──────────┐
   │Клиент │   │Оператор  │
   └───────┘   └──────────┘
```
