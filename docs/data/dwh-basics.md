# Хранилища данных (DWH) — основы

Хранилище данных (Data Warehouse, DWH) — это специализированная база данных, предназначенная для **аналитики и отчётов**, а не для операционной работы (OLTP). В отличие от OLTP-систем (банковские транзакции, CRM), DWH оптимизирован для чтения больших объёмов данных и сложных агрегаций.

## OLTP vs OLAP

| Характеристика | OLTP (операционная БД) | OLAP / DWH (аналитическая) |
|---------------|----------------------|--------------------------|
| **Цель** | Обработка транзакций | Анализ и отчётность |
| **Запросы** | INSERT / UPDATE / DELETE (частые, мелкие) | SELECT (редкие, сложные, агрегации) |
| **Нормализация** | Да (3NF, минимум дублирования) | Денормализована (звёздная схема) |
| **Индексы** | Много индексов (быстрый поиск записи) | Минимум (полное сканирование) |
| **Данные** | Текущие (последние дни/часы) | Исторические (годы) |
| **Примеры** | PostgreSQL, MySQL, Oracle | ClickHouse, Snowflake, Greenplum |

## Как устроен DWH

```
    OLTP-системы           DWH                BI / Reports
  ┌──────────┐          ┌─────────┐          ┌────────────┐
  │ CRM      │─ETL─────→│ Staging │─→ DWH ──→│ Metabase   │
  ├──────────┤          │ Area    │   Layer   ├────────────┤
  │ ERP      │─ETL─────→│ (сырые  │          │ Tableau    │
  ├──────────┤          │  данные)│          ├────────────┤
  │ Payment  │─ETL─────→│         │          │ Superset   │
  └──────────┘          └─────────┘          └────────────┘
```

### Слой Staging

Сырые данные «как есть» из источников. Используется только для ETL, пользователи сюда не ходят.

### Слой DWH (Core Layer)

Данные очищены, согласованы, приведены к единой модели. Хранятся в **звёздной** или **снежинкообразной** схеме.

### Слой Data Marts (Витрины данных)

Агрегированные данные для конкретных бизнес-подразделений: Sales Mart, Finance Mart, HR Mart.

## Модели данных в DWH

### Звёздная схема (Star Schema)

```
                    ┌──────────────┐
                    │  Dim_Customer│
                    │ (customer_id, │
                    │  name, city) │
                    └──────┬───────┘
                           │
┌──────────────┐    ┌──────┴───────┐    ┌──────────────┐
│  Dim_Product │────│  Fact_Sales  │────│  Dim_Date    │
│ (product_id, │    │(sale_id,     │    │(date_id,     │
│  name, price)│    │ customer_id, │    │  year, month)│
└──────────────┘    │ product_id,  │    └──────────────┘
                    │ date_id,     │
                    │ amount)      │
                    └──────────────┘
```

- **Fact table** — факты (продажи, клики, звонки). Растут быстро.
- **Dimension table** — измерения (клиент, товар, дата). Меняются редко.

### Схема Снежинка (Snowflake)

Нормализованные dimension-таблицы. Экономит место, но сложнее для запросов.

## Когда нужен DWH

- Данные из нескольких систем нужно сводить в один отчёт
- История нужна «навсегда» (транзакции 5-летней давности)
- OLTP-система тормозит от сложных аналитических запросов
- Бизнес просит дашборды с обновлением раз в день / час

## Когда DWH НЕ нужен

- Данных мало (тысячи записей) — хватит PostgreSQL с индексами
- Нужна аналитика в реальном времени — Kafka Streams / ClickHouse
- Нет исторических данных — можно делать запросы напрямую к OLTP
- Команда из 2 человек — DWH требует отдельной инфраструктуры

## Современные DWH-решения

| Продукт | Облачный / On-prem | Особенность |
|---------|-------------------|-------------|
| **ClickHouse** | On-prem / Cloud | Колоночная БД, очень быстрый SELECT |
| **Snowflake** | Cloud only | Полностью managed, отдельное хранение и вычисления |
| **BigQuery** | Cloud (GCP) | Serverless, плата за просканированный объём |
| **Redshift** | Cloud (AWS) | PostgreSQL-совместимый |
| **Greenplum** | On-prem | MPP-архитектура, open source |
| **Vertica** | On-prem / Cloud | Колоночная, высокая компрессия |

## Как начать

1. Определите источники данных (CRM, ERP, логи)
2. Спроектируйте звёздную схему для одной бизнес-области (например, продажи)
3. Настройте простой ETL (например, через dbt или Airflow)
4. Загрузите тестовые данные
5. Постройте первый дашборд в Metabase / Superset
