# Миграции данных — стратегии

Миграция данных — это перенос данных из одной системы / формата / схемы в другую. Часто сопровождает переход на новую систему, рефакторинг БД или разделение монолита на микросервисы.

## Зачем нужна миграция

- Переход с legacy-системы на новую
- Разделение БД монолита на несколько (per service)
- Смена СУБД (MySQL → PostgreSQL, Oracle → ClickHouse)
- Объединение данных после M&A
- Версионирование схемы (schema migration)

## Типы миграций

| Тип | Масштаб | Пример |
|-----|---------|--------|
| **Schema migration** | Изменение структуры (схемы) | Добавить колонку, переименовать таблицу |
| **Data migration** | Перенос содержимого | Перелить данные из CRM старой в новую |
| **Application migration** | Перенос + код | Переписать монолит на микросервисы |
| **Cloud migration** | Смена инфраструктуры | On-prem → AWS / Azure |

## Стратегии миграции

### 1. Big Bang (разовый перенос)

**Как:** Остановить старую систему, перенести все данные, включить новую.

**Плюсы:** Просто, дёшево, быстро.

**Минусы:** Если что-то пошло не так — простой на дни / недели.

**Когда использовать:** Маленький объём данных, не критичный downtime.

### 2. Trickle Migration (поэтапный)

**Как:** Переносить данные частями (по модулям, по клиентам, по регионам).

**Плюсы:** Можно остановить в любой момент, меньше риска.

**Минусы:** Нужно синхронизировать две системы параллельно (dual write).

**Когда использовать:** Большой объём, сложная логика, высокие требования к availability.

### 3. Parallel Run (параллельная работа)

**Как:** Обе системы работают одновременно. Данные пишутся в обе. После валидации — отключение старой.

**Плюсы:** Можно сравнить результаты, откат в любой момент.

**Минусы:** Дорого (две инфраструктуры), сложно синхронизировать.

**Когда использовать:** Финансовая / медицинская система, где ошибка недопустима.

### 4. Strangler Fig (постепенное замещение)

**Как:** Части функциональности заменяются по одной. Старая и новая системы сосуществуют.

**Плюсы:** Минимальный риск, каждая часть тестируется отдельно.

**Минусы:** Долго, сложная маршрутизация запросов.

**Когда использовать:** Миграция монолита на микросервисы.

## ETL vs ELT в контексте миграции

| | ETL | ELT |
|--|-----|-----|
| **Порядок** | Извлечь → Трансформировать → Загрузить | Извлечь → Загрузить → Трансформировать |
| **Где трансформация** | В отдельном пайплайне | В целевой БД |
| **Скорость загрузки** | Медленнее (трансформация до загрузки) | Быстрее (сначала загружаем) |
| **Когда выбрать** | Сложная логика, слабая целевая БД | Мощная DWH (Snowflake, BigQuery) |

## Типичный план миграции

1. **Аудит** — собрать информацию о всех источниках данных
2. **Маппинг** — как поля старой схемы соответствуют новой
3. **Тестовая миграция** — прогнать пайплайн на копии данных
4. **Data validation** — сравнить количество записей, суммы, контрольные суммы
5. **Dry run** — полный прогон в staging-среде
6. **Production migration** — по плану с откатом (rollback plan)
7. **Верификация** — бизнес-пользователи проверяют данные
8. **Cut-off** — отключение старой системы

## Data Quality при миграции

| Проблема | Как выявить | Как решить |
|----------|-------------|------------|
| Потеря данных | Сравнить COUNT(*) в источнике и приёмнике | Двойная запись, idempotency |
| Дубликаты | GROUP BY + HAVING COUNT(*) > 1 | Дедупликация на этапе трансформации |
| NULL-значения | NOT NULL constraints | Default values, fallback |
| Неверный формат | Regex, pattern matching | Casting, transformation |
| Referential integrity | Проверка FK | Загрузка в правильном порядке |

## Инструменты для миграции

| Инструмент | Для чего |
|-----------|---------|
| **Flyway / Liquibase** | Schema migration (версионирование схемы) |
| **Apache NiFi** | Data flow / migration UI |
| **dbt** | ELT-трансформации в DWH |
| **Debezium + Kafka** | CDC — миграция без остановки |
| **AWS DMS / Azure DMS** | Cloud migration managed services |
| **pg_dump / pg_restore** | PostgreSQL — быстрый дамп |

## Практические советы

- **Всегда имейте rollback plan** — знать, как вернуть всё обратно, так же важно, как как сделать миграцию
- **Мигрируйте данные идемпотентно** — повторный запуск не должен создавать дубликаты
- **Валидируйте каждую партию** — не всю миграцию целиком, а каждую сотню / тысячу записей
- **Замеряйте время** — миграция, которая длится 3 дня вместо 3 часов, требует другого подхода
- **Коммуницируйте с бизнесом** — бизнес должен знать, что система будет недоступна / замедлена
- **Используйте feature flags** — новая система скрыта за флагом, можно тестировать на части пользователей
