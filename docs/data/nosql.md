# NoSQL — когда и зачем

NoSQL (Not Only SQL) — категория баз данных, которые не используют реляционную модель и SQL. Каждый тип NoSQL решает определённую проблему, с которой реляционные БД справляются плохо.

## Типы NoSQL

### Key-Value (ключ-значение)

Простейшая модель: по ключу получаем значение.

**Примеры:** Redis, Memcached, DynamoDB

**Когда использовать:** кэширование, сессии пользователей, счётчики.

**Когда НЕ использовать:** сложные запросы, связи между данными.

### Document (документная)

Хранит JSON-документы. Гибкая схема — каждый документ может иметь свой набор полей.

**Примеры:** MongoDB, CouchDB, Firestore

**Когда использовать:** каталоги товаров (разные товары — разные поля), логи, контент-менеджмент, прототипы.

**Когда НЕ использовать:** транзакции, строгие согласованные данные (бухгалтерия).

### Column Family (широко-столбцовая)

Данные организованы в строки с переменным количеством столбцов. Оптимизирована для запросов по диапазону.

**Примеры:** Cassandra, HBase, ScyllaDB

**Когда использовать:** временные ряды (метрики, логи), IoT, «большие данные», запись > чтения.

**Когда НЕ использовать:** сложные join'ы, ad-hoc запросы.

### Graph (графовая)

Данные — узлы и связи. Оптимизирована для запросов про связи между сущностями.

**Примеры:** Neo4j, Amazon Neptune, ArangoDB

**Когда использовать:** социальные сети, рекомендательные системы, графы знаний, анализ мошенничества.

**Когда НЕ использовать:** простые CRUD без глубоких связей.

## SQL vs NoSQL

| Критерий | SQL | NoSQL |
|----------|-----|-------|
| Схема | Фиксированная (schema-on-write) | Гибкая (schema-on-read) |
| Масштабирование | Вертикальное (проще) | Горизонтальное (легче) |
| Транзакции | ACID, строгие | BASE, eventual consistency |
| Сложные запросы | JOIN, подзапросы | Ограничены или своя логика |
| Зрелость | >50 лет | ~15–20 лет |

## CAP-теорема

Распределённая система может обеспечить только **два из трёх**:

- **Consistency (C)** — все у видят одни и те же данные
- **Availability (A)** — система отвечает на любой запрос
- **Partition Tolerance (P)** — система работает при разрыве сети

NoSQL базы выбирают AP или CP. SQL — CA.

## Когда выбирать NoSQL

**Выберите NoSQL, если:**
- Нужно горизонтальное масштабирование (террабайты данных)
- Схема данных постоянно меняется
- Высокая нагрузка на запись
- Используете Event Sourcing или CQRS

**Оставайтесь на SQL, если:**
- Данные структурированы и стабильны
- Нужны транзакции и консистентность
- Сложные отчёты и join'ы
- Команда знает SQL лучше, чем NoSQL
