ADR — Architecture Decision Records

ADR — короткий документ, фиксирующий архитектурное решение: контекст, рассмотренные варианты, выбранный вариант и его последствия. Каждое решение — отдельный файл в репозитории.

API-first design

API-first — это подход к разработке, при котором API проектируется и специфицируется до начала написания кода. API становится контрактом, вокруг которого строятся и фронтенд, и бэкенд.

BABOK — обзор для системного аналитика

BABOK (Business Analysis Body of Knowledge) — международный свод знаний по бизнес-анализу, издаваемый IIBA (International Institute of Business Analysis). Это не методология, а систематизированное описание практик, задач и компетенций аналитика.

Cost-Benefit Analysis (CBA)

Cost-Benefit Analysis — метод сравнения затрат на реализацию решения с выгодами от его внедрения. Позволяет ответить на вопрос: стоит ли это делать?

CQRS — Command Query Responsibility Segregation

CQRS — паттерн, разделяющий операции чтения (Query) и записи (Command). В традиционной архитектуре одна модель данных и для чтения, и для записи. CQRS предлагает разные модели.

Data Flow Diagram (DFD)

DFD — диаграмма потоков данных. Показывает, как данные движутся между процессами, хранилищами и внешними сущностями. В отличие от BPMN, DFD фокусируется на данных, а не на последовательности шагов.

ETL — основы

ETL (Extract, Transform, Load) — это процесс извлечения данных из источников, их трансформации и загрузки в целевую систему (обычно DWH). Позволяет собрать данные из разных систем в одном месте для анализа.

Event Storming

Event Storming — это воркшоп-техника для моделирования бизнес-процессов через события. Придумал Alberto Brandolini. Это не диаграмма, а формат встречи, где вся команда наклеивает стикеры на огромную доску.

Event-Driven Architecture (введение)

Event-Driven Architecture (EDA) — архитектурный стиль, в котором компоненты общаются через события, а не через прямые вызовы. Если микросервисы — про независимость деплоя, то EDA — про независимость во времени и пространстве.

Event-Driven Architecture (детально)

Вы уже знаете основы EDA из раздела интеграции. Теперь рассмотрим архитектурные аспекты: как проектировать события, как организовывать потоки и как EDA меняет дизайн системы.

Impact Mapping

Impact Mapping — стратегическая техника: как бизнес-цели связаны с поведением пользователей и конкретной функциональностью.

Kafka — событийный streaming

Apache Kafka — это распределённая платформа для обработки событийных потоков. В отличие от RabbitMQ, Kafka не удаляет сообщения после прочтения — она хранит их в логе определённое время (retention).

RabbitMQ — очереди сообщений

RabbitMQ — популярный брокер сообщений. Реализует протокол AMQP 0-9-1. Используется для task queues, RPC, и маршрутизации сообщений по сложным правилам.

SOLID для системного аналитика

SOLID — пять принципов объектно-ориентированного проектирования. Разработчики используют их для написания кода. Аналитик использует их для проектирования системы: как делить на модули, как описывать интерфейсы, как управлять зависимостями.

UML — Component diagram

Component diagram показывает, как компоненты системы связаны друг с другом через интерфейсы. Это мост между логической моделью (классы) и физической (деплой).

User Story Mapping

User Story Mapping — техника визуализации продукта через пользовательские истории, организованные по сценариям использования. Придумал Jeff Patton.

Анализ стейкхолдеров (Power/Interest Grid)

Стейкхолдеры — это люди и группы, которые влияют на проект или зависят от его результатов. Задача аналитика — понять, кто на что влияет, и выстроить коммуникацию соответственно.

Асинхронное взаимодействие (Message Queue)

До сих пор мы говорили о синхронном REST: клиент делает запрос и ждёт ответ. Но есть сценарии, где ждать нельзя или не нужно. Для этого существует асинхронное взаимодействие через очереди сообщений.

Коучинг — базовые техники

Коучинг — это метод развития, при котором коуч не даёт советов, а задаёт вопросы, помогающие человеку самому найти решение.

Менторство для аналитиков

Менторство — это передача опыта от более опытного аналитика менее опытному. В отличие от коучинга, ментор делится своими знаниями и опытом, а не просто задаёт вопросы.

Миграции данных — стратегии

Миграция данных — это перенос данных из одной системы / формата / схемы в другую. Часто сопровождает переход на новую систему, рефакторинг БД или разделение монолита на микросервисы.

Микросервисы — базовые паттерны

Переход на микросервисы — это не про технологию, а про организацию. Каждый паттерн микросервисной архитектуры решает конкретную проблему, которая возникает при делении системы на части.

Отраслевая аналитика (FinTech, Retail, Logistics)

Системный аналитик обычно работает не в вакууме, а в конкретной отрасли. Разные отрасли имеют разную специфику: терминологию, процессы, требования к системе, регуляцию.

Путь Data-аналитика

Data-аналитик (аналитик данных) — это специалист, который работает с данными: собирает, очищает, анализирует и визуализирует. В отличие от системного аналитика, фокус не на требованиях к системе, а на данных и выводах из них.

Путь Solution-архитектора

Solution Architect (SArch) — это роль, которая соединяет бизнес и техническую реализацию. Solution Architect проектирует решение на уровне системы: выбор технологий, интеграции, архитектурные стили, оценка стоимости.

Управление рисками

Риск — это событие, которое может произойти и повлиять на проект (положительно или отрицательно). Управление рисками — процесс идентификации, анализа, планирования реакции и мониторинга рисков.

Управление требованиями (Requirements Management)

Управление требованиями — это процесс сбора, документирования, отслеживания и контроля изменений требований на протяжении всего жизненного цикла продукта.

Хранилища данных (DWH) — основы

Хранилище данных (Data Warehouse, DWH) — это специализированная база данных, предназначенная для аналитики и отчётов, а не для операционной работы (OLTP). В отличие от OLTP-систем (банковские транзакции, CRM), DWH оптимизирован для чтения больших объёмов данных и сложных агрегаций.
