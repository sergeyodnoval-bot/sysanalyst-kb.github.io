# Управление требованиями (Requirements Management)

Управление требованиями — это процесс сбора, документирования, отслеживания и контроля изменений требований на протяжении всего жизненного цикла продукта.

## Зачем управлять требованиями

- **Единый источник правды** — все участники проекта работают с одной версией требований
- **Прослеживаемость (traceability)** — каждое требование можно связать с реализацией и тестами
- **Контроль изменений** — не допустить, чтобы требования менялись хаотично
- **Избежать потери** — ни одно требование не «потеряется» по пути в разработку

## Жизненный цикл требования

1. **Capture** — сбор (интервью, воркшопы, документ-анализ)
2. **Analysis** — анализ, уточнение, декомпозиция
3. **Specification** — запись в agreed-upon формате (User Story, Use Case, SRS)
4. **Validation** — проверка, что требование верно понято
5. **Approval** — утверждение стейкхолдерами
6. **Implementation** — передача в разработку
7. **Verification** — проверка, что реализовано верно
8. **Maintenance** — обновление при изменениях

## Трассировка требований (Traceability)

Матрица трассировки (Requirements Traceability Matrix, RTM) связывает требования с:

- **Источником** — кто сказал (интервью, regulation, contract)
- **Дизайном** — какой компонент / API реализует
- **Тестами** — какой тест проверяет
- **Рисками** — какие риски связаны

```
Требование ──→ Функциональный блок ──→ Тест-кейс
   ↑                                         │
   └────────────── Статус прохождения ←──────┘
```

### Пример RTM

| ID требования | Источник | Компонент | Тест-кейс | Статус |
|--------------|----------|-----------|-----------|--------|
| REQ-001 | Интервью с Ивановым | User Service | TC-101 | Passed |
| REQ-002 | Contract §3.2 | Payment | TC-201, TC-202 | Failed |

## Управление изменениями

Изменения требований неизбежны. Задача — не «запретить изменения», а **контролировать их влияние**:

1. **Change Request (CR)** — формальная заявка на изменение
2. **Impact Analysis** — оценка влияния на сроки, бюджет, архитектуру
3. **Change Control Board (CCB)** — группа, принимающая решение (approve / reject / defer)
4. **Update** — обновление документации и уведомление команды

## Атрибуты требования

Каждое requirement должно иметь метаданные:

| Атрибут | Описание | Пример |
|---------|----------|--------|
| ID | Уникальный идентификатор | REQ-042 |
| Title | Краткое название | «Авторизация по SMS» |
| Status | Текущее состояние | Draft / Approved / Implemented |
| Priority | Важность | Must / Should / Could |
| Source | Источник | Stakeholder interview 2024-03-15 |
| Owner | Ответственный | Иванов И. |
| Risk | Риск нереализации | High (регуляторные требования) |

## Инструменты для управления требованиями

| Инструмент | Уровень | Когда использовать |
|-----------|---------|-------------------|
| **Jira + Confluence** | Базовый | Agile-команды, небольшие проекты |
| **IBM DOORS** | Enterprise | Медицинские, аэрокосмические, defence |
| **Siemens Polarion** | Enterprise | Automotive, промышленность |
| **Notion / Airtable** | Стартап | Гибкие, без жёстких процессов |
| **Excel** | Прототип | Только на старте — не для production |

## Практические советы

- **Не документируйте всё подряд** — каждое requirement должно приносить ценность
- **Используйте Unique ID** — чтобы можно было ссылаться из тестов, кода, коммитов
- **Ведите историю изменений** — кто, когда и почему изменил требование
- **Автоматизируйте трассировку** — если RTM ведётся вручную, она устареет за неделю
- **Синхронизируйте с тестами** — тест-кейсы должны ссылаться на ID требований
