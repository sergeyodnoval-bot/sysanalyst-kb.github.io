# Git и контроль версий — обзорно

:::note
Git — это система контроля версий, которая отслеживает изменения в файлах, позволяет возвращаться к любой предыдущей версии и организовывать совместную работу над кодом (и не только). Для аналитика Git — способ хранить спецификации и документы как код.
:::

Представьте, что вы пишете курсовую. Вы сохраняете файл, потом правите, сохраняете, правите… И вдруг понимаете, что предыдущий вариант был лучше. Но вы не можете вернуться — файл уже перезаписан.

Вы начинаете делать: «диплом_v1.doc», «диплом_v2.doc», «диплом_финальный.doc», «диплом_финальный_2.doc»… Через неделю у вас 15 файлов и вы не помните, в каком из них последняя версия.

Git решает эту проблему. Он автоматически сохраняет **историю всех изменений** и позволяет в любой момент вернуться к любой версии.

## Что такое система контроля версий (VCS)

Система контроля версий (Version Control System) — это программа, которая:

- Отслеживает каждое изменение в каждом файле
- Помнит, кто, когда и что изменил
- Позволяет вернуться к любой предыдущей версии
- Упрощает совместную работу (несколько человек меняют одни и те же файлы)

Git — самая популярная VCS. Её создал Линус Торвальдс в 2005 году для разработки ядра Linux. Сейчас Git — стандарт де-факто в IT.

## Основные понятия Git

### Репозиторий (repo)
Папка, в которой Git отслеживает изменения. Репозиторий содержит все файлы проекта плюс служебную папку `.git` с историей изменений.

### Коммит (commit)
Снимок состояния файлов в определённый момент. Каждый коммит имеет:
- **Уникальный идентификатор** (хеш) — например, `a1b2c3d4`
- **Сообщение** — краткое описание того, что было сделано
- **Автора** и **дату**

### Ветка (branch)
Параллельная версия проекта. Основная ветка обычно называется `main` (или `master`). От неё можно создавать **ветки для новых фич** — работать в них, а потом **сливать** (merge) обратно.

Пример типового Git-процесса:
1. Вы начали задачу — создали ветку `feature/add-login`
2. Работаете, делаете коммиты
3. Завершили — делаете `merge` в `main`

### Слияние (merge)
Перенос изменений из одной ветки в другую. Если два человека меняли один и тот же файл — возникает **конфликт**, который нужно разрешить вручную.

### Пул-реквест (Pull Request / PR)
Предложение внести изменения в основную ветку. В командах PR проходит ревью: кто-то смотрит код, комментирует, утверждает.

## Как Git используется в команде

```
            main ─────●────────────────●────
                      \                /
feature/login          ●──●──●──●──●──●
                         коммиты
```

- Каждый разработчик работает в своей ветке
- Коммитит часто (каждая логическая единица работы)
- Открывает PR, когда готов
- После ревью и тестов — merge в main

## Почему аналитику нужен Git

Традиционно Git используют разработчики, но в современном IT всё больше команд практикуют **docs-as-code** — подход, при котором документация хранится и управляется как код:

### 1. Версионирование спецификаций

SRS меняется со временем. Git хранит все версии — всегда можно посмотреть, как выглядело требование месяц назад и почему оно изменилось.

### 2. Code review для документов

Те же PR можно использовать для ревью документации. Архитектор смотрит диаграмму, оставляет комментарий, утверждает.

### 3. Совместная работа

Несколько аналитиков могут одновременно работать над разными разделами документации в разных ветках.

### 4. История изменений

На вопрос «кто и когда изменил это требование?» Git отвечает одной командой.

### 5. Интеграция с CI/CD

Документация публикуется автоматически при merge в main (как этот сайт на GitHub Pages).

## Основные команды Git (для понимания, не запоминать)

| Команда | Что делает |
|---------|-----------|
| `git init` | Создать новый репозиторий |
| `git clone` | Скачать репозиторий из GitHub |
| `git add` | Добавить файлы к следующему коммиту |
| `git commit` | Сохранить снимок изменений |
| `git push` | Отправить коммиты на сервер |
| `git pull` | Забрать изменения с сервера |
| `git branch` | Посмотреть / создать ветки |
| `git merge` | Слить ветки |
| `git log` | Посмотреть историю коммитов |
| `git diff` | Посмотреть различия между версиями |

Скорее всего, аналитику не нужно помнить синтаксис — достаточно понимать концепции. Git-клиенты (GitHub Desktop, VS Code, SourceTree) имеют удобный интерфейс без командной строки.

## Почему это важно для аналитика

Даже если вы не будете сами пользоваться Git — вы будете работать в Git-ориентированной среде:

- **Бэклог в Jira** ссылается на коммиты — вы видите, что задача в разработке
- **PR в GitHub/GitLab** содержит описание изменений — вы можете понять, что было сделано
- **Docs-as-code** — вы пишете спецификации в Markdown, кладёте в репозиторий, и они автоматически публикуются
- **Понимание Git-флоу** помогает общаться с разработчиками на одном языке

## Проверь себя

**1. Чем коммит отличается от обычного сохранения файла (Ctrl+S)?**

Ctrl+S просто записывает текущее состояние на диск, теряя предыдущую версию. Коммит сохраняет снимок всех файлов одновременно с сообщением, автором и датой — можно вернуться к любому коммиту в истории.

**2. Зачем нужны ветки в Git?**

Чтобы работать над разными задачами параллельно, не мешая друг другу. Одна ветка — одна фича или одна задача. Когда фича готова, ветка сливается в основную.

**3. Может ли аналитик использовать Git, если он не пишет код?**

Да. Для docs-as-code, хранения спецификаций, совместной работы над документацией. Многие аналитики хранят шаблоны, BPMN-диаграммы (в формате XML), схемы API в Git.
