# Основы документации системного аналитика

:::note
Документация — это основной «продукт» работы аналитика. Хорошая документация делает требования понятными, однозначными и реализуемыми. Плохая документация — причина срывов сроков, багов и конфликтов.
:::

Представьте рецепт пирога. Если в рецепте написано «добавить муку, яйца, сахар, испечь» — пирог, скорее всего, не получится. Сколько муки? Какие яйца? Сколько печь? При какой температуре?

Хороший рецепт даёт точные пропорции, последовательность шагов и ожидаемый результат.

Документация аналитика — это такой же рецепт, только вместо пирога — программное обеспечение.

## Зачем нужна документация

- **Чтобы не забыть** — требования, согласованные устно, забываются за неделю
- **Чтобы согласовать** — заказчик подписывает документ и не может потом сказать «я не то имел в виду»
- **Чтобы передать** — новый разработчик читает документацию и понимает, что нужно делать
- **Чтобы измерить** — по документации проверяют, всё ли реализовано

## Основные документы аналитика

### 1. Vision / Product Scope (концепция)

Одностраничный документ, отвечающий на вопросы:
- Какую проблему решаем?
- Кто целевая аудитория?
- Каковы границы системы (что делаем, а что — нет)?

**Когда пишется**: в самом начале, до детального анализа.

**Кто читает**: заказчик, менеджер продукта, команда.

### 2. SRS (Software Requirements Specification)

Подробный документ с описанием всех требований к системе.

**Содержит**:
- Функциональные требования (что система должна делать)
- Нефункциональные требования (производительность, безопасность, совместимость)
- Ограничения (технологии, бюджет, сроки)
- Интерфейсы (API, пользовательские экраны)

**Когда пишется**: этап анализа требований (в Waterfall — полностью, в Agile — частично).

### 3. User Stories

Короткие формулировки требований от лица пользователя:
«Как зарегистрированный пользователь, я хочу скачать отчёт в PDF, чтобы отправить его клиенту».

Каждая User Story сопровождается критериями приёмки (BDD-сценариями).

**Когда пишется**: в Agile — каждый спринт, ведётся в бэклоге.

### 4. Use Case / Вариант использования

Описание сценария взаимодействия пользователя с системой. Включает основной поток, альтернативы и исключения.

**Когда пишется**: для сложных сценариев, где важен пошаговый сценарий.

### 5. BDD-сценарии

Сценарии в формате Given-When-Then:
- Given: пользователь авторизован
- When: он нажимает «Скачать отчёт»
- Then: система начинает загрузку PDF-файла

### 6. Техническая документация

Включает:
- API-спецификации (OpenAPI / Swagger)
- Диаграммы последовательности (Sequence diagram)
- Модели данных (ER-диаграммы)
- Описание архитектуры (C4 diagrams)

### 7. Meeting Notes (протоколы встреч)

Запись решений, принятых на встречах. Кто, что, когда решил, какие есть риски.

## Принципы хорошей документации

### 1. Одна мысль — один документ

Не смешивайте бизнес-требования с техническими деталями в одном месте. Для каждого типа информации — свой документ.

### 2. Пишите для читателя

Разработчику нужны детали реализации. Заказчику — бизнес-ценность. Один документ может иметь разные разделы для разных аудиторий.

### 3. Избегайте двусмысленности

| Плохо | Хорошо |
|-------|--------|
| Система должна быть быстрой | API отвечает не дольше 500 мс |
| Удобный интерфейс | Кнопка «Заказать» находится в правом верхнем углу |
| Много пользователей | 5000 одновременных пользователей |

### 4. Поддерживайте актуальность

Нет ничего хуже устаревшей документации. Если требование изменилось — измените документ. Иначе разработчик реализует не то.

### 5. Используйте шаблоны

Шаблоны экономят время и обеспечивают полноту. В каждой компании есть свои шаблоны — изучите их.

## Почему это важно для аналитика

Документация — это **лицо аналитика**. По качеству документации судят о профессионализме:

- Хорошая документация экономит время команды: разработчик не задаёт лишних вопросов, тестировщик пишет корректные тест-кейсы
- Документация — доказательство работы: без неё вы не можете подтвердить, что требование было согласовано
- Документация живёт дольше проекта: через год к ней вернутся, чтобы вспомнить, как всё устроено

## Ключевые термины

| Термин | Значение |
|--------|---------|
| **SRS** | Software Requirements Specification — спецификация требований |
| **FRD** | Functional Requirements Document — функциональные требования |
| **Vision** | Концепция продукта (зачем, для кого) |
| **User Story** | Формулировка требования от лица пользователя |
| **Use Case** | Вариант использования — пошаговый сценарий |
| **BDD** | Behaviour-Driven Development — сценарии Given-When-Then |
| **Meeting Notes** | Протокол встречи |
| **API Spec** | Спецификация API (OpenAPI / Swagger) |
| **AC** | Acceptance Criteria — критерии приёмки |

## Проверь себя

**1. Какой документ нужно написать в первую очередь?**

Vision / Product Scope. Прежде чем писать детальные требования, нужно понять, какую проблему решаем и для кого. Иначе рискуете написать требования к ненужной системе.

**2. Почему плохая документация хуже, чем её отсутствие?**

Потому что устаревшая или двусмысленная документация вводит в заблуждение. Разработчик может реализовать не то, что нужно, и скажет: «но ведь так написано в документации». Отсутствие документации хотя бы заставляет задавать вопросы.

**3. Как часто нужно обновлять документацию?**

Каждый раз, когда меняются требования. В Agile — в конце каждого спринта, после уточнения бэклога. В идеале — документ должен отражать текущее состояние системы, а не то, какой она задумывалась год назад.
