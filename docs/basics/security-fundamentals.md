# Основы безопасности для системного аналитика

Безопасность — не про хакеров и шифрование ради шифрования. Это про **кто к чему имеет доступ** и **как данные защищены при передаче**. Аналитику достаточно понимать 3 базовые темы: шифрование, аутентификация и сертификаты.

## Шифрование

### Симметричное

Один ключ и на замок, и на отмычку. Быстро, но ключ надо передать — в этом уязвимость.

**Пример:** Wi-Fi пароль — все в офисе знают один пароль.

### Асимметричное

Два ключа: **публичный** (можно всем) и **приватный** (только у владельца). Данные, зашифрованные публичным ключом, расшифровываются только приватным.

**Пример:** Вы даёте миру открытый замок, а ключ от него храните у себя. Любой может закрыть этим замком посылку, но открыть — только вы.

### TLS/SSL

Протокол, который объединяет оба подхода:

| Шаг | Что происходит |
|-----|---------------|
| 1 | Клиент стучится к серверу |
| 2 | Сервер показывает сертификат (публичный ключ + подпись) |
| 3 | Клиент проверяет сертификат, генерирует симметричный ключ, шифрует его публичным ключом сервера |
| 4 | Сервер расшифровывает приватным ключом — теперь у обоих есть общий ключ |
| 5 | Дальше общаются по симметричному шифрованию (быстро) |

## Сертификаты

Сертификат — это паспорт сайта. Его выдают **Удостоверяющие центры** (CA — Certificate Authority).

**Зачем:** чтобы вы не отправили пароль мошеннику, который притворяется сайтом банка.

Проверка цепочки: браузер доверяет CA → CA подписал сертификат Яндекс.Такси → сертификат предъявил Яндекс.Такси — значит, я точно разговариваю с Яндекс.Такси.

**Что важно аналитику:**
- HTTPS = TLS, не магия
- Если интеграция идёт по HTTPS, шифрование уже есть на транспортном уровне
- Для mTLS (двусторонний) сертификат нужен и клиенту — часто встречается в B2B интеграциях

## Аутентификация и авторизация

| Термин | Что значит | Вопрос |
|--------|-----------|--------|
| **Аутентификация** | «Кто ты?» | Вход по логину/паролю, по токену, по сертификату |
| **Авторизация** | «Что тебе можно?» | Права: админ, редактор, читатель |

### Basic Auth

Логин и пароль в заголовке HTTP, закодированные в Base64.
- Не шифрует, только кодирует
- Без HTTPS — пароль в открытом виде
- **Где встречается:** старые API, тестовые стенды

### Bearer Token (JWT)

После входа сервер выдаёт токен — строку, которую клиент прикладывает к каждому запросу.

**JWT (JSON Web Token):** токен, внутри которого JSON с данными (кто, когда истекёт, роли). Подписан, поэтому его нельзя подделать.

```
header.payload.signature
```

**Где встречается:** современные REST API, микросервисы.

### OAuth 2.0

Протокол **делегирования доступа** — сайт разрешает другому сайту что-то сделать от имени пользователя.

**Пример:** «Войти через Google» — Google говорит сайту: «это Сергей, ему можно почту читать», но пароль Google сайту не передаёт.

### API Keys

Статический ключ, который сервис выдаёт клиенту для доступа к API.

| Плюс | Минус |
|------|-------|
| Просто: одна строка | Не различает пользователей |
| Легко встроить в код | Ключ живёт вечно, пока не отзовут |

**Где встречается:** SaaS-сервисы (OpenAI, Stripe), платёжные шлюзы.

## Почему это важно аналитику

- Вы описываете требования к безопасности в SRS и технических заданиях
- Выбираете способ аутентификации для интеграции
- Понимаете ограничения: если API передаёт персональные данные — нужен HTTPS, часто — шифрование на уровне приложения
- Архитектор или разработчик скажет «тут mTLS», а вы понимаете, что это значит и сколько стоит

## Ключевые термины

| Термин | Значение |
|--------|---------|
| TLS | Протокол шифрования при передаче |
| CA | Центр, который выпускает сертификаты |
| JWT | Токен с данными внутри |
| OAuth 2.0 | Протокол делегирования доступа |
| mTLS | TLS с проверкой сертификата с обеих сторон |

## Проверь себя

1. **Чем отличается Basic Auth от Bearer Token?**
   *Ответ:* Basic Auth передаёт логин и пароль на каждый запрос (в Base64, не шифрованные). Bearer Token — одноразовая строка, полученная после входа. Token можно отозвать, пароль — нет.

2. **Зачем нужен Удостоверяющий центр (CA)?**
   *Ответ:* Чтобы браузер мог проверить, что сертификат сайта настоящий. CA подписывает сертификат, и браузер доверяет цепочке: CA → сертификат сайта.

3. **Что такое mTLS и когда он используется?**
   *Ответ:* Двусторонний TLS — и клиент, и сервер предъявляют сертификаты. Используется в B2B интеграциях (например, при обмене данными между банками), где обе стороны должны быть уверены друг в друге.

4. **В чём разница между аутентификацией и авторизацией?**
   *Ответ:* Аутентификация отвечает на вопрос «кто ты?», авторизация — «что тебе можно?». Сначала ты доказываешь личность, потом получаешь права.
