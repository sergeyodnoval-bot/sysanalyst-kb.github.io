# Что такое API

:::note
API — это меню ресторана. Оно говорит, что кухня умеет готовить, какие блюда доступны и как их заказывать. Без меню вы не знаете, что можно заказать.
:::

Вспомните метафору ресторана из прошлой статьи. Вы (клиент) общаетесь с кухней (сервером) через официанта. Но как вы понимаете, что вообще можно заказать? Вы смотрите в меню.

**API (Application Programming Interface)** — это и есть меню для программ. Оно описывает, какие «блюда» (функции) предлагает сервер, как их «заказывать» (какие данные отправить) и что вы получите в ответ.

## Что такое API простыми словами

API — это контракт между клиентом и сервером. Он говорит:

- **Что я умею делать.** Например, «я могу показать погоду в любом городе»
- **Как меня об этом попросить.** Например, «отправь запрос на адрес /weather с параметром city»
- **Что я верну в ответ.** Например, «верну температуру, влажность и скорость ветра в формате JSON»

Без API клиент не знает, что можно сделать. С API — может чётко запросить нужное и получить предсказуемый результат.

## Пример из жизни

Когда вы смотрите погоду в приложении на телефоне, происходит следующее:

1. Приложение (клиент) знает, что у сервиса погоды есть API
2. Приложение отправляет запрос: «Покажи погоду в Москве»
3. Сервер погоды обрабатывает запрос, достаёт данные из базы
4. Сервер возвращает ответ: `{ "city": "Moscow", "temp": 18, "humidity": 65 }`
5. Приложение показывает вам красивую картинку с температурой

Вы не видите API, но он работает каждый раз, когда приложение получает данные из интернета.

## Какие бывают API

**REST API** — самый популярный тип. Работает через HTTP (как веб-сайты). Простой, понятный, используется в 90% случаев. Мы подробно разберём его в следующих статьях.

**GraphQL** — более современный подход. Клиент может запросить ровно те данные, которые нужны, без лишнего. Сложнее в реализации, но гибче.

**SOAP** — старый стандарт, используется в корпоративных системах (банки, госуслуги). Сложный, но надёжный.

**WebSocket** — для двусторонней связи в реальном времени. Используется в чатах, онлайн-играх, трейдинговых платформах.

Вам пока не нужно понимать разницу между ними. Достаточно знать, что API бывают разные, и выбор зависит от задачи.

## Почему API — это хорошо

**Стандартизация.** Если все клиенты знают API, они могут работать с сервером одинаково. Не нужно каждый раз придумывать новый способ общения.

**Безопасность.** API может ограничивать, кто имеет доступ и что можно делать. Например, показать погоду может кто угодно, а изменить профиль — только авторизованный пользователь.

**Независимость.** Сервер может меняться внутри (новая база данных, другой язык программирования), но если API остаётся тем же — клиенты ничего не замечают.

**Переиспользование.** Один API могут использовать десятки клиентов: веб-сайт, мобильное приложение, умные часы, сторонние сервисы.

## Почему это важно для аналитика

Когда вы проектируете систему, которая должна общаться с другими системами (а таких систем большинство), вы проектируете API.

Вам нужно решить:

- **Какие функции предоставить?** Что клиенты смогут делать через API?
- **В каком формате передавать данные?** JSON, XML, или что-то ещё?
- **Как обрабатывать ошибки?** Что вернуть, если данных нет или пользователь не авторизован?
- **Как обеспечить безопасность?** Кто может вызывать API? Нужна ли авторизация?
- **Как версионировать?** Что делать, когда нужно изменить API, но старые клиенты ещё работают?

Это непростые вопросы, и ответы на них определяют, насколько удобной и надёжной будет система. Хороший API — это половина успеха интеграции.

## Ключевые термины

- **API (Application Programming Interface)** — интерфейс взаимодействия между программами
- **Endpoint** — конкретный «адрес» в API, куда отправляется запрос (например, `/weather`)
- **Request (запрос)** — сообщение от клиента к серверу
- **Response (ответ)** — сообщение от сервера к клиенту
- **JSON** — формат обмена данными, понятный и людям, и машинам

## Проверь себя

1. Что произойдёт, если сервер изменит формат ответа API?
2. Зачем нужно версионировать API?
3. Почему API называют «контрактом»?

**Ответы:** 1) Все клиенты, которые ожидали старый формат, сломаются. 2) Чтобы старые клиенты продолжали работать, пока новые используют обновлённую версию. 3) Потому что API чётко описывает, что клиент может ожидать от сервера — как контракт фиксирует обязательства сторон.
