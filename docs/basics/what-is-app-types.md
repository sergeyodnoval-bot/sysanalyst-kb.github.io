# Типы приложений: веб, мобайл, десктоп

:::note
Приложения делятся на три основных типа по среде выполнения: веб (в браузере), мобайл (на телефоне), десктоп (на компьютере). У каждого — свои особенности, ограничения и сценарии использования. Выбор типа приложения — одно из первых архитектурных решений в проекте.
:::

Представьте, что вам нужно добраться из точки А в точку Б. Можно поехать на велосипеде (быстро на короткие расстояния, но в дождь мокро), на машине (комфортно в любую погоду, но нужна парковка) или на поезде (надёжно и много места, но ходит по расписанию).

Так же и с приложениями: веб, мобайл и десктоп — это разные «транспортные средства» для доставки функциональности пользователю. Каждый хорош для своего сценария.

## Десктоп-приложения

Устанавливаются на компьютер пользователя. Работают непосредственно под управлением ОС (Windows, macOS, Linux).

**Примеры**: Microsoft Word, Adobe Photoshop, 1С:Предприятие, Slack (десктоп-версия)

**Плюсы**:
- Полный доступ к ресурсам компьютера (файловая система, устройства, USB)
- Работают без интернета
- Максимальная производительность
- Богатый пользовательский интерфейс

**Минусы**:
- Требуют установки и обновления
- Привязаны к ОС (под каждую нужна своя версия)
- Сложнее распространять и обновлять
- Данные хранятся локально (меньше контроля)

## Веб-приложения

Работают в браузере. Пользователю не нужно ничего устанавливать — достаточно открыть ссылку.

**Примеры**: Gmail, Trello, Google Docs, Figma

**Плюсы**:
- Доступны с любого устройства с браузером
- Обновления мгновенные (достаточно обновить сервер)
- Не требуют установки
- Данные на сервере — можно работать с любого устройства

**Минусы**:
- Требуют интернета (если не использовать PWA)
- Ограниченный доступ к устройству (камера, файлы — через разрешения браузера)
- Зависимость от браузера (разные браузеры — разное поведение)
- Меньшая производительность, чем у десктопа

**Разновидности веб-приложений**:

| Тип | Как работает | Пример |
|-----|-------------|--------|
| **SPA** (Single Page Application) | Вся логика в браузере, сервер только отдаёт данные | Gmail, Trello |
| **SSR** (Server-Side Rendering) | Сервер собирает HTML-страницу и отдаёт готовую | Wikipedia, новостные сайты |
| **PWA** (Progressive Web App) | Веб-приложение, которое можно «установить» на устройство | Pinterest, Starbucks |

## Мобильные приложения

Устанавливаются на смартфон или планшет через App Store / Google Play.

**Примеры**: Instagram, Telegram, Uber, банковские приложения

**Плюсы**:
- Push-уведомления
- Доступ к функциям телефона (GPS, камера, акселерометр, контакты)
- Работают оффлайн (если данные закэшированы)
- Оптимизированы под сенсорный интерфейс
- Продаются через магазины приложений

**Минусы**:
- Нужно разрабатывать отдельно для iOS и Android (или использовать кроссплатформу)
- Долгий процесс публикации обновлений (модерация в магазинах)
- Меньший экран, специфические жесты
- Выше стоимость разработки (две платформы)

**Разновидности**:

| Тип | Описание | Примеры |
|-----|----------|---------|
| **Native** | Под каждую платформу отдельный код | Swift (iOS), Kotlin (Android) |
| **Cross-platform** | Один код для обеих платформ | React Native, Flutter |
| **Hybrid** | Веб-приложение в обёртке нативного приложения | Ionic, Cordova |

## Как выбрать тип приложения

Типовое решение зависит от сценария:

- **Внутренний корпоративный инструмент** → веб (доступ с любого ПК, легко обновлять)
- **Продукт для потребителей (B2C)** → мобайл (телефон всегда под рукой) + веб (для первого знакомства)
- **Редактор / тяжелый софт** → десктоп (нужна производительность)
- **MVP / прототип** → веб (быстрее запустить)
- **Для склада / производства** → мобайл на базе Android (дешёвые терминалы)

## Почему это важно для аналитика

При анализе требований нужно всегда понимать, **на какой платформе будет работать система**:

- **Функциональные требования** зависят от возможностей платформы (мобильное приложение не может постоянно сканировать QR-коды — сядет батарея)
- **UX/UI** сильно различается на десктопе и мобилке (большая форма с 20 полями неудобна на телефоне)
- **Производительность**: веб-приложение на слабом ноутбуке в IE11 — это не то же самое, что десктоп на мощном ПК
- **Безопасность**: мобильное приложение может использовать биометрию, веб — только через браузер
- **Интеграции**: десктоп может работать с USB-устройствами, веб — только через WebUSB (ограниченно)

## Ключевые термины

| Термин | Значение |
|--------|---------|
| **SPA** | Single Page Application — вся логика в браузере |
| **SSR** | Server-Side Rendering — страницы собираются на сервере |
| **PWA** | Progressive Web App — сайт, который ведёт себя как приложение |
| **Native app** | Приложение под конкретную ОС на её родном языке |
| **Cross-platform** | Единый код для iOS и Android (React Native, Flutter) |
| **Hybrid app** | Веб-приложение в нативной обёртке |

## Проверь себя

**1. Какое приложение вы выберете для системы управления складом с терминалами сбора данных?**

Мобильное (на Android), так как терминалы работают на Android, они мобильны, и приложению нужен доступ к сканеру штрихкодов.

**2. Почему банки делают и мобильное приложение, и веб-версию?**

Мобильное — для повседневных операций (проверить баланс, перевести деньги). Веб — для сложных операций (открыть вклад, скачать выписку, оформить кредит), где нужен большой экран и клавиатура.

**3. Чем SPA отличается от SSR?**

В SPA вся логика интерфейса загружается один раз и работает в браузере, сервер только отдаёт данные. В SSR сервер каждый раз собирает готовую HTML-страницу. SPA быстрее реагирует на действия пользователя, SSR лучше индексируется поисковиками.
