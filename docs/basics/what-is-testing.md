# Основы тестирования и QA

:::note
Тестирование — это проверка того, что программа работает согласно требованиям и не делает того, чего не должна. QA (Quality Assurance) — более широкая концепция: не только поиск багов, но и предотвращение ошибок на всех этапах разработки.
:::

Представьте, что вы заказали пиццу. Вы проверили, что:
1. Пицца горячая (функциональность)
2. С теми ингредиентами, которые заказывали (корректность)
3. Вкусная (качество)
4. Курьер не пролил лимонад по пути (отсутствие багов)
5. Укладывается в заявленные 60 минут (производительность)

Если бы вы не проверили — съели бы холодную пиццу не с тем набором, и заплатили бы за испорченный лимонад. Тестирование в IT — такой же акт проверки, только вместо пиццы — программный продукт.

## Что такое QA и чем отличается от тестирования

**QA (Quality Assurance)** — весь процесс обеспечения качества: стандарты, процессы, метрики, код-ревью, CI/CD. QA отвечает на вопрос «как сделать так, чтобы багов было как можно меньше?»

**Тестирование** — часть QA. Это конкретная проверка программы: запуск тестов, поиск ошибок, проверка соответствия требованиям.

## Виды тестирования

### По уровню

**Модульное (unit) тестирование**
- Проверка отдельных функций или методов
- Пишут разработчики
- Пример: функция расчёта скидки — проверить, что при сумме >1000 скидка 10%

**Интеграционное тестирование**
- Проверка взаимодействия между модулями
- Пример: после регистрации пользователя данные должны появиться в базе

**Системное тестирование**
- Проверка всей системы целиком
- Пример: полный сценарий «заказ товара» — от поиска до оплаты

**Приёмочное тестирование (UAT)**
- Проверка заказчиком или пользователем
- Пример: заказчик проходит по всем user stories и подтверждает, что всё работает

### По способу

**Ручное тестирование (manual)**
- Человек кликает в интерфейсе, проверяет сценарии
- Нужно для UX, неочевидных сценариев
- Дорого и медленно, но незаменимо

**Автоматизированное тестирование (automation)**
- Скрипты сами выполняют проверки
- Быстро и надёжно для регресса (повторных проверок)
- Требует разработки тестов

### По цели

| Вид | Что проверяем | Пример |
|-----|--------------|--------|
| **Функциональное** | Программа делает то, что должна | Кнопка «Отправить» отправляет |
| **Нагрузочное** | Программа выдерживает нагрузку | 1000 пользователей одновременно |
| **UI/UX** | Удобство и внешний вид | Кнопка видна, шрифт читаем |
| **Безопасность** | Защита от взлома | Нельзя войти без пароля |
| **Регрессионное** | Новый код не сломал старое | После обновления всё работает как раньше |
| **Smoke-тест** | Базовая работоспособность | Главная страница открывается |

## Роль аналитика в тестировании

Аналитик — не тестировщик, но активно участвует в процессе:

### 1. Пишет критерии приёмки (Acceptance Criteria)

Это часть задачи аналитика — описать условие, при котором задача считается выполненной:
- «Поле email не позволяет ввести некорректный адрес»
- «После нажатия «Сохранить» данные появляются в таблице»

### 2. Участвует в UAT

Приёмочное тестирование проводит аналитик вместе с заказчиком. Аналитик готовит сценарии, заказчик их выполняет и подтверждает.

### 3. Проверяет тест-кейсы

Тестировщики пишут тест-кейсы на основе требований. Аналитик проверяет, все ли сценарии покрыты и правильно ли они отражают требования.

### 4. Участвует в баг-трекинге

Заводит баги, проверяет их приоритет, подтверждает исправление.

## Почему это важно для аналитика

- Требование должно быть **тестируемым**: «система должна быть быстрой» — плохое требование, «время ответа API не более 500 мс при 100 одновременных запросах» — тестируемое
- Аналитик отвечает за то, что баг — это баг, а не фича, которую забыли согласовать
- Знание видов тестирования помогает писать корректные критерии приёмки (functional vs non-functional vs security)
- Участие в тестировании — лучший способ понять, правильные ли требования вы написали

## Ключевые термины

| Термин | Значение |
|--------|---------|
| **QA** | Quality Assurance — обеспечение качества |
| **QC** | Quality Control — контроль качества (тестирование) |
| **Баг (Bug)** | Ошибка в программе |
| **Дефект** | Отклонение от требований |
| **Тест-кейс** | Описание одного тестового сценария |
| **UAT** | User Acceptance Testing — приёмочное тестирование |
| **Регресс** | Повторное тестирование после изменений |
| **TC (Test Case)** | Документ с шагами и ожидаемым результатом |

## Проверь себя

**1. Чем отличается QA от тестирования?**

QA — это система процессов, предотвращающая ошибки (стандарты, ревью, CI/CD). Тестирование — конкретная проверка программы на наличие ошибок. QA шире, тестирование — часть QA.

**2. Почему требование должно быть тестируемым?**

Потому что иначе невозможно проверить, выполнено ли оно. «Удобный интерфейс» нельзя протестировать объективно. «Кнопка «Заказать» доступна не более чем в 2 клика от главной» — можно.

**3. Какую роль играет аналитик в UAT?**

Аналитик готовит сценарии приёмочного тестирования, сопровождает заказчика в процессе, фиксирует результаты, помогает оформлять замечания. По итогам UAT аналитик обновляет требования, если выявлены расхождения.
