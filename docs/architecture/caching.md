:::info TL;DR
Кэширование — сохранение результата дорогой операции, чтобы при повторном запросе отдать его быстро. Аналитик специфицирует: что кэшировать, как долго хранить, как инвалидировать (обновлять). Кэш бывает на клиенте (браузер, CDN), на сервере (in-memory, Redis) и на уровне БД.
:::

## Зачем кэширование

Без кэша каждый запрос доходит до БД или внешнего API:

```
Пользователь ──→ API ──→ БД (10ms) ──→ ответ
Пользователь ──→ API ──→ БД (10ms) ──→ ответ  (10 000 раз = 100 сек)
```

С кэшем:

```
Пользователь ──→ API ──→ Redis ──(miss)──→ БД (10ms) ──→ Redis ──→ ответ
Пользователь ──→ API ──→ Redis ──(hit)──────────────────────────→ ответ (1ms)
```

**Выигрыш:** latency падает с 10ms до 1ms, нагрузка на БД снижается в 10–100 раз.

## Где бывает кэш

```
┌────────┐
│ Browser│ ←─ Browser cache (HTTP кэш)
└───┬────┘
    │
┌───▼────┐
│  CDN   │ ←─ Edge cache (статика, изображения, API-ответы)
└───┬────┘
    │
┌───▼────┐
│Gateway │ ←─ API Gateway cache
└───┬────┘
    │
┌───▼────┐
│Service │ ←─ In-memory cache (Caffeine, Guava)
└───┬────┘
    │
┌───▼────┐
│  Redis │ ←─ Distributed cache (shared между сервисами)
└───┬────┘
    │
┌───▼────┐
│   БД   │ ←─ Database cache (буферный кэш)
└────────┘
```

**Для аналитика:** на каждом уровне — свои требования к TTL, инвалидации, объёму.

## Стратегии кэширования

### 1. Cache-Aside (ленивое заполнение)

Самый популярный паттерн. Приложение само управляет кэшем:

```
Запрос ──→ читаем из кэша
               │
         ┌─────┴─────┐
         ↓           ↓
       Hit (есть)   Miss (нет)
         │           │
         ↓           ↓
       Ответ    читаем из БД
                   │
                   ↓
              сохраняем в кэш
                   │
                   ↓
                Ответ
```

**Плюсы:** просто, кэшируется только то, что реально запрашивают.
**Минусы:** N+1 проблема — первый запрос всегда медленный.

### 2. Write-Through (синхронная запись)

Данные пишутся одновременно в БД и в кэш:

```
Запись ──→ пишем в кэш ──→ пишем в БД ──→ OK
```

**Плюсы:** кэш всегда актуален.
**Минусы:** запись медленнее (две операции).

### 3. Write-Behind (асинхронная запись)

Данные пишутся в кэш, а потом асинхронно в БД:

```
Запись ──→ пишем в кэш ──→ OK (мгновенно)
                           ──→ пишем в БД (фоном)
```

**Плюсы:** быстрая запись.
**Минусы:** риск потери данных при падении до записи в БД.

### 4. TTL (Time-To-Live)

Данные живут в кэше фиксированное время, потом удаляются:

```
Кэш: {ключ: значение, ttl: 600}  ← через 600 сек удалится
```

**Плюсы:** просто, не нужна инвалидация.
**Минусы:** данные могут быть устаревшими (stale) до истечения TTL.

## Инвалидация кэша

Самая сложная проблема в кэшировании. Данные в кэше устарели, а в БД — новые.

### Способы инвалидации

| Метод | Как работает | Когда использовать |
|-------|-------------|-------------------|
| **TTL** | Данные живут N секунд | Данные меняются редко, допустима небольшая задержка |
| **Event-based** | При изменении данных отправляется событие «очистить кэш» | Нужна мгновенная актуальность |
| **Key-based** | Изменяем ключ (order_v1 → order_v2) | Статика, версионированные данные |
| **Manual** | Админ нажимает «очистить кэш» | Аварийные ситуации |

## Redis

Самый популярный in-memory data store для кэширования.

**Возможности:**
- Хранение: строки, хэши, списки, множества
- TTL для каждого ключа
- Pub/Sub для уведомлений
- Persistence (периодическая запись на диск)
- Cluster mode (шардирование)

**Для аналитика:** Redis — не просто кэш, а универсальный data store. В спецификации указывать: какие данные хранить, какой TTL, нужна ли персистентность.

## CDN (Content Delivery Network)

Сеть серверов по всему миру, которые кэшируют статический контент (HTML, JS, CSS, изображения).

```
Пользователь (Россия) ──→ CDN Edge (Москва) ──→ Origin Server (EU)
                              ↓ (кэш)
                         Ответ из Edge
```

**Для аналитика:**
- CDN ускоряет загрузку статики в 5–10 раз
- Правила кэширования задаются через HTTP-заголовки: `Cache-Control: max-age=3600`
- CDN не подходит для персонализированного контента

## Кэширование API-ответов

### HTTP-кэширование

Аналитик специфицирует HTTP-заголовки для кэширования:

| Заголовок | Описание | Пример |
|-----------|----------|--------|
| `Cache-Control: max-age=3600` | Кэшировать на 1 час | Статичный список стран |
| `Cache-Control: no-cache` | Не кэшировать (проверять актуальность) | Баланс пользователя |
| `Cache-Control: private` | Кэшировать только в браузере (не CDN) | Личные данные |
| `ETag: "abc123"` | Версия ресурса для условного запроса | `If-None-Match: "abc123"` |
| `Expires: Sat, 01 Jan 2025...` | Устаревший формат, используйте Cache-Control | — |

## Требования к кэшированию (спецификация)

Аналитик должен определить:

| Параметр | Пример |
|----------|--------|
| Что кэшировать | Список продуктов, справочники, результаты поиска |
| TTL | 5 минут для продуктов, 1 день для справочников |
| Стратегия | Cache-Aside + TTL |
| Инвалидация | Event-based: при изменении продукта → сброс кэша |
| Объём кэша | ~500 MB | |
| Технология | Redis (shared cache между сервисами) |
| HTTP-кэширование | Cache-Control для статики: max-age=3600 |
| CDN | CloudFront / Cloudflare для статического контента |

### Чего не кэшировать

- Персональные данные пользователя (без `private`)
- Динамический контент (баланс, статус заказа)
- Данные, которые меняются каждую секунду (курсы валют)
- Контент по запросу (PDF-отчёты с датой генерации)

## Ключевые термины

- **Cache Hit** — данные найдены в кэше
- **Cache Miss** — данных в кэше нет, нужно читать из источника
- **TTL (Time-To-Live)** — время жизни записи в кэше
- **Cache-Aside** — стратегия: приложение само управляет кэшем
- **Write-Through** — запись одновременно в кэш и БД
- **CDN** — распределённая сеть кэширующих серверов
- **Cache Invalidation** — обновление устаревших данных в кэше

## Проверь себя

1. **Какие бывают стратегии кэширования и чем отличаются?**
   *Ответ:* Cache-Aside (ленивое, популярное), Write-Through (синхронное), Write-Behind (асинхронное), TTL (авто-удаление). Выбор зависит от того, критична ли актуальность данных.

2. **Почему инвалидация кэша — сложная проблема?**
   *Ответ:* Нужно знать, что данные изменились, и успеть очистить кэш до того, как их запросят stale. Простое TTL даёт задержку, event-based сложнее, но точнее.

3. **Какие HTTP-заголовки управляют кэшированием?**
   *Ответ:* Cache-Control (max-age, no-cache, private), ETag (условный запрос), Expires (устарел).
