:::info TL;DR
Distributed tracing (распределённая трассировка) позволяет проследить путь запроса через все сервисы микросервисной архитектуры. Когда пользователь нажимает кнопку «Оформить заказ», запрос проходит через 5–10 сервисов. Tracing показывает, где возникла задержка или ошибка. Аналитик специфицирует, какие операции трейсить и какие метаданные передавать.
:::

## Зачем аналитику понимать tracing

Без трассировки расследование проблемы в микросервисах выглядит так:
1. Пользователь жалуется на медленный ответ
2. DevOps проверяет логи каждого сервиса по отдельности
3. Находит, что payment-service ответил за 3 секунды
4. Но не знает, почему — может, БД тормозила, может, внешний API

С трассировкой:

```
Запрос: POST /orders
  │
  ├─ gateway: 50ms
  ├─ order-svc: 200ms
  │   ├─ DB query: 150ms
  │   └─ payment-svc: 800ms ◄── вот проблема!
  │       ├─ HTTP call to bank: 700ms ◄── external API тормозит
  │       └─ Cache check: 100ms
  └─ notification-svc: 100ms
```

**Аналитик использует трассировку для:** спецификации NFR, анализа узких мест, определения SLA для внешних интеграций.

## Как работает distributed tracing

Основная идея: **контекст трассировки** передаётся между сервисами через HTTP-заголовки или заголовки сообщений.

```
Trace = всё дерево вызовов для одного запроса.
Span = один шаг/операция внутри одного сервиса.
```

```
Trace ID: abc123
  │
  ├─ Span A: API Gateway (root span)
  │   │  duration: 50ms
  │   │
  │   ├─ Span B: order-svc
  │   │   │  duration: 200ms
  │   │   │  parent: Span A
  │   │   │
  │   │   ├─ Span C: DB query
  │   │   │   duration: 150ms
  │   │   │   parent: Span B
  │   │   │
  │   │   └─ Span D: HTTP call to payment-svc
  │   │       duration: 800ms
  │   │       parent: Span B
  │   │
  │   └─ Span E: notification-svc
  │       duration: 100ms
  │       parent: Span A
```

**Trace ID** = общий идентификатор запроса.
**Span ID** = идентификатор конкретной операции.
**Parent Span ID** = ссылка на родительский спан (выстроит дерево).

## OpenTelemetry (OTel)

OpenTelemetry — стандарт де-факто для сбора трейсов, метрик и логов. Вытесняет OpenTracing и OpenCensus.

**Что даёт OpenTelemetry:**
- Единый SDK для любого языка
- Автоматическая инструментация популярных библиотек (HTTP-клиенты, gRPC, DB-драйверы)
- Экспорт в любую систему: Jaeger, Zipkin, Grafana Tempo, Datadog, New Relic
- Поддержка трейсов, метрик и логов

**Для аналитика:** требование «поддержка OpenTelemetry» в спецификации = система совместима с любым бэкендом трассировки.

## Системы трассировки

### Jaeger

Open-source от Uber. Показывает дерево вызовов, длительность каждого спана, теги и логи.

**Когда выбирать:** бюджет ограничен, нужна самописная инфраструктура.

### Zipkin

Старый open-source стандарт от Twitter. Проще, но функционально беднее Jaeger.

**Когда выбирать:** legacy-системы, минимальные требования.

### Grafana Tempo

Современная система от создателей Grafana, хранит трейсы в object storage (S3, GCS).

**Когда выбирать:** уже есть Grafana + Loki, хотим единый UI для метрик, логов и трейсов.

## Семплирование (Sampling)

Трассировка каждого запроса может быть дорогой. Используется сэмплирование:

| Тип | Как работает | Когда |
|-----|-------------|-------|
| **Head-based** | Решаем в начале: трейсить или нет | Постоянный % (например, 5% запросов) |
| **Tail-based** | Решаем в конце: сохранять трейс или нет | Сохраняем только «интересные» (ошибки, медленные) |
| **Adaptive** | Автоматически меняем % сэмплирования | Когда много ошибок — увеличиваем |

**Для аналитика:** специфицировать политику сэмплирования — сколько трейсов хранить, какие обязательно (ошибки P0).

## Требования к трассировке (спецификация)

При описании требований к трассировке аналитик должен определить:

| Параметр | Пример |
|----------|--------|
| Протокол | OpenTelemetry (OTLP) |
| Какие операции трейсить | Все внешние вызовы (HTTP, gRPC, DB, очереди) |
| Сэмплирование | 100% для P0 операций, 10% для остальных |
| Хранение | 7 дней в горячем хранилище, 30 дней в холодном |
| Retention трейсов | 30 дней |
| Обязательные поля trace ID | Trace ID, Span ID, Service name, Duration |
| Какие ошибки сохранять | Все трейсы с error = true |
| Compliance | Не трейсить тела запросов с PII |

### Контекст трассировки в API

Аналитик должен требовать, чтобы **все сервисы передавали заголовки трассировки**:

```
X-Request-ID: abc-123          ← уникальный ID запроса (для пользователя)
Traceparent: 00-abc123...-def456-01  ← W3C Trace Context (для трейсинга)
```

**Почему важно:** без Traceparent невозможно связать запросы между сервисами.

## Ключевые термины

- **Trace** — полный путь запроса через все сервисы
- **Span** — одна операция в одном сервисе (с длительностью)
- **Trace ID** — идентификатор всего дерева вызовов
- **Span Context** — метаданные, передаваемые между сервисами
- **OpenTelemetry** — стандарт сбора трейсов, метрик, логов
- **Sampling** — стратегия отбора трейсов для хранения
- **Jaeger** — open-source система визуализации трейсов

## Проверь себя

1. **Чем trace отличается от span?**
   *Ответ:* Trace — полный путь запроса (дерево вызовов). Span — один шаг внутри одного сервиса. Trace состоит из спанов.

2. **Зачем нужен Trace ID и как он передаётся между сервисами?**
   *Ответ:* Trace ID связывает вызовы разных сервисов в один запрос. Передаётся через HTTP-заголовки (W3C Trace Context: `traceparent`).

3. **Что такое sampling и зачем он нужен?**
   *Ответ:* Трейсить каждый запрос дорого. Sampling определяет, какие трейсы сохранять: случайный %, только ошибки, только медленные.
