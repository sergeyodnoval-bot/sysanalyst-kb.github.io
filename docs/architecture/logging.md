:::info TL;DR
Логирование — запись событий системы в структурированном виде. Аналитик специфицирует: какие события логировать, как хранить, сколько хранить, кто имеет доступ. В современных системах логи собираются в централизованные системы (ELK, Loki), где их ищут, фильтруют и строят дашборды.
:::

## Что логировать

Аналитик определяет **категории** событий для логирования:

| Категория | Примеры | Уровень |
|-----------|---------|---------|
| **Действия пользователя** | Вход, выход, создание заказа | INFO |
| **Бизнес-события** | Оплата проведена, отчёт сгенерирован | INFO |
| **Вызовы API** | Входящие запросы, коды ответов | INFO / WARN |
| **Ошибки** | Исключения, падения сервисов | ERROR |
| **Системные события** | Запуск/остановка сервиса, деплой | INFO |
| **Безопасность** | Неудачные попытки входа, доступ к запрещённым ресурсам | WARN |
| **Производительность** | Время выполнения запросов (> 1 сек) | WARN |

**Правило для аналитика:** если событие нужно для расследования инцидента, аудита или метрик — оно должно логироваться.

## Уровни логирования

Стандартные уровни (RFC 5424):

```
FATAL — система неработоспособна
ERROR — ошибка, но система жива
WARN  — что-то пошло не так, но не критично
INFO  — бизнес-событие, штатная операция
DEBUG — детали для разработчика (не включать в прод без нужды)
TRACE — шаги внутри операции (только для отладки)
```

**Требование к прод-среде:** обычно INFO + WARN + ERROR. DEBUG и TRACE включают временно для диагностики.

## Структурированные vs неструктурированные логи

| Неструктурированные | Структурированные (JSON) |
|--------------------|-------------------------|
| `2024-01-15 10:30:00 ERROR: payment failed for order 123` | `{"timestamp":"...","level":"ERROR","service":"payment","order_id":123,"error":"insufficient_funds","user_id":456}` |
| Трудно парсить | Легко фильтровать в Kibana |
| Нельзя агрегировать по полям | Агрегация по `order_id`, `user_id` |
| Занимает меньше места | Чуть больше места, но намного полезнее |

**Требование аналитика:** все новые системы пишут логи в JSON (структурированные).

## Агрегация логов

В микросервисной архитектуре каждый сервис пишет свой лог-файл. Чтобы найти проблему, нужна централизованная система:

```
Service A ──→ File A
Service B ──→ File B      ❌ (децентрализовано)
Service C ──→ File C

Service A ──┐
Service B ──┼──→ Log Aggregator ──→ Storage ──→ UI (Kibana/Grafana)
Service C ──┘      (Filebeat /            │
                   Fluentd /               ├── JSON → ELK
                   Vector)                 └── LogQL → Loki
```

### ELK (Elasticsearch + Logstash + Kibana)

Классический стек. Logstash парсит логи, Elasticsearch индексирует, Kibana визуализирует.

**Когда выбирать:** нужен полнотекстовый поиск, сложная фильтрация, долгое хранение.
**Для аналитика:** самый популярный стек — если бюджет позволяет (Elasticsearch требует ресурсов).

### Loki (Grafana Loki)

Хранит только метаданные и сжатые логи, индексирует метки (labels), а не содержимое.

**Когда выбирать:** бюджет ограничен, логи в основном нужны для корреляции с метриками.
**Для аналитика:** дешевле ELK, хорошо дружит с Prometheus и Grafana.

## Требования к логированию (спецификация)

При описании требований к логированию аналитик должен определить:

| Параметр | Пример |
|----------|--------|
| Формат | JSON с обязательными полями: timestamp, level, service, message |
| Уровни | INFO + WARN + ERROR (продакшен), DEBUG (dev/staging) |
| Retention | 30 дней в горячем хранилище, 1 год в холодном |
| Категории | Действия пользователя, бизнес-события, ошибки |
| Объём | Оценка: ~100 MB в день на сервис |
| Доступ | Разработчики — read, администраторы — read+delete |
| Compliance | GDPR: не логировать PII (email, телефон, адрес) |

### Чего не должно быть в логах

- Пароли, токены, секреты
- Персональные данные (PII) — если без них нельзя, то маскировать
- Полные тела запросов с чувствительными данными
- Данные банковских карт (PCI DSS)

## Ключевые термины

- **Структурированное логирование** — логи в JSON с фиксированными полями
- **Log Aggregator** — система сбора логов со всех сервисов
- **ELK** — Elasticsearch, Logstash, Kibana (стек для логов)
- **Loki** — система сбора логов от Grafana, дешевле ELK
- **Retention** — срок хранения логов
- **Log level** — уровень важности: FATAL, ERROR, WARN, INFO, DEBUG, TRACE

## Проверь себя

1. **Какие уровни логирования включать в продакшене?**
   *Ответ:* INFO + WARN + ERROR. DEBUG и TRACE только временно для диагностики.

2. **Чем ELK отличается от Loki?**
   *Ответ:* ELK индексирует всё содержимое логов (дорого), Loki — только метки (labels). ELK лучше для поиска, Loki — дешевле и лучше для корреляции с метриками.

3. **Что нельзя писать в логи?**
   *Ответ:* Пароли, токены, PII (email, телефон), данные карт. Если нужно для аудита — маскировать.
