:::info TL;DR
Resilience patterns (паттерны отказоустойчивости) — способы сделать систему устойчивой к сбоям. Аналитик специфицирует, как система должна вести себя при ошибках: повторять запросы или нет, как долго ждать, что делать при превышении лимитов.
:::

## Зачем аналитику resilience

При проектировании системы аналитик описывает не только **happy path** (всё работает), но и **unhappy path**:

- Сервис A не отвечает → что делает сервис B?
- Внешнее API тормозит → сколько ждать?
- Пришло 10 000 запросов в секунду → что с остальными?

Ответы на эти вопросы — требования к отказоустойчивости. Без них разработчики реализуют поведение по умолчанию (обычно — падение с ошибкой).

## Паттерны

### 1. Retry (повтор)

**Проблема:** временная ошибка сети или БД — запрос не прошёл.
**Решение:** повторить запрос через некоторое время.

```
Запрос ──→ Ошибка ──→ ждать 100ms ──→ повторить ──→ успех
```

**Виды retry:**

- **Immediate** — повторить сразу (опасно: завалим сервис)
- **Fixed delay** — ждать N мс между попытками
- **Exponential backoff** — ждать 100ms, 200ms, 400ms, 800ms (рекомендуется)
- **Jitter** — добавить случайность к задержке (чтобы не было «волны» повторов)

**Требования аналитика:**
| Параметр | Значение |
|----------|----------|
| Max retries | 3 |
| Delay | Exponential backoff: 100ms → 200ms → 400ms |
| Какие ошибки повторять | 5xx, timeout, network error |
| Какие не повторять | 4xx (кроме 429), validation error |

**Важно:** повторять только **идемпотентные** операции. Повтор списания денег = плохо.

### 2. Timeout (тайм-аут)

**Проблема:** сервис не отвечает, клиент ждёт вечно (или 30 секунд по умолчанию — и все потоки заняты).
**Решение:** установить максимальное время ожидания ответа.

```
Запрос ──→ жду 500ms ──→ превышен timeout ──→ ошибка
```

**Типовые требования:**
| Компонент | Timeout |
|-----------|---------|
| API → API Gateway | 30 сек |
| Gateway → сервис | 10 сек |
| Сервис → БД | 2 сек |
| Сервис → внешнее API | 5 сек |
| Сервис → очередь | 3 сек |

**Правило:** timeout должен быть соизмерим с SLA сервиса. Если сервис обещает ответ за 500 мс (P95), ставить timeout 2 сек, а не 30 сек.

### 3. Circuit Breaker (предохранитель)

**Проблема:** внешний сервис упал и не восстанавливается. Каждый запрос к нему висит до timeout, съедая ресурсы.
**Решение:** после N ошибок перестать отправлять запросы и сразу возвращать fallback.

```
Состояния:

Closed (всё хорошо) ──(N ошибок)──→ Open (не пускаю)
        ↑                                      │
        └──(timeout прошёл)──── Half-Open ←────┘
                                │
                          ┌─────┴─────┐
                          ↓           ↓
                      успех → Closed   ошибка → Open
```

| Состояние | Поведение |
|-----------|-----------|
| **Closed** | Запросы проходят, считаем ошибки |
| **Open** | Запросы отклоняются мгновенно (fallback) |
| **Half-Open** | Пробный запрос: успех → Closed, ошибка → Open |

**Требования аналитика:**
| Параметр | Значение |
|----------|----------|
| Порог ошибок для Open | 5 ошибок за 10 секунд |
| Время восстановления (Open → Half-Open) | 30 секунд |
| Fallback | Пустой ответ + callback для уведомления |
| Какие ошибки считать | 5xx, timeout, connection refused |

### 4. Bulkhead (перегородка)

**Проблема:** один медленный сервис «съедает» все потоки приложения, блокируя остальные.
**Решение:** ограничить количество одновременных запросов к каждому сервису.

```
Без Bulkhead:                 С Bulkhead:
┌──────────────────┐          ┌──────────────────┐
│   Thread Pool    │          │  Pool A  │ Pool B │
│ ┌──────────────┐ │          │ ┌──────┐ │ ┌────┐ │
│ │  Request A   │ │          │ │ Req A│ │ │ReqC│ │
│ │  Request B   │ │          │ │ Req B│ │ │ReqD│ │
│ │  Request C   │ │  ❌ B    │ └──────┘ │ └────┘ │
│ │  Request D   │ │  blocks │  max: 5  │ max: 5 │
│ └──────────────┘ │          └──────────┘─────────┘
└──────────────────┘
```

**Требования аналитика:**
| Параметр | Значение |
|----------|----------|
| Max concurrent | 10 запросов к сервису |
| Queue size | 20 запросов в очереди |
| Поведение при переполнении | Reject + 503 Service Unavailable |

### 5. Rate Limiting (ограничение запросов)

**Проблема:** клиент (или злоумышленник) шлёт 1000 запросов в секунду, сервис падает.
**Решение:** ограничить количество запросов от одного клиента в единицу времени.

```
Клиент X: 100 запросов / минуту → лимит: 10/мин → 90 запросов rejected
```

**Алгоритмы:**

| Алгоритм | Как работает | Плюсы | Минусы |
|----------|-------------|-------|--------|
| **Token Bucket** | Токены накапливаются с постоянной скоростью | Сглаживает нагрузки | Сложнее в реализации |
| **Fixed Window** | Счётчик сбрасывается каждую минуту | Просто | Проблема с краями окна |
| **Sliding Window** | Окно времени скользит | Справедливо | Требует памяти |
| **Leaky Bucket** | Фиксированная скорость обработки | Равномерно | Не гибкий |

**HTTP-статусы:**
- `429 Too Many Requests` — превышен лимит
- Заголовок `Retry-After: 60` — через сколько можно повторить

**Требования аналитика:**
| Параметр | Значение |
|----------|----------|
| Лимит | 100 запросов/минуту на пользователя |
| Алгоритм | Token Bucket |
| Burst | До 20 запросов за 1 секунду |
| Ответ при превышении | 429 + Retry-After |
| Исключения | Золотые клиенты — 1000/мин |

## Комбинирование паттернов

Паттерны используются вместе:

```
Запрос
  │
  ▼
[Circuit Breaker] ──(Open)──→ Fallback
  │
  ▼ (Closed)
[Rate Limiter] ──(Exceeded)──→ 429
  │
  ▼ (OK)
[Bulkhead] ──(Full)──→ 503
  │
  ▼ (Free)
[Timeout]
  │
  ▼
[Retry] ←── Экспоненциальный backoff
  │
  ▼
  Успех / Финальная ошибка
```

## Что специфицирует аналитик

| Паттерн | Вопрос аналитика |
|---------|-----------------|
| **Retry** | Какие ошибки повторять? Сколько раз? С какой задержкой? |
| **Timeout** | Сколько ждать ответа от каждого сервиса? |
| **Circuit Breaker** | После скольких ошибок «отключать» сервис? Какой fallback? |
| **Bulkhead** | Сколько одновременных запросов допустимо к каждому сервису? |
| **Rate Limiting** | Сколько запросов в минуту от одного пользователя? Какой burst? |

## Ключевые термины

- **Retry** — повтор запроса при ошибке
- **Timeout** — максимальное время ожидания ответа
- **Circuit Breaker** — предохранитель: после N ошибок не пускает запросы
- **Bulkhead** — изоляция потоков между сервисами
- **Rate Limiting** — ограничение количества запросов
- **Exponential Backoff** — увеличение задержки между повторами
- **Fallback** — запасной ответ при ошибке
- **Idempotency** — повторный запрос даёт тот же результат

## Проверь себя

1. **Какие ошибки можно повторять, а какие нет?**
   *Ответ:* Можно: 5xx, timeout, network error. Нельзя: 4xx (кроме 429), validation error. Повторять можно только идемпотентные операции.

2. **Как работают состояния Circuit Breaker?**
   *Ответ:* Closed → ошибки → Open → timeout → Half-Open → пробный запрос → успех = Closed, ошибка = Open.

3. **Зачем нужен Bulkhead и как он связан с timeouts?**
   *Ответ:* Bulkhead изолирует потоки, чтобы один медленный сервис не заблокировал всё приложение. Без timeouts Bulkhead не спасёт — нужно и то, и другое.
