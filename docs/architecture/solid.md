# SOLID для системного аналитика

SOLID — пять принципов объектно-ориентированного проектирования. Разработчики используют их для написания кода. Аналитик использует их для проектирования системы: как делить на модули, как описывать интерфейсы, как управлять зависимостями.

## Принципы

### S — Single Responsibility (единственная ответственность)

**Для разработчика:** класс имеет одну причину для изменения.

**Для аналитика:** модуль / микросервис / компонент отвечает за одну бизнес-способность.

**Пример.** Сервис «Заказы» отвечает за создание и управление заказами. Он не должен заниматься отправкой email — это ответственность сервиса «Уведомления».

### O — Open/Closed (открыт для расширения, закрыт для изменения)

**Для разработчика:** можно добавить новое поведение без изменения существующего кода.

**Для аналитика:** система должна поддерживать новые бизнес-правила без переписывания старых.

**Пример.** Система расчёта скидок: добавление нового типа скидки не должно ломать существующие. Проектируйте через плагины или стратегии.

### L — Liskov Substitution (подстановка Лисков)

**Для разработчика:** наследник должен вести себя как родитель.

**Для аналитика:** замена одного компонента на другой не должна ломать систему. Если вы описываете «способ оплаты», любой способ (карта, SBP, PayPal) должен работать одинаково с точки зрения вызывающего.

### I — Interface Segregation (разделение интерфейсов)

**Для разработчика:** много маленьких интерфейсов лучше одного большого.

**Для аналитика:** API / контракты должны быть узкими и специфичными. Не заставляйте клиента implement'ить методы, которые ему не нужны.

**Пример.** Вместо одного API `UserService` с методами `create`, `update`, `delete`, `ban`, `sendEmail`, `generateReport` — разбейте на отдельные сервисы или endpoints.

### D — Dependency Inversion (инверсия зависимостей)

**Для разработчика:** код зависит от абстракций, а не от конкретных реализаций.

**Для аналитика:** модули верхнего уровня не должны зависеть от деталей реализации нижнего. Бизнес-логика не должна знать, какая БД используется.

## Как SOLID помогает аналитику

- **Определять границы микросервисов** (SRP)
- **Писать требования** — каждое требование должно касаться одной сущности
- **Проектировать API** — узкие, специализированные endpoints
- **Описывать нефункциональные требования** — «система должна допускать добавление новых типов X без изменения существующей логики»
