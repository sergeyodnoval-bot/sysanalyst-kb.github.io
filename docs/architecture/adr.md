# ADR — Architecture Decision Records

ADR — короткий документ, фиксирующий архитектурное решение: контекст, рассмотренные варианты, выбранный вариант и его последствия. Каждое решение — отдельный файл в репозитории.

## Зачем ADR

Без ADR архитектурные решения живут в головах, чатах и устных договорённостях. Через полгода никто не помнит, почему выбрали Kafka, а не RabbitMQ. Через год — начинают переделывать.

ADR фиксирует:

- **Почему** приняли решение (а не только что решили)
- **Что рассматривали** (какие альтернативы)
- **Кто решил** (автор и стейкхолдеры)
- **Когда** (дата)
- **Последствия** (что это значит для системы)

## Структура ADR

```markdown
# ADR-001: Выбор брокера сообщений

## Статус
Принято

## Контекст
Нам нужно асинхронное взаимодействие между Order Service и Notification Service.
Объём: до 1000 сообщений в секунду.
Требование: надёжная доставка, dead letter queue.

## Варианты
1. RabbitMQ — зрелый брокер с гибкой маршрутизацией
2. Kafka — распределённый лог с высокой пропускной способностью
3. Amazon SQS — managed решение, не нужно администрировать

## Решение
RabbitMQ. Критерии: гибкая маршрутизация, DLQ из коробки,
команда имеет опыт с RabbitMQ.

## Последствия
- Потребуется администрирование RabbitMQ
- При росте нагрузки > 10k msg/s — пересмотреть
- Мониторинг через Prometheus + Grafana
```

## Число ADR

- Каждый ADR имеет уникальный номер. Последовательный.
- Номер не меняется, даже если решение отменено.
- ADR-000 может быть шаблоном или индексом.

## Когда писать ADR

- Выбор технологии (БД, брокер, язык)
- Выбор архитектурного стиля (микросервисы vs монолит)
- Изменение контракта между сервисами
- Решение, которое сложно отменить
- Решение, затрагивающее другие команды

## Когда НЕ писать

- Выбор цвета кнопки
- Какой ORM использовать в рамках одного сервиса
- Формат логов — если это не затрагивает других

## Lifecycle

- **Proposed** — предложено, обсуждается
- **Accepted** — принято
- **Deprecated** — заменено новым ADR
- **Superseded** — решение отменено новым ADR

## Что дальше

- **SOLID** — принципы для оценки архитектурных решений
- **Event-Driven Architecture** — запись решений в ADR
