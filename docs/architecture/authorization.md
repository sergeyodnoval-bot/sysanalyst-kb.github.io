:::info TL;DR
Авторизация определяет, кто (пользователь/роль) что может делать (читать/писать/удалять) с каким ресурсом (документ/заказ/отчёт). Аналитик специфицирует матрицу доступа: роли, их права, и на каких условиях. Основные модели: RBAC (роли), ABAC (атрибуты), ACL (списки).
:::

## Аутентификация vs Авторизация

Эти два понятия часто путают. Разница принципиальна:

```
Аутентификация (AuthN):  «Кто ты?»  → логин + пароль, JWT, OAuth
Авторизация (AuthZ):     «Что тебе можно?» → права доступа, роли
```

**Аналитик должен всегда уточнять:** бизнес хочет «кто может зайти» (authN) или «кто что может делать» (authZ). Чаще нужно и то, и другое — но это разные требования.

## Модели авторизации

### 1. ACL (Access Control List)

**Как работает:** у каждого ресурса есть список пользователей/групп с правами.

```
Документ: «Отчёт по продажам Q1»
  ┌─────────────────────┐
  │ Иван → read, write  │
  │ Пётр → read         │
  │ Мария → read, write │
  │ Анна → admin        │
  └─────────────────────┘
```

**Плюсы:** гибко, просто понять.
**Минусы:** не масштабируется (на 1000 пользователей и 1000 документов — 1 млн записей).

**Когда использовать:** файловые системы, небольшие приложения.

### 2. RBAC (Role-Based Access Control)

**Как работает:** права назначаются не пользователям, а ролям. Пользователю назначается роль.

```
Роли:                  Права:
┌─────────────┐        ┌──────────────────────┐
│ Администратор│──────→│ read, write, delete   │
├─────────────┤        ├──────────────────────┤
│ Менеджер    │──────→│ read, write           │
├─────────────┤        ├──────────────────────┤
│ Аналитик    │──────→│ read                  │
├─────────────┤        ├──────────────────────┤
│ Аудитор     │──────→│ read (только логи)    │
└─────────────┘        └──────────────────────┘

Пользователи → роли:
  Иван → Администратор
  Пётр → Менеджер
  Мария → Аналитик
  Анна → Аудитор
```

**Плюсы:** масштабируется (ролей мало, пользователей много), просто администрировать.
**Минусы:** негибко — если доступ зависит от контекста (например, «менеджер может редактировать только свои заказы»).

**Когда использовать:** корпоративные системы, CRM, ERP, большинство бизнес-приложений.

### 3. ABAC (Attribute-Based Access Control)

**Как работает:** доступ определяется правилами на основе атрибутов:
- Атрибуты пользователя: роль, отдел, стаж, локация
- Атрибуты ресурса: тип документа, статус, цена, владелец
- Атрибуты окружения: время суток, IP-адрес, устройство

```
Правило: «Менеджер может редактировать заказы своего отдела,
         если сумма заказа < $10 000, в рабочее время»
         │            │                │              │
         role =       department =     amount <       time =
         'manager'    user.department  10000          09:00-18:00
```

**Плюсы:** максимально гибко, контекстно-зависимые правила.
**Минусы:** сложно администрировать, трудно отлаживать («почему я не вижу эту кнопку?»).

**Когда использовать:** сложные домены (банкинг, healthcare, FinTech), где доступ зависит от множества факторов.

### Сравнение

| Критерий | ACL | RBAC | ABAC |
|----------|-----|------|------|
| Сложность | Низкая | Средняя | Высокая |
| Масштабируемость | Плохая | Хорошая | Хорошая |
| Гибкость | Высокая | Средняя | Высокая |
| Администрирование | Кошмар | Просто | Сложно |
| Контекст | Нет | Нет | Да |

## Как аналитик специфицирует авторизацию

### Матрица доступа (RACI для данных)

| Роль | Заказы (read) | Заказы (write) | Отчёты | Админка | Логи |
|------|---------------|----------------|--------|---------|------|
| Администратор | ✅ | ✅ | ✅ | ✅ | ✅ |
| Менеджер | ✅ | ✅ (свои) | ✅ | ❌ | ❌ |
| Аналитик | ✅ | ❌ | ✅ (read) | ❌ | ❌ |
| Аудитор | ❌ | ❌ | ❌ | ❌ | ✅ (read) |

### Требования к авторизации (спецификация)

| Параметр | Пример |
|----------|--------|
| Модель | RBAC |
| Роли | Администратор, Менеджер, Аналитик, Аудитор |
| Права | Create, Read, Update, Delete (CRUD) для каждой сущности |
| Иерархия | Администратор включает права Менеджера |
| Data scope | Менеджер видит только заказы своего отдела |
| Audit | Все изменения прав логируются |
| UI | Пользователь не видит кнопок, на которые нет прав |

### Типовые вопросы аналитика при сборе требований:

1. Какие роли нужны?
2. Какие действия может выполнять каждая роль?
3. Есть ли иерархия ролей (админ включает права других)?
4. Может ли пользователь иметь несколько ролей?
5. Ограничен ли доступ по данным (вижу только свои/своего отдела)?
6. Нужен ли audit-log изменений прав?

## Авторизация в API

В REST API авторизация обычно реализуется через:

- **JWT-токен** — содержит роли/права в payload (claims)
- **API Key** — ключ привязан к роли
- **OAuth 2.0 scopes** — область доступа (read, write, admin)

Пример JWT с ролями:

```json
{
  "sub": "user_123",
  "roles": ["manager", "analyst"],
  "department": "sales",
  "permissions": ["orders:read", "orders:write"]
}
```

## Ключевые термины

- **AuthN (Authentication)** — проверка, кто пользователь
- **AuthZ (Authorization)** — проверка, что пользователю можно
- **RBAC** — модель на основе ролей
- **ABAC** — модель на основе атрибутов
- **ACL** — список доступа для каждого ресурса
- **JWT Claims** — данные в JWT-токене (роли, права)
- **OAuth Scopes** — области доступа в OAuth 2.0

## Проверь себя

1. **Чем RBAC отличается от ABAC?**
   *Ответ:* RBAC — права привязаны к роли, негибко. ABAC — права вычисляются по правилам на основе атрибутов (роль, отдел, время). ABAC гибче, но сложнее.

2. **Что такое ACL и в чём его проблема?**
   *Ответ:* У каждого ресурса список пользователей с правами. Проблема: не масштабируется — при 1000+ пользователей становится неадминистрируемым.

3. **Какие вопросы задать бизнесу при сборе требований к авторизации?**
   *Ответ:* Какие роли? Какие права у каждой? Иерархия ролей? Несколько ролей у одного пользователя? Ограничение по данным? Нужен аудит?
