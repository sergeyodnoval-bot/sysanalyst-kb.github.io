:::info TL;DR
Мониторинг позволяет понять, работает ли система и насколько хорошо. Observability (наблюдаемость) — более широкое понятие: способность понять внутреннее состояние системы по внешним данным (логам, метрикам, трейсам). Аналитик специфицирует метрики, пороги алертов и SLO для системы.
:::

## Monitoring vs Observability

| | Monitoring | Observability |
|--|-----------|---------------|
| **Вопрос** | Система работает? | Почему система не работает? |
| **Что нужно** | Заранее определённые метрики | Логи + метрики + трейсы |
| **Реагирует на** | Известные проблемы | Известные и неизвестные проблемы |
| **Пример** | CPU > 90% → алерт | По логам + трейсам нашли причину |

**Аналитику важно:** при спецификации системы требуйте observability (логи + метрики + трейсы), а не просто мониторинг.

## Три столпа observability

```
         ┌──────────────────┐
         │   Observability  │
         ├──────────────────┤
         │ Logs  (логи)     │ — что произошло
         │ Metrics (метрики)│ — сколько, как часто
         │ Traces (трейсы)  │ — где именно, в каком сервисе
         └──────────────────┘
```

## Типы метрик

### 1. Метрики доступности (Availability / Uptime)

- **Uptime** — процент времени, когда система отвечает
- **Health check** — эндпоинт `/health` возвращает 200 OK
- **SLA** — договорной уровень доступности (99.9%, 99.99%)

**Типовые требования:**
- Uptime: 99.9% (≈ 8.7 часов простоя в год)
- Health check раз в 10 секунд
- Время восстановления (MTTR) < 1 час

### 2. Метрики производительности (Performance)

| Метрика | Что измеряет | Типичный порог |
|---------|-------------|----------------|
| **Latency (P95)** | Время ответа для 95% запросов | < 500 мс |
| **Throughput** | Запросов в секунду (RPS) | Зависит от системы |
| **Error rate** | % ошибок (5xx, 4xx) | < 1% |
| **CPU / Memory** | Загрузка ресурсов | < 80% |

**P95 vs P99:** P95 — «типичная» задержка для пользователя. P99 — «худший случай», важна для SLA.

### 3. Бизнес-метрики

Метрики, которые важны бизнесу:

| Метрика | Описание |
|---------|----------|
| DAU / MAU | Активные пользователи в день/месяц |
| Conversion rate | % пользователей, сделавших целевое действие |
| Revenue per user | Средний доход с пользователя |
| Order success rate | % успешно оформленных заказов |

**Роль аналитика:** определить, какие бизнес-метрики логировать, чтобы можно было коррелировать с техническими.

## Prometheus и Grafana

### Prometheus

Система сбора метрик с pull-моделью: сама забирает метрики с сервисов.

```
Service A ──(/metrics)──┐
Service B ──(/metrics)──┼──→ Prometheus ──→ Grafana
Service C ──(/metrics)──┘
```

**Как это выглядит для разработки:** каждый сервис отдаёт эндпоинт `/metrics` с метриками в формате Prometheus. Аналитик специфицирует, какие метрики должен отдавать сервис.

### Grafana

Визуализация метрик: дашборды, графики, алерты.

**Для аналитика:** типовые дашборды:
- **RED (Rate, Errors, Duration)** — для каждого сервиса
- **USE (Utilization, Saturation, Errors)** — для каждого ресурса

## Алертинг (Alerting)

### Типы алертов

| Тип | Описание | Пример |
|-----|----------|--------|
| **Threshold** | Метрика превысила порог | CPU > 90% 5 минут |
| **Absent** | Сервис перестал отдавать метрики | Service X down |
| **Trend** | Метрика растёт/падает | Error rate растёт последний час |
| **Composite** | Комбинация условий | Latency > 500 мс AND error rate > 1% |

### Severity алертов

| Severity | Реакция | Пример |
|----------|---------|--------|
| **P0** (Critical) | Будить команду ночью | Сервис не отвечает |
| **P1** (High) | Реагировать в рабочее время | Latency > 2 сек |
| **P2** (Medium) | Создать задачу на починку | CPU > 80% |
| **P3** (Low) | Заметка в бэклог | Метрика нарастает |

## SLO, SLI, SLA

**SLA (Service Level Agreement)** — договорной уровень сервиса (юридический).
**SLO (Service Level Objective)** — целевой показатель внутри команды.
**SLI (Service Level Indicator)** — конкретная метрика.

Пример:
- **SLI:** latency P95 запросов к сервису payments
- **SLO:** latency P95 < 500 мс, 99.9% времени за месяц
- **SLA:** 99.9% uptime, компенсация при нарушении

**Роль аналитика:** определить SLI (что измеряем), помочь установить SLO (какой порог) и проверить, что SLA выполним.

## Типовые требования к мониторингу (спецификация)

При описании требований к мониторингу аналитик должен определить:

| Параметр | Пример |
|----------|--------|
| Health check | Эндпоинт `/health`, интервал 10 сек |
| Метрики | RED: Rate, Errors, Duration для каждого API-метода |
| Дашборды | Grafana: общий дашборд + дашборд для каждого сервиса |
| Алерты | P0: сервис не отвечает 5 мин; P1: error rate > 5% |
| Retention метрик | 30 дней с шагом 15 сек, 1 год с шагом 1 час |
| SLO | Latency P95 < 500 мс, uptime 99.9% |
| Аудит | Логи алертов хранить 1 год |

## Ключевые термины

- **Observability** — способность понять состояние системы по внешним данным
- **Prometheus** — система сбора и хранения метрик
- **Grafana** — визуализация метрик и алертинг
- **RED** — Rate, Errors, Duration (метрики для сервисов)
- **USE** — Utilization, Saturation, Errors (метрики для ресурсов)
- **SLO** — целевой уровень сервиса
- **SLA** — договорной уровень сервиса (с компенсациями)
- **P95 / P99** — перцентили задержки

## Проверь себя

1. **В чём разница между monitoring и observability?**
   *Ответ:* Monitoring отвечает «работает ли?», observability — «почему не работает?». Observability = метрики + логи + трейсы.

2. **Что такое RED и USE?**
   *Ответ:* RED — Rate, Errors, Duration (для сервисов). USE — Utilization, Saturation, Errors (для ресурсов). Это шаблоны метрик.

3. **Чем SLO отличается от SLA?**
   *Ответ:* SLO — внутренняя цель команды, SLA — юридическое обязательство. SLO обычно жёстче, чем SLA.
