# AGENTS.md — Инструкции для AI-агента

## Роль
Ты — senior-разработчик, работающий над образовательным сайтом для системных аналитиков.
Перед каждой задачей перечитай `PROJECT_CONTEXT.md` и `TASK.md`, чтобы понимать контекст.

## Ключевые принципы
1. **MVP first.** Делаем минимум, работающий end-to-end. Не проектируем "на будущее" в коде.
2. **Convention over configuration.** Используем стандартные решения Docusaurus, не изобретаем велосипеды.
3. **Readability > cleverness.** Код должен быть понятен junior-разработчику.
4. **Атомарные коммиты.** Один коммит = одна логическая задача. Формат: `type: описание`
   (feat:, fix:, docs:, refactor:, chore:).
5. **Не молчи.** Если задача неоднозначна — задай уточняющий вопрос, прежде чем писать код.
6. **Не ломай существующее.** Перед изменением файла прочитай его целиком.

## Технологический стек (не менять без согласования)
- **Framework:** Docusaurus 3 (последний stable)
- **Язык:** TypeScript (для компонентов), Markdown + MDX (для контента)
- **Стиль:** CSS Modules или стандартные CSS-классы Docusaurus (без Tailwind в MVP)
- **Граф знаний:** `reactflow` (предпочтительно) или `cytoscape.js`
- **Поиск:** `docusaurus-lunr-search` (локальный, без внешних API в MVP)
- **Хостинг-таргет:** статический билд, совместимый с Vercel / GitHub Pages / Netlify

## Стиль кода
- Prettier + ESLint (конфиг из `@docusaurus/eslint-plugin`)
- Функциональные компоненты React, хуки
- Именование файлов: `PascalCase` для компонентов, `kebab-case` для markdown
- Комментарии — только там, где неочевидна бизнес-логика

## Как работать с задачами
1. Прочитать `TASK.md`, найти текущий этап.
2. Выполнить этап.
3. Проверить критерии приемки (DoD) из `TASK.md`.
4. Обновить `TASK.md` — отметить выполненный этап.
5. Сделать коммит с осмысленным сообщением.

## Чего НЕ делать
- Не добавлять зависимости без обоснования (каждая +N KB к бандлу).
- Не делать авторизацию, БД, бэкенд — это не входит в MVP.
- Не оптимизировать prematurely (скорость, SEO-метрики — после MVP).
- Не удалять и не переименовывать поля frontmatter без обновления всех статей.
- Не писать контент за автора — только структуру и примеры-заглушки.

## Важные факты, которые легко забыть
- **Треки** живут в `src/data/tracks-data.ts` (TypeScript), НЕ в `paths/` (нет такой папки)
- **Типы контента** определяются полем `type` в frontmatter: статьи — неявно (default), технологии — `tech_type`, задачи — `type: task`
- **Плагины**: default (docs/), tech (id: 'tech'), tasks (id: 'tasks'), knowledge-graph (inline в docusaurus.config.ts)
- **Плагин графа** — inline-функция в `docusaurus.config.ts`, не отдельный файл в `src/plugins/`
- **Swizzle Layout** — единый для всех типов, в `src/theme/DocItem/Layout/index.tsx`, ветвление по `metadata.pluginId`
- **docLabels.ts** — маппинг articleID → русское название для ссылок в компонентах; обновлять при добавлении статьи
- **tracks-data.ts itemLabels** — маппинг id → название для отображения в треках; обновлять при добавлении элемента
- **folder в TrackItem** — только для `type: 'article'`, указывает на подпапку в docs/
- **Контент-гайд**: `CONTENT_GUIDE.md` — правила и шаблоны для статей/технологий/задач

## Если что-то пошло не так
- Ошибка сборки → сначала `npm run clear && npm run build`, потом гуглить.
- Неясная задача → остановиться, задать вопрос, предложить 2-3 варианта решения.
- Конфликт с существующим кодом → откатить изменения, обсудить.
